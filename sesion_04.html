MPI

El estándar de paso de mensajes. Más explícito que OpenMP —cada comunicación se escribe a mano— pero el único modelo que funciona cuando los procesos viven en máquinas distintas.

Filosofía SPMD

Casi todos los programas MPI siguen el modelo SPMD (Single Program, Multiple Data): un único ejecutable, lanzado en múltiples copias (procesos), donde cada proceso decide qué hacer mirando su identificador. No hay maestro «físico» como en OpenMP: todos los procesos son iguales y el rol de cada uno lo determina el código.

if (rank == 0) ⟶ comportamientos distintos por proceso.

SPMD: una sola fuente, múltiples procesos, cada uno con su propio rank y memoria privada.

Cuatro funciones obligatorias (el «hola mundo»)

#include <mpi.h>
#include <stdio.h>

int main(int argc, char *argv[]) {
    int rank, nproc;

    MPI_Init(&argc, &argv);              // 1. arranca MPI
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);  // 2. ¿quién soy?
    MPI_Comm_size(MPI_COMM_WORLD, &nproc); // 3. ¿cuántos somos?

    printf("Hola desde el proceso %d de %d\n", rank, nproc);

    MPI_Finalize();                       // 4. cierra MPI
    return 0;
}

Para compilar: mpicc hola.c -o hola. Para lanzar 8 procesos: mpirun -np 8 ./hola.

El parámetro MPI_COMM_WORLD es el comunicador por defecto: el grupo de todos los procesos arrancados con mpirun. Puedes crear comunicadores más pequeños para subconjuntos, pero MPI_COMM_WORLD basta para casi todo.

Comunicación punto a punto

La más básica: un proceso envía, otro recibe.

// Emisor
MPI_Send(buf, count, datatype, dest, tag, comm);

// Receptor
MPI_Recv(buf, count, datatype, source, tag, comm, &status);

Los parámetros:

  • buf: dirección del buffer (puntero a los datos).
  • count: cuántos elementos.
  • datatype: tipo MPI (MPI_INT, MPI_DOUBLE, MPI_CHAR...).
  • dest / source: rank del otro extremo. En la recepción puedes usar MPI_ANY_SOURCE si no te importa.
  • tag: etiqueta entera para distinguir mensajes. MPI_ANY_TAG acepta cualquiera.
  • comm: comunicador (normalmente MPI_COMM_WORLD).
  • status: información del mensaje recibido (origen real, tamaño...). Si no la necesitas, MPI_STATUS_IGNORE.

Las cuatro variantes de envío

Esta es una de las preguntas frecuentes de examen. Las cuatro modalidades cambian qué tiene que pasar para que send retorne:

FunciónRetorna cuando...Pros / contras
MPI_Ssend El receptor ha empezado a recibir Sincronización máxima, sin buffer extra
MPI_Bsend Los datos se han copiado a un buffer de usuario No espera al receptor, pero requiere reservar buffer con MPI_Buffer_attach
MPI_Rsend Se asume que el recv ya está activo Más rápido, pero peligroso: si no hay recv esperando, error
MPI_Send (estándar) La implementación decide entre síncrono o con buffer Cómodo, pero el comportamiento depende del tamaño del mensaje y de la implementación

Comunicación no bloqueante

Los send/recv anteriores son bloqueantes: la función no retorna hasta que el sistema garantiza que el buffer se puede reutilizar (o que el dato está). Para solapar cómputo y comunicación, MPI ofrece versiones no bloqueantes con prefijo I:

MPI_Request req;
MPI_Irecv(buf, n, MPI_DOUBLE, vecino, tag, MPI_COMM_WORLD, &req);

// hago otro trabajo mientras llega el mensaje...
calcular_otra_cosa();

MPI_Wait(&req, MPI_STATUS_IGNORE);   // ahora sí, espero a que llegue
// el buffer ya es seguro de usar

Funciones útiles del lado no bloqueante:

  • MPI_Isend, MPI_Irecv: arrancan la operación y devuelven un MPI_Request.
  • MPI_Wait(&req): bloquea hasta que la operación se completa.
  • MPI_Test(&req, &flag): pregunta sin bloquear si ya está. Pone flag a true si sí.
  • MPI_Waitall, MPI_Waitany, MPI_Waitsome: para varias request a la vez.
Patrón muy típico: arrancar todos los Irecv al principio del paso, mandar Isend a continuación, hacer el cómputo del interior del dominio (que no necesita los datos de los vecinos), y finalmente Waitall antes de procesar los bordes. Es la base de los códigos de difusión y stencil.

Comunicación colectiva

Cuando intervienen todos los procesos de un comunicador. Más cómoda y, casi siempre, más eficiente que ensamblar la misma operación con Send/Recv manuales.

Las colectivas más usadas. Reglas de oro: todas las llamadas en un comunicador, todos los procesos las invocan, los buffers de envío en root pueden tener tamaños distintos a los de recepción.

ATAJO MUY COMÚN

MPI_Reduce y MPI_Allreduce

Si todos los procesos van a necesitar el resultado de la reducción (por ejemplo, el error global en un solver iterativo), usa MPI_Allreduce. Equivale a Reduce + Bcast pero la implementación es notablemente más eficiente.

Ejemplo: integral por trapecios

El proceso 0 reparte el intervalo, cada uno calcula su parte, y al final se suman las integrales parciales.

double integral_local(double a, double b, int n) {
    double h = (b - a) / n, s = (f(a) + f(b)) / 2.0;
    for (int i = 1; i < n; i++) s += f(a + i*h);
    return s * h;
}

int main(int argc, char **argv) {
    MPI_Init(&argc, &argv);
    int rank, nproc;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &nproc);

    double a = 0.0, b = 1.0;
    int n = 1000000;
    double h_global = (b - a) / nproc;

    // cada proceso integra su trozo
    double a_local = a + rank * h_global;
    double b_local = a_local + h_global;
    double integ_local = integral_local(a_local, b_local, n / nproc);
    double integ_total;

    MPI_Reduce(&integ_local, &integ_total, 1, MPI_DOUBLE,
               MPI_SUM, 0, MPI_COMM_WORLD);

    if (rank == 0) printf("Integral = %.10f\n", integ_total);
    MPI_Finalize();
    return 0;
}

Cuando los datos no son contiguos

MPI envía bloques contiguos de un mismo tipo. Cuando esto no se cumple —arrays de punteros, estructuras complejas, subarrays con stride— hay tres opciones:

Tipos derivados

Construyes un nuevo MPI_Datatype que describe la disposición real en memoria. Más limpio para uso repetido.

Funciones: MPI_Type_vector, MPI_Type_indexed, MPI_Type_struct. Hay que llamar a MPI_Type_commit antes de usar y MPI_Type_free al final.

Pack / Unpack

Tú mismo serializas en un buffer contiguo con MPI_Pack, envías como MPI_PACKED, y el receptor deserializa con MPI_Unpack.

Más flexible para datos que cambian de forma; menos eficiente si se hace en bucle interno.

Modelo simple de tiempo de comunicación

Aparecerá una y otra vez en el análisis de algoritmos. El tiempo de mandar n datos es aproximadamente:

FÓRMULA QUE HAY QUE RECORDAR

tcom = ts + n · tw

ts (startup): coste fijo por iniciar una comunicación, independiente del tamaño. Latencia.

tw (word time): coste por unidad de dato. Ancho de banda inverso.

Implicación práctica: muchos mensajes pequeños son catastróficos. Suele compensar agrupar datos y enviar un único mensaje grande, aunque retrasarse un poco.

Cuidado con los deadlocks

El error clásico:

// AMBOS PROCESOS HACEN ESTO
if (rank == 0) {
    MPI_Ssend(buf, n, MPI_DOUBLE, 1, ...);   // espera al recv del 1
    MPI_Recv (buf, n, MPI_DOUBLE, 1, ...);
} else if (rank == 1) {
    MPI_Ssend(buf, n, MPI_DOUBLE, 0, ...);   // espera al recv del 0
    MPI_Recv (buf, n, MPI_DOUBLE, 0, ...);
}

Ambos esperan que el otro reciba primero: deadlock. Soluciones:

  • Asimetría explícita: uno envía y otro recibe primero.
  • Usar MPI_Sendrecv (intercambio atómico, libre de deadlock).
  • Usar versiones no bloqueantes (MPI_Isend + MPI_Recv).

Práctica de recuperación

El examen del curso suele incluir preguntas concretas de sintaxis. Si te equivocas en una tarjeta, marca esa idea para revisar mañana —recordar tras un fallo refuerza más que repasar lo que ya sabes—.

¿Qué significa SPMD en el contexto de MPI?
Single Program, Multiple Data: un único ejecutable lanzado en N copias, cada una con su rank. Cada proceso decide qué hacer en función de ese identificador.
¿Cuáles son las cuatro funciones MPI de las que no puedes prescindir?
MPI_Init, MPI_Comm_rank, MPI_Comm_size, MPI_Finalize. Definen el «esqueleto» mínimo de cualquier programa MPI.
Diferencia entre MPI_Send y MPI_Ssend.
MPI_Send es el envío estándar: la implementación decide si usa buffer o sincroniza con el receptor. MPI_Ssend obliga a sincronía: no retorna hasta que el receptor empieza a recibir. Es más predecible pero potencialmente más lento.
¿Para qué sirven MPI_Isend e MPI_Wait?
Para comunicación no bloqueante. MPI_Isend arranca el envío y devuelve enseguida un MPI_Request. Permite hacer otro cómputo mientras el mensaje viaja. MPI_Wait bloquea hasta que esa comunicación termina, haciendo el buffer seguro de reutilizar.
¿Qué hace MPI_Reduce y en qué se diferencia de MPI_Allreduce?
MPI_Reduce combina valores de todos los procesos en uno solo (con una operación: suma, max, min...) y deja el resultado en un root elegido. MPI_Allreduce hace lo mismo pero deja el resultado en todos los procesos. Internamente es más eficiente que Reduce+Bcast.
El tiempo de mandar n datos es tcom = ts + n·tw. ¿Qué consecuencia práctica tiene?
Que mandar muchos mensajes pequeños es muy ineficiente por el coste fijo ts. Conviene agrupar (usando tipos derivados, MPI_Pack, o estructuras propias) y mandar pocos mensajes grandes.
¿Qué es un deadlock en MPI y cómo se evita?
Situación en la que dos o más procesos se esperan mutuamente. Típicamente ocurre con Ssend/Send recíprocos. Se evita rompiendo la simetría (uno envía y el otro recibe primero), usando MPI_Sendrecv o usando no bloqueantes.
Repaso de la sesión 2: ¿qué ventaja tiene MPI sobre OpenMP?
Funciona en sistemas con memoria distribuida (clusters). OpenMP solo sirve dentro de un nodo con memoria compartida. MPI escala a miles de nodos sin tener que cambiar el modelo de programación.
Cómo enviar una columna de una matriz almacenada por filas (los elementos no son contiguos).
Tres opciones: (1) crear un MPI_Datatype con MPI_Type_vector que describa el stride; (2) copiar manualmente a un buffer contiguo antes de enviar; (3) usar MPI_Pack/Unpack.
