MPI

El estándar de paso de mensajes. Más explícito que OpenMP —cada comunicación se escribe a mano— pero el único modelo que funciona cuando los procesos viven en máquinas distintas.

De PVM al estándar MPI

Antes de MPI, cada fabricante tenía su propia librería de paso de mensajes —incompatibles entre sí—. PVM (Parallel Virtual Machine, ~1993) fue el primer intento serio de portabilidad. Poco después, el MPI Forum, con participantes de unas 40 organizaciones, consensúa una especificación publicada en mayo de 1994 y actualizada en junio de 1995.

MPI ofrece:

  • Estandarización: una API documentada y estable.
  • Portabilidad: el mismo código compila en multiprocesadores de memoria compartida, clusters, multicomputadores de paso de mensajes y sistemas heterogéneos.
  • Buenas prestaciones cuando hay implementación nativa eficiente.
  • Amplia funcionalidad: del orden de 140 funciones cubriendo desde el «hola mundo» hasta topologías virtuales y E/S paralela.
  • Implementaciones propietarias y libres: mpich, LAM/MPI, OpenMPI. En el laboratorio 2.1 está instalada OpenMPI.
MPI es una especificación, no una librería. Cuando instalas OpenMPI, mpich o Intel MPI lo que cambia es la implementación; la API es la misma. Eso significa que tu código funcionará igual en todas, salvo detalles de cómo se lanzan los procesos.

Filosofía SPMD

Casi todos los programas MPI siguen el modelo SPMD (Single Program, Multiple Data): un único ejecutable, lanzado en múltiples copias (procesos), donde cada proceso decide qué hacer mirando su identificador. No hay maestro «físico» como en OpenMP: todos los procesos son iguales y el rol de cada uno lo determina el código.

SPMD: una sola fuente, múltiples procesos, cada uno con su propio rank y memoria privada. El maestro lógico suele ser rank == 0, pero es una convención del programa, no de MPI.

Compilación y ejecución

La compilación se hace con un wrapper del compilador C que añade las cabeceras y bibliotecas MPI:

$ mpicc programa.c -o programa            # wrapper de gcc/clang
$ mpiicc programa.c -o programa           # si usas Intel ICC
$ mpic++ programa.cpp -o programa         # para C++

El lanzamiento crea los procesos:

$ mpirun -np 4 ./programa                 # 4 procesos, todos en el host actual
$ mpirun -np 8 -hostfile hostfile.txt ./programa

El fichero hostfile.txt indica nodos del cluster y cuántos slots (procesos) admite cada uno. En un solo nodo basta:

localhost slots=6

Con eso, mpirun permite lanzar hasta 6 procesos en local. Si quieres saltarte la limitación, las versiones recientes de OpenMPI admiten el flag --oversubscribe, menos formal pero útil para pruebas.

Importante: -np N indica cuántos procesos arrancar, no cuántos cores hay. Puedes lanzar más procesos que cores —cada proceso es independiente—, pero no escalará: competirán por la CPU. El rendimiento real exige un proceso por core aproximadamente.

Las cuatro funciones obligatorias (esqueleto mínimo)

#include <mpi.h>
#include <stdio.h>

int main(int argc, char *argv[]) {
    int rank, nproc;

    MPI_Init(&argc, &argv);              // 1. arranca MPI
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);  // 2. ¿quién soy?
    MPI_Comm_size(MPI_COMM_WORLD, &nproc); // 3. ¿cuántos somos?

    printf("Hola desde el proceso %d de %d\n", rank, nproc);

    MPI_Finalize();                       // 4. cierra MPI
    return 0;
}

Forma general de las funciones MPI: error = MPI_Nombre(parámetros). Todas devuelven un código de error (MPI_SUCCESS en caso correcto), aunque normalmente se ignora.

El parámetro MPI_COMM_WORLD es el comunicador por defecto: el grupo de todos los procesos arrancados con mpirun. Puedes crear comunicadores más pequeños para subconjuntos (lo veremos abajo), pero para el 90 % del código basta con MPI_COMM_WORLD.

Comunicación punto a punto

La operación más básica: un proceso envía, otro recibe.

// Emisor
MPI_Send(void *buf, int count, MPI_Datatype tipo,
         int destino, int tag, MPI_Comm comm);

// Receptor
MPI_Recv(void *buf, int count, MPI_Datatype tipo,
         int origen, int tag, MPI_Comm comm, MPI_Status *status);

Los parámetros:

  • buf: dirección del buffer (puntero a los datos a enviar/recibir).
  • count: cuántos elementos.
  • tipo: tipo MPI de cada elemento.
  • destino / origen: rank del otro extremo. En la recepción puedes usar MPI_ANY_SOURCE si no te importa quién envió.
  • tag: etiqueta entera para distinguir mensajes entre los mismos pares. MPI_ANY_TAG acepta cualquiera.
  • comm: comunicador (normalmente MPI_COMM_WORLD).
  • status: en Recv, información del mensaje recibido (origen real, tamaño...). Si no la necesitas, MPI_STATUS_IGNORE.

Las cuatro variantes de envío bloqueante

Pregunta clásica de examen. Las cuatro modalidades cambian qué tiene que pasar para que la llamada de envío retorne; todas las versiones de recv retornan cuando el mensaje se ha recibido.

FunciónRetorna cuando…Pros / contras
MPI_Ssend (síncrono) El receptor ha empezado a recibir Sincronización máxima, sin buffer extra. Sobrecarga por sincronización.
MPI_Bsend (con buffer) Los datos se han copiado al buffer suministrado por el usuario Acaba siempre, independiente del receptor. Elimina sobrecarga por sincronización. Requiere reservar buffer (MPI_Buffer_attach).
MPI_Send (estándar) La implementación decide entre síncrono o con buffer Cómodo, pero comportamiento depende del tamaño del mensaje y de la implementación.
MPI_Rsend (ready) Se asume que el recv ya está activo Reduce sobrecarga por sincronización. Si el recv no estaba listo, error: requiere estructura del programa que lo garantice.

Comunicación no bloqueante

Los send/recv anteriores son bloqueantes: la función no retorna hasta que el sistema garantiza que el buffer se puede reutilizar (o que el dato ya llegó). Para solapar cómputo y comunicación, MPI ofrece versiones no bloqueantes con prefijo I (de immediate):

MPI_Request req;
MPI_Irecv(buf, n, MPI_DOUBLE, vecino, tag, MPI_COMM_WORLD, &req);

// hago otro trabajo mientras llega el mensaje...
calcular_otra_cosa();

MPI_Wait(&req, MPI_STATUS_IGNORE);   // ahora sí, espero a que llegue
// el buffer ya es seguro de usar

Funciones útiles del lado no bloqueante:

  • MPI_Isend(buf, count, datatype, dest, tag, comm, &request) y MPI_Irecv(...): arrancan la operación y devuelven un MPI_Request.
  • MPI_Wait(&req, &status): bloquea hasta que la operación se completa.
  • MPI_Test(&req, &flag, &status): pregunta sin bloquear si ya está. Pone flag a no-cero si sí.
  • MPI_Waitall, MPI_Waitany, MPI_Waitsome: para varias request a la vez.
Patrón muy típico en códigos de stencil/difusión: arrancar todos los Irecv al principio del paso, lanzar Isend a los vecinos, hacer el cómputo del interior del dominio (que no necesita los datos de los vecinos), y finalmente Waitall antes de procesar los bordes. Así la comunicación se solapa con el cómputo interior y a efectos prácticos «desaparece».

Comunicaciones colectivas

Cuando intervienen todos los procesos de un comunicador. Más cómoda y, casi siempre, más eficiente que ensamblar la misma operación con Send/Recv manuales (la implementación usa árboles de comunicación log p).

Regla de oro: una operación colectiva la deben invocar todos los procesos del comunicador, en el mismo orden. Si un proceso se la salta, el programa se cuelga.

Otras colectivas no dibujadas: MPI_Barrier (sincronización pura), MPI_Alltoall (intercambio total), MPI_Reduce_scatter, MPI_Scan (reducción prefija).

ATAJO MUY COMÚN

MPI_Reduce y MPI_Allreduce

Si todos los procesos van a necesitar el resultado de la reducción (por ejemplo, el error global en un solver iterativo), usa MPI_Allreduce. Equivale a Reduce + Bcast pero la implementación es notablemente más eficiente (un solo árbol de comunicación bidireccional).

Ejemplo canónico: integral por trapecios

Problema: aproximar ∫ab f(x) dx con la regla del trapecio sobre n subintervalos, repartiendo el trabajo entre p procesos.

Estrategia (slide pág. 15):

  • Cada proceso necesita saber: número total p, su rank, intervalo [a, b] y número de subintervalos n.
  • El proceso 0 lee los datos, los reparte, recibe las integrales parciales y combina el resultado.
  • Cada uno calcula su trozo del intervalo y aplica la regla del trapecio localmente.

Versión con send/recv punto a punto

double integral_local(double a, double b, int n, double h) {
    double s = (f(a) + f(b)) / 2.0, x = a;
    for (int i = 1; i <= n - 1; i++) { x += h; s += f(x); }
    return s * h;
}

int main(int argc, char **argv) {
    int rank, p, n, local_n, source, dest = 0, tag = 50;
    float a, b, h, local_a, local_b, integral, total;
    MPI_Status status;

    MPI_Init(&argc, &argv);
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &p);

    Get_data(rank, p, &a, &b, &n);             // reparte a, b, n a todos
    h = (b - a) / n;
    local_n = n / p;
    local_a = a + rank * local_n * h;
    local_b = local_a + local_n * h;
    integral = integral_local(local_a, local_b, local_n, h);

    if (rank == 0) {                            // el 0 recoge y combina
        total = integral;
        for (source = 1; source < p; source++) {
            MPI_Recv(&integral, 1, MPI_FLOAT, source, tag,
                     MPI_COMM_WORLD, &status);
            total += integral;
        }
        printf("Integral entre %f y %f = %f\n", a, b, total);
    } else {
        MPI_Send(&integral, 1, MPI_FLOAT, dest, tag, MPI_COMM_WORLD);
    }
    MPI_Finalize();
    return 0;
}

Coste de los envíos individuales de Get_data (3 envíos por cada proceso destinatario):

CUENTAS DE COMUNICACIONES

tGet_data, p2p = 3 (p − 1)(ts + tw)

El 0 envía 3 valores a cada uno de los p − 1 demás. Cada envío paga la latencia ts + 1 dato × tw. La suma da el coste indicado.

Versión con colectivas: Bcast + Reduce

void Get_data(int rank, float *a, float *b, int *n) {
    if (rank == 0) {
        printf("Enter a, b, n\n");
        scanf("%f %f %d", a, b, n);
    }
    MPI_Bcast(a, 1, MPI_FLOAT, 0, MPI_COMM_WORLD);
    MPI_Bcast(b, 1, MPI_FLOAT, 0, MPI_COMM_WORLD);
    MPI_Bcast(n, 1, MPI_INT,   0, MPI_COMM_WORLD);
}

// En el main:
MPI_Reduce(&integral, &total, 1, MPI_FLOAT, MPI_SUM,
           0, MPI_COMM_WORLD);

Coste con broadcast: 3 (tsb + twb), donde tsb y twb son las constantes para el árbol de broadcast (logarítmicas en p, no lineales). Diferencia clave: el modelo punto a punto es O(p); el broadcast es O(log p). En clusters con muchos procesos, la diferencia es enorme.

Agrupación de datos

Tres mensajes pequeños cuestan tres latencias ts. Un mensaje grande con los mismos datos paga una sola. Por eso conviene agruparlos. MPI ofrece tres mecanismos:

1 · Array contiguo

Lo más simple. Si los datos a enviar son del mismo tipo, los pones en un array y mandas un único mensaje. Sirve para datos homogéneos.

2 · Tipos derivados

Construyes un nuevo MPI_Datatype que describe la disposición real en memoria. La inversión en crear el tipo se amortiza si lo usas muchas veces.

3 · Pack / Unpack

Tú mismo serializas en un buffer contiguo con MPI_Pack, envías como MPI_PACKED, y el receptor deserializa con MPI_Unpack. Más flexible, menos eficiente.

Tipos derivados: constructores

ConstructorPara qué sirve
MPI_Type_contiguous Datos adyacentes del mismo tipo. Equivale a un «mini-array».
MPI_Type_vector Bloques idénticos igualmente espaciados (con stride). Útil para columnas de matrices almacenadas por filas.
MPI_Type_indexed Bloques de tamaños distintos en posiciones arbitrarias.
MPI_Type_create_struct Estructura heterogénea: cada bloque puede ser de un tipo distinto. El caso general.

Todo tipo derivado pasa por dos fases obligatorias:

  1. Construcción con uno de los constructores anteriores.
  2. Commit con MPI_Type_commit(&tipo) antes de poder usarlo en una comunicación.
  3. Al terminar, MPI_Type_free(&tipo) libera recursos.

Ejemplo: estructura con dos floats y un int

typedef struct { float a, b; int n; } INDATA_TYPE;

void Build_derived_type(INDATA_TYPE *indata, MPI_Datatype *new_type) {
    int         block_lengths[3] = { 1, 1, 1 };
    MPI_Aint    displacements[3], addresses[4];
    MPI_Datatype typelist[3] = { MPI_FLOAT, MPI_FLOAT, MPI_INT };

    // Calcular desplazamientos relativos al inicio de la estructura
    MPI_Get_address(indata,        &addresses[0]);
    MPI_Get_address(&(indata->a), &addresses[1]);
    MPI_Get_address(&(indata->b), &addresses[2]);
    MPI_Get_address(&(indata->n), &addresses[3]);
    displacements[0] = addresses[1] - addresses[0];
    displacements[1] = addresses[2] - addresses[0];
    displacements[2] = addresses[3] - addresses[0];

    MPI_Type_create_struct(3, block_lengths, displacements,
                           typelist, new_type);
    MPI_Type_commit(new_type);
}

// Uso: broadcast de la estructura como un único mensaje
INDATA_TYPE datos;
MPI_Datatype mpi_indata;
Build_derived_type(&datos, &mpi_indata);
MPI_Bcast(&datos, 1, mpi_indata, 0, MPI_COMM_WORLD);

El detalle importante: las direcciones se obtienen con MPI_Get_address, no con el operador & de C, porque MPI necesita un tipo de dirección portable (MPI_Aint) que puede no coincidir con un puntero estándar.

Ejemplo alternativo con Pack/Unpack

char buffer[100];
int position = 0;

if (rank == 0) {
    MPI_Pack(&a, 1, MPI_FLOAT, buffer, 100, &position, MPI_COMM_WORLD);
    MPI_Pack(&b, 1, MPI_FLOAT, buffer, 100, &position, MPI_COMM_WORLD);
    MPI_Pack(&n, 1, MPI_INT,   buffer, 100, &position, MPI_COMM_WORLD);
    MPI_Bcast(buffer, 100, MPI_PACKED, 0, MPI_COMM_WORLD);
} else {
    MPI_Bcast(buffer, 100, MPI_PACKED, 0, MPI_COMM_WORLD);
    MPI_Unpack(buffer, 100, &position, &a, 1, MPI_FLOAT, MPI_COMM_WORLD);
    MPI_Unpack(buffer, 100, &position, &b, 1, MPI_FLOAT, MPI_COMM_WORLD);
    MPI_Unpack(buffer, 100, &position, &n, 1, MPI_INT,   MPI_COMM_WORLD);
}

El position es entrada y salida: MPI lo actualiza tras cada pack/unpack para apuntar al siguiente hueco del buffer. Hay que desempaquetar en el mismo orden en que se empaquetó.

Comunicadores (avanzado)

Un comunicador consta de dos cosas:

  • Un grupo: colección ordenada de procesos a los que se asignan ranks 0…p−1.
  • Un contexto: identificador que el sistema usa internamente para evitar que mensajes de un comunicador interfieran con los de otro.

Además, se le puede asociar una topología virtual (cartesiana o de grafo).

Tipos de comunicadores

  • Intra-comunicadores: los habituales. Para comunicarse dentro de un mismo grupo (MPI_COMM_WORLD es uno).
  • Inter-comunicadores: para comunicar procesos en dos comunicadores distintos. Usado cuando se desarrollan librerías independientes con sus propios comunicadores.

Crear un comunicador a partir de un grupo

Caso: tienes p = q² procesos organizados como malla q×q. Quieres un comunicador con los procesos de la primera fila.

MPI_Group MPI_GROUP_WORLD, first_row_group;
MPI_Comm  first_row_comm;
int *process_ranks = malloc(q * sizeof(int));
for (int i = 0; i < q; i++) process_ranks[i] = i;

MPI_Comm_group(MPI_COMM_WORLD, &MPI_GROUP_WORLD);        // local
MPI_Group_incl(MPI_GROUP_WORLD, q, process_ranks,
               &first_row_group);                          // local
MPI_Comm_create(MPI_COMM_WORLD, first_row_group,
                &first_row_comm);                          // colectiva

Importante: MPI_Comm_create es colectiva sobre el comunicador original. Todos los procesos en MPI_COMM_WORLD deben llamarla, incluso los que no vayan a formar parte del nuevo grupo.

Atajo: MPI_Comm_split

Crea un nuevo comunicador para cada valor distinto de la clave de partición. Los procesos con la misma clave acaban en el mismo comunicador.

int my_row = rank / q;
MPI_Comm my_row_comm;
MPI_Comm_split(MPI_COMM_WORLD, my_row, rank, &my_row_comm);
// Crea q comunicadores nuevos, uno por cada fila de la malla.
// Dentro de cada uno, los ranks se asignan según el orden de la 2.ª clave.

Para excluir un proceso del nuevo comunicador, pásale MPI_UNDEFINED como clave; recibe MPI_COMM_NULL.

Topologías cartesianas (avanzado)

Cuando los procesos representan elementos de una malla (típico en problemas físicos: stencil 2D/3D), pedir a MPI que asocie una topología cartesiana al comunicador permite:

  • Direccionar procesos por coordenadas ((i, j)) en lugar de ranks lineales.
  • Que el runtime reordene la asignación de procesos a nodos físicos para que vecinos lógicos queden cerca.
  • Particionar la malla en sub-mallas con una llamada.
int dims[2]    = { q, q };       // malla q × q
int periods[2] = { 0, 0 };       // no circular en ninguna dim
int reorder    = 1;              // permite reordenamiento físico
MPI_Comm grid;

MPI_Cart_create(MPI_COMM_WORLD, 2, dims, periods, reorder, &grid);

int coords[2], rank_grid;
MPI_Comm_rank(grid, &rank_grid);
MPI_Cart_coords(grid, rank_grid, 2, coords);  // rank → (i, j)
MPI_Cart_rank  (grid, coords, &rank_grid);     // (i, j) → rank

Para extraer una sub-malla (por ejemplo, todos los procesos de la misma fila):

int remain_dims[2] = { 0, 1 };   // retén la segunda dim, descarta la primera
MPI_Comm row_comm;
MPI_Cart_sub(grid, remain_dims, &row_comm);
// Crea q comunicadores, uno por fila. Es colectiva.

Modelo simple de tiempo de comunicación

Aparece una y otra vez en el análisis de algoritmos. El tiempo de mandar n datos es aproximadamente:

FÓRMULA QUE HAY QUE RECORDAR

tcom = ts + n · tw

ts (startup): coste fijo por iniciar una comunicación, independiente del tamaño. Latencia.

tw (word time): coste por unidad de dato. Inverso del ancho de banda.

Implicación práctica: muchos mensajes pequeños son catastróficos. Si tienes que enviar k cantidades, fusionarlas en un solo mensaje cambia el coste de k(ts + tw) a ts + k·tw —ahorras (k − 1) latencias—.

Para colectivas como broadcast o reduce, las implementaciones eficientes usan árboles binarios o «recursive doubling», con coste O(log p) en lugar del O(p) de un bucle ingenuo. Las fórmulas detalladas se ven en la sesión 6.

Cuidado con los deadlocks

El error clásico:

// AMBOS PROCESOS EJECUTAN ESTO SIMULTÁNEAMENTE
if (rank == 0) {
    MPI_Ssend(buf, n, MPI_DOUBLE, 1, tag, comm);   // espera al recv del 1
    MPI_Recv (buf, n, MPI_DOUBLE, 1, tag, comm, ...);
} else if (rank == 1) {
    MPI_Ssend(buf, n, MPI_DOUBLE, 0, tag, comm);   // espera al recv del 0
    MPI_Recv (buf, n, MPI_DOUBLE, 0, tag, comm, ...);
}

Ambos esperan que el otro reciba primero: deadlock. Soluciones:

  • Asimetría explícita: el rank par envía y luego recibe; el impar al revés.
  • MPI_Sendrecv: una sola llamada que combina envío y recepción de forma libre de deadlock.
  • No bloqueantes: MPI_Isend retorna inmediatamente; el deadlock no aparece.

Con MPI_Send (estándar) el deadlock puede no manifestarse para mensajes pequeños (la implementación los pone en buffer) pero sí para mensajes grandes. Bug clásicamente «funciona en mi máquina» que estalla al subir el problema.

Guía resuelta: ejercicios de la Semana 4

Ejercicio 1 · ampliar hello.c con saludo circular

Enunciado: que al final cada proceso i mande un saludo al proceso (i+1) % p y, tras recibirlo, lo muestre.

Razonamiento: patrón anillo. Cada proceso envía al sucesor y recibe del predecesor (que es (i-1+p) % p). Con Ssend habría deadlock; con Send «estándar» puede funcionar con mensajes pequeños pero es frágil. Solución limpia: o Sendrecv, o asimetría explícita, o no bloqueantes.

int dest = (rank + 1) % p;
int src  = (rank - 1 + p) % p;
char out[100], in[100];
int tag = 99;

sprintf(out, "saludo del proceso %d", rank);

// Opción A: Sendrecv (recomendada — libre de deadlock)
MPI_Sendrecv(out, strlen(out)+1, MPI_CHAR, dest, tag,
             in,  100,             MPI_CHAR, src,  tag,
             MPI_COMM_WORLD, &status);

printf("proceso %d recibió: \"%s\"\n", rank, in);

Con Sendrecv, MPI se encarga de orquestar las dos operaciones simultáneamente sin riesgo de bloqueo. Los buffers de envío y recepción deben ser distintos.

Ejercicio 2 · ampliar broadcast.c con cadena de 10 caracteres

Enunciado: además de a, b, n, distribuir una cadena de 10 caracteres.

Razonamiento: un nuevo Bcast sobre un array de char con tipo MPI_CHAR y count = 10.

char cadena[10];

if (rank == 0) {
    printf("Cadena (max 9 caracteres): ");
    scanf("%9s", cadena);                // %9s para dejar sitio al \0
}

MPI_Bcast(&a,     1,  MPI_FLOAT, 0, MPI_COMM_WORLD);
MPI_Bcast(&b,     1,  MPI_FLOAT, 0, MPI_COMM_WORLD);
MPI_Bcast(&n,     1,  MPI_INT,   0, MPI_COMM_WORLD);
MPI_Bcast(cadena, 10, MPI_CHAR,  0, MPI_COMM_WORLD);   // nuevo

if (rank != 0)
    printf("P%d: a=%f b=%f n=%d cadena=%s\n", rank, a, b, n, cadena);

Mejora opcional: agruparlo todo con MPI_Pack o un tipo derivado para un único Bcast; pagas una latencia en lugar de cuatro.

Ejercicio 3 · ampliar reduce.c con máximo por posición

Enunciado: además de la suma, calcular el máximo de cada posición de los arrays locales.

Razonamiento: MPI_Reduce opera elemento a elemento cuando count > 1. Basta una segunda reducción con MPI_MAX sobre el mismo array.

int *suma   = NULL;
int *maximo = NULL;
if (rank == 0) {
    suma   = malloc(TAMA * sizeof(int));
    maximo = malloc(TAMA * sizeof(int));
}

MPI_Reduce(mis_datos, suma,   TAMA, MPI_INT, MPI_SUM, 0, MPI_COMM_WORLD);
MPI_Reduce(mis_datos, maximo, TAMA, MPI_INT, MPI_MAX, 0, MPI_COMM_WORLD);

if (rank == 0) {
    printf("Sumas: ");   escribe(suma,   TAMA);
    printf("Máximos: "); escribe(maximo, TAMA);
    free(suma); free(maximo);
}

Con los arrays del PDF (0..9 en P0, 10000..10009 en P1, 20000..20009 en P2), el máximo posición por posición es el array del proceso con rank mayor: 20000, 20001, …, 20009.

Ejercicio 4 (opcional) · MPI_Type_create_struct

Enunciado: completar el código de la pág. 36 del PDF para que sea ejecutable, comprobando que el tipo derivado se construye correctamente y los procesos reciben los datos por broadcast.

#include <mpi.h>
#include <stdio.h>

typedef struct { float a, b; int n; } INDATA_TYPE;

void Build_derived_type(INDATA_TYPE *indata, MPI_Datatype *new_type) {
    int          bl[3]   = { 1, 1, 1 };
    MPI_Aint     disp[3], addr[4];
    MPI_Datatype tl[3]   = { MPI_FLOAT, MPI_FLOAT, MPI_INT };

    MPI_Get_address(indata,        &addr[0]);
    MPI_Get_address(&(indata->a), &addr[1]);
    MPI_Get_address(&(indata->b), &addr[2]);
    MPI_Get_address(&(indata->n), &addr[3]);
    disp[0] = addr[1] - addr[0];
    disp[1] = addr[2] - addr[0];
    disp[2] = addr[3] - addr[0];

    MPI_Type_create_struct(3, bl, disp, tl, new_type);
    MPI_Type_commit(new_type);
}

int main(int argc, char **argv) {
    MPI_Init(&argc, &argv);
    int rank;
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);

    INDATA_TYPE datos;
    if (rank == 0) {
        printf("Enter a, b, n\n");
        scanf("%f %f %d", &datos.a, &datos.b, &datos.n);
    }

    MPI_Datatype tipo;
    Build_derived_type(&datos, &tipo);
    MPI_Bcast(&datos, 1, tipo, 0, MPI_COMM_WORLD);
    MPI_Type_free(&tipo);

    if (rank != 0)
        printf("P%d: a=%f b=%f n=%d\n", rank, datos.a, datos.b, datos.n);

    MPI_Finalize();
    return 0;
}

Punto crítico: si el proceso 0 construye el tipo derivado después de leer los datos, y los demás antes de recibir, el resultado es el mismo —los desplazamientos solo dependen de la disposición de la estructura, no de los valores—. Pero cada proceso debe construir su tipo (es local), y todos deben hacer commit antes de la primera comunicación con él.

Práctica de recuperación

El examen del curso suele incluir preguntas concretas de sintaxis y de razonamiento sobre costes. Si te equivocas en una tarjeta, marca esa idea para revisar mañana —recordar tras un fallo refuerza más que repasar lo que ya sabes—.

¿Qué significa SPMD en el contexto de MPI?
Single Program, Multiple Data: un único ejecutable lanzado en N copias, cada una con su rank. El comportamiento por proceso lo determina el código a partir del rank.
¿Cuáles son las cuatro funciones MPI de las que no puedes prescindir?
MPI_Init, MPI_Comm_rank, MPI_Comm_size, MPI_Finalize. Definen el esqueleto mínimo de cualquier programa MPI.
Diferencia entre MPI_Send, MPI_Ssend, MPI_Bsend y MPI_Rsend.
Ssend sincroniza: retorna cuando el receptor empieza a recibir. Bsend copia a un buffer de usuario y retorna; necesita Buffer_attach. Send es estándar: la implementación elige entre los anteriores según el tamaño. Rsend asume que el Recv ya está activo (es el más rápido pero el más peligroso).
¿Para qué sirven MPI_Isend e MPI_Wait?
Para comunicación no bloqueante. Isend arranca el envío y devuelve enseguida un MPI_Request. Permite hacer otro cómputo mientras el mensaje viaja. Wait bloquea hasta que esa comunicación termina, haciendo el buffer seguro de reutilizar.
¿Qué hace MPI_Reduce y en qué se diferencia de MPI_Allreduce?
Reduce combina valores de todos los procesos en uno solo (suma, max, min…) y deja el resultado en un root elegido. Allreduce hace lo mismo pero deja el resultado en todos los procesos. Internamente es más eficiente que Reduce+Bcast.
El tiempo de mandar n datos es tcom = ts + n·tw. ¿Qué consecuencia práctica tiene?
Que mandar muchos mensajes pequeños es muy ineficiente por el coste fijo ts. Conviene agrupar (con array contiguo, tipos derivados o MPI_Pack) y mandar pocos mensajes grandes. Pasar de k(ts+tw) a ts+k·tw ahorra (k−1) latencias.
¿Cuándo se manifiesta un deadlock por Send/Recv y cómo se evita?
Cuando dos procesos hacen Ssend mutuo antes de Recv: ambos esperan que el otro reciba primero. Con Send estándar puede parecer que funciona con mensajes pequeños y fallar al crecer. Soluciones: asimetría (uno envía, otro recibe primero), MPI_Sendrecv, o no bloqueantes.
¿Qué hace MPI_Sendrecv?
Una llamada combinada: envía a un destino y recibe de una fuente en una sola operación, libre de deadlock. Los buffers de envío y recepción deben ser distintos.
Cómo enviar una columna de una matriz almacenada por filas (los elementos no son contiguos).
Tres opciones: (1) construir un MPI_Datatype con MPI_Type_vector que describa el stride; (2) copiar manualmente a un buffer contiguo antes de enviar; (3) usar MPI_Pack/Unpack.
¿Qué dos cosas tiene que hacer el programador con un tipo derivado antes de poder usarlo?
Construirlo (con Type_contiguous/vector/indexed/create_struct) y hacerle MPI_Type_commit. Al final del programa, conviene liberarlo con MPI_Type_free.
¿Por qué se usa MPI_Get_address en lugar del operador & de C al construir un tipo derivado?
Porque MPI necesita representar direcciones con su propio tipo portable (MPI_Aint), que en algunos sistemas no coincide con un puntero estándar. MPI_Get_address garantiza que los desplazamientos calculados son válidos para MPI.
¿Qué hace MPI_Comm_split(comm, color, key, &new_comm)?
Reparte los procesos del comunicador en sub-comunicadores: todos los procesos con el mismo color acaban juntos. La key ordena los ranks dentro del nuevo comunicador. Es colectiva sobre comm.
Repaso de la sesión 2: ¿qué ventaja tiene MPI sobre OpenMP?
Funciona en sistemas con memoria distribuida (clusters). OpenMP solo sirve dentro de un nodo con memoria compartida. MPI escala a miles de nodos sin cambiar de modelo de programación, a cambio de hacer toda la comunicación explícita.
