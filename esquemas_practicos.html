1 · Paralelismo de datos

La misma operación sobre cada elemento. Es el esquema más simple y el más usado.

Problema. Calcula C[i] = A[i] + B[i] para i = 0..N−1, con N grande (millones).

1.a — Solución con OpenMP

#include <stdio.h>
#include <stdlib.h>
#include <omp.h>

int main() {
    int    N = 1000000;
    double *A = malloc(N * sizeof(double));
    double *B = malloc(N * sizeof(double));
    double *C = malloc(N * sizeof(double));

    for (int i = 0; i < N; i++) { A[i] = i; B[i] = 2*i; }

    #pragma omp parallel for schedule(static)
    for (int i = 0; i < N; i++) {
        C[i] = A[i] + B[i];
    }

    printf("C[1000] = %f\n", C[1000]);
    free(A); free(B); free(C);
}
Lectura línea por línea
  • malloc(N * sizeof(double)) → reserva memoria para N doubles. sizeof(double) son 8 bytes; total 8·N bytes. Devuelve un puntero.
  • #pragma omp parallel for → fusiona dos directivas: parallel (abre la región con varios hilos) + for (reparte las iteraciones del bucle siguiente). Es la forma más común y compacta.
  • schedule(static) → reparto en bloques contiguos: si hay 4 hilos y N=1M, hilo 0 hace [0..249.999], hilo 1 [250.000..499.999], etc. Static es óptimo cuando cada iteración tarda igual (aquí, todas son una suma).
  • La variable i del bucle es automáticamente private por OpenMP en una directiva for. No hay que declararlo.
  • Las variables A, B, C, N son shared por defecto. ¿Por qué es seguro? Porque cada hilo escribe en posiciones distintas de C (su rango de i) y solo lee de A y B. No hay carrera.
¿Por qué funciona? No hay dependencia entre iteraciones: el cálculo de C[5] no necesita C[4]. Esa es la condición para usar paralelismo de datos puro.

1.b — Misma solución con MPI

#include <mpi.h>

int main(int argc, char **argv) {
    int rank, size;
    int N = 1000000;

    MPI_Init(&argc, &argv);
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);
    MPI_Comm_size(MPI_COMM_WORLD, &size);

    int local_n = N / size;                 // asumimos N divisible por size
    double *A_local = malloc(local_n * sizeof(double));
    double *B_local = malloc(local_n * sizeof(double));
    double *C_local = malloc(local_n * sizeof(double));
    double *A = NULL, *B = NULL, *C = NULL;

    if (rank == 0) {
        A = malloc(N * sizeof(double));
        B = malloc(N * sizeof(double));
        C = malloc(N * sizeof(double));
        for (int i = 0; i < N; i++) { A[i] = i; B[i] = 2*i; }
    }

    MPI_Scatter(A, local_n, MPI_DOUBLE,
                A_local, local_n, MPI_DOUBLE,
                0, MPI_COMM_WORLD);
    MPI_Scatter(B, local_n, MPI_DOUBLE,
                B_local, local_n, MPI_DOUBLE,
                0, MPI_COMM_WORLD);

    for (int i = 0; i < local_n; i++)
        C_local[i] = A_local[i] + B_local[i];

    MPI_Gather(C_local, local_n, MPI_DOUBLE,
               C,       local_n, MPI_DOUBLE,
               0, MPI_COMM_WORLD);

    if (rank == 0) printf("C[1000] = %f\n", C[1000]);

    MPI_Finalize();
}
Funciones nuevas, parámetro a parámetro
  • MPI_Scatter(sendbuf, sendcount, sendtype, recvbuf, recvcount, recvtype, root, comm):
    • sendbuf = puntero al array completo (solo importa en el root); NULL en los demás.
    • sendcount = cuántos elementos manda a cada proceso (no el total).
    • sendtype = tipo MPI de cada elemento (MPI_DOUBLE).
    • recvbuf = dónde guarda cada proceso (incluido el root) su trozo.
    • recvcount, recvtype = simétricos al send.
    • root = quién es el que reparte (aquí 0).
    • comm = comunicador.
  • MPI_Gather = exacta inversa de Scatter. El root junta los trozos.
Comparación clave OpenMP vs MPI. En OpenMP no hay reparto explícito de datos: la memoria es compartida. En MPI hay que mover los datos (Scatter/Gather) porque cada proceso solo ve su trozo. Más código, pero escala a clusters.

2 · Síncrono iterativo (Jacobi / ecuación del calor)

Iteramos un patrón estable; cada paso depende del anterior. Barrera entre pasos.

Problema. Distribución de temperatura en una barra de N puntos. En cada paso, la nueva temperatura de cada punto interior es la media de sus dos vecinos: U_new[i] = (U_old[i-1] + U_old[i+1]) / 2. Repetir hasta convergencia.

2.a — Versión OpenMP

double U_old[N], U_new[N];

// Inicialización: extremos fijos a T calientes/frías
U_old[0] = 100.0; U_old[N-1] = 0.0;

for (int paso = 0; paso < PASOS; paso++) {

    #pragma omp parallel for
    for (int i = 1; i < N-1; i++)
        U_new[i] = 0.5 * (U_old[i-1] + U_old[i+1]);

    // barrera implícita al final del parallel for

    #pragma omp parallel for
    for (int i = 1; i < N-1; i++)
        U_old[i] = U_new[i];
}
Por qué dos bucles y por qué dos arrays
  • Doble buffer: usamos U_old para leer y U_new para escribir. Si usásemos un solo array, al actualizar U[5] ya no tendríamos el valor antiguo para calcular U[6].
  • El primer parallel for calcula todos los U_new en paralelo. La barrera implícita al cerrar la región paralela garantiza que todos los U_new están listos antes de copiar.
  • El segundo parallel for copia U_new a U_old para la siguiente iteración. (En la práctica se intercambian los punteros, no se copia, pero conceptualmente es esto.)
  • Si los hilos no se sincronizaran entre bucles, unos seguirían calculando con valores del paso anterior y otros con valores nuevos: caos.

2.b — Versión MPI con halos (la versión que cae en examen)

int rank, size;
MPI_Init(&argc, &argv);
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

int local_n = N / size;
// Reservamos local_n + 2: una celda fantasma a cada lado (halo)
double U_old[local_n + 2], U_new[local_n + 2];

for (int paso = 0; paso < PASOS; paso++) {

    // 1. Intercambio de fronteras con vecinos
    int izq = rank - 1;
    int der = rank + 1;
    if (izq < 0)      izq = MPI_PROC_NULL;
    if (der >= size)  der = MPI_PROC_NULL;

    MPI_Sendrecv(&U_old[1],         1, MPI_DOUBLE, izq, 0,
                 &U_old[local_n+1], 1, MPI_DOUBLE, der, 0,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);

    MPI_Sendrecv(&U_old[local_n],   1, MPI_DOUBLE, der, 1,
                 &U_old[0],         1, MPI_DOUBLE, izq, 1,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);

    // 2. Actualización local
    for (int i = 1; i <= local_n; i++)
        U_new[i] = 0.5 * (U_old[i-1] + U_old[i+1]);

    // 3. Copia para siguiente iteración
    for (int i = 1; i <= local_n; i++)
        U_old[i] = U_new[i];
}
Conceptos nuevos: halo y MPI_Sendrecv
  • Halo (celdas fantasma): el proceso 1 necesita el último valor del proceso 0 (su vecino izquierdo) y el primero del proceso 2 (su vecino derecho). Reservamos 2 celdas extra (índices 0 y local_n+1) para guardarlos. El cómputo se hace de i=1 a i=local_n; los halos se rellenan por comunicación.
  • MPI_Sendrecv(sendbuf, sc, st, dest, stag, recvbuf, rc, rt, source, rtag, comm, status) → envía y recibe en una sola llamada, evitando deadlocks. Parámetros:
    • sendbuf, sc, st, dest, stag = qué/cuánto/tipo/a quién/etiqueta de envío.
    • recvbuf, rc, rt, source, rtag = mismo para recepción.
    • El sistema garantiza que no se bloquea aunque todos los procesos lo hagan a la vez.
  • MPI_PROC_NULL → "vecino nulo": comunicación que no hace nada, no produce error. Útil para los extremos (rank 0 no tiene vecino izquierdo).
  • MPI_STATUS_IGNORE → no nos interesa la información del status (de quién y qué tag llegó); ahorramos memoria.
¿Por qué dos Sendrecv y no uno? El primero intercambia el extremo izquierdo del rango (manda al vecino izquierdo, recibe del derecho). El segundo, simétrico. Si lo hicieras con dos MPI_Send seguidos sin recv en medio, se bloquearía con mensajes grandes.

3 · Paralelismo relajado (asíncrono)

Como el síncrono, pero sin barreras. Cada proceso usa los datos más recientes que tenga.

Problema. Igual que Jacobi, pero usando Gauss-Seidel: cuando actualizas U[i], ya usas el U[i-1] recién calculado en esta misma iteración. Converge más rápido cuando el algoritmo lo permite. 
// Versión OpenMP relajada: un solo array, sin barrera
double U[N];

for (int paso = 0; paso < PASOS; paso++) {
    #pragma omp parallel for schedule(static)
    for (int i = 1; i < N-1; i++) {
        U[i] = 0.5 * (U[i-1] + U[i+1]);
    }
}
Qué cambia respecto al síncrono
  • Un solo array: no hay U_old ni U_new. Cuando un hilo lee U[i-1], puede que ya esté actualizado por otro hilo (valor nuevo) o no (valor antiguo). Aceptamos la incertidumbre.
  • No hay barrera interna. La directiva parallel for sigue teniendo una barrera al final, pero dentro del bucle los hilos no se sincronizan: el que va más rápido lee lo que haya en ese momento.
  • Resultado no determinista: dos ejecuciones pueden converger por caminos distintos al mismo valor final. Aceptable en métodos numéricos con garantías de convergencia.
¿Cuándo se permite? Solo cuando el algoritmo converge igual al usar valores ligeramente desactualizados. Métodos iterativos de Gauss-Seidel, optimización por gradiente paralelo, simulated annealing. No es válido en problemas que dependen del orden estricto.

4 · Grafo de dependencias (reducción en árbol)

Tareas con dependencias arbitrarias formando un DAG. Ejemplo canónico: la reducción en árbol.

Problema. Sumar N números en log2(N) pasos en lugar de N. En el primer paso, los pares de números se suman. En el segundo paso, los pares de sumas parciales. Y así sucesivamente. 
// Reducción en árbol con MPI: cada paso involucra la mitad de procesos
int rank, size;
double mi_valor = ...;   // cada proceso tiene un valor inicial

for (int paso = 1; paso < size; paso *= 2) {
    if ((rank % (2*paso)) == 0) {
        // Soy "receptor" en este paso
        if (rank + paso < size) {
            double recibido;
            MPI_Recv(&recibido, 1, MPI_DOUBLE, rank + paso, 0,
                     MPI_COMM_WORLD, MPI_STATUS_IGNORE);
            mi_valor += recibido;
        }
    } else if ((rank % paso) == 0) {
        // Soy "emisor" en este paso
        MPI_Send(&mi_valor, 1, MPI_DOUBLE, rank - paso, 0, MPI_COMM_WORLD);
        break;
    }
}

if (rank == 0)
    printf("Suma total: %f\n", mi_valor);
Cómo razonar este código (con 8 procesos)
  • Paso 1 (paso=1): pares emiten a su vecino par. (0←1, 2←3, 4←5, 6←7). Quedan vivos los procesos 0, 2, 4, 6.
  • Paso 2 (paso=2): el 2 manda al 0, el 6 al 4. Quedan vivos 0 y 4.
  • Paso 3 (paso=4): el 4 manda al 0. Queda solo el 0 con la suma total.
  • rank % (2*paso) == 0 identifica al receptor: los ranks múltiplos de 2, 4, 8...
  • rank + paso < size es la protección por si N no es potencia de 2 (no hay vecino que mande).
  • break en el emisor → ya entregó su suma, no participa en siguientes pasos.
Conclusión. En lugar de N−1 sumas en serie, hacemos log₂(N) pasos, cada uno con varias sumas en paralelo. Es el algoritmo que hay dentro de MPI_Reduce.

5 · Pipeline (segmentación)

Flujo de datos que pasa por etapas en cascada. Cada etapa procesa y pasa al siguiente.

Problema. Procesar un flujo de imágenes en 3 etapas: (1) leer, (2) aplicar filtro de bordes, (3) escribir comprimida. Cada etapa la ejecuta un proceso MPI distinto. 
int rank, size;
MPI_Init(&argc, &argv);
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);    // asumimos size == 3

int NUM_IMAGENES = 100;
double buf[TAM_IMAGEN];

if (rank == 0) {
    // ===== Etapa 1: lectura =====
    for (int k = 0; k < NUM_IMAGENES; k++) {
        leer_imagen(k, buf);
        MPI_Send(buf, TAM_IMAGEN, MPI_DOUBLE, 1, 0, MPI_COMM_WORLD);
    }
}
else if (rank == 1) {
    // ===== Etapa 2: filtro =====
    for (int k = 0; k < NUM_IMAGENES; k++) {
        MPI_Recv(buf, TAM_IMAGEN, MPI_DOUBLE, 0, 0,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
        filtro_bordes(buf);
        MPI_Send(buf, TAM_IMAGEN, MPI_DOUBLE, 2, 0, MPI_COMM_WORLD);
    }
}
else if (rank == 2) {
    // ===== Etapa 3: escritura =====
    for (int k = 0; k < NUM_IMAGENES; k++) {
        MPI_Recv(buf, TAM_IMAGEN, MPI_DOUBLE, 1, 0,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
        escribir_imagen(k, buf);
    }
}

MPI_Finalize();
Cómo se comporta el pipeline en el tiempo
  • Llenado: en t=1 solo trabaja la etapa 1 (lee imagen 0). En t=2, etapa 1 lee la 1 y etapa 2 filtra la 0. En t=3, etapa 1 lee la 2, etapa 2 filtra la 1, etapa 3 escribe la 0. Las 3 etapas ya trabajan en paralelo.
  • Régimen permanente: las 3 etapas procesan a la vez 3 imágenes distintas del flujo. Throughput = 1 / (etapa más lenta).
  • Vaciado: al final, etapa 1 termina, luego etapa 2, luego etapa 3.
  • Cada proceso solo conoce a su vecino izquierdo y derecho en la cadena, no a los demás.
  • El tag 0 es irrelevante aquí (solo un tipo de mensaje); en pipelines complejos se usa para distinguir paquetes de control de paquetes de datos.
Cuello de botella. Si la etapa 2 tarda el doble que las otras, el pipeline al completo va al ritmo de la 2. Soluciones: replicar la etapa lenta (dos procesos haciendo filtro alternativamente) o reorganizar trabajo entre etapas.

6 · Divide y vencerás (mergesort paralelo)

Recursivo: divide, resuelve las partes en paralelo, combina. Tareas OpenMP.

Problema. Ordenar un array de N elementos. Mergesort: divides en dos mitades, ordenas cada una recursivamente, mezclas. 
#define UMBRAL 10000

void mergesort(int *v, int ini, int fin) {
    if (fin - ini <= 1) return;

    int mid = (ini + fin) / 2;

    if (fin - ini > UMBRAL) {
        #pragma omp task shared(v)
        mergesort(v, ini, mid);
        #pragma omp task shared(v)
        mergesort(v, mid, fin);
        #pragma omp taskwait
    } else {
        // Subproblema pequeño: resolver secuencial, no merece crear tareas
        mergesort(v, ini, mid);
        mergesort(v, mid, fin);
    }

    merge(v, ini, mid, fin);
}

int main() {
    int v[N];
    // ...llenar v...

    #pragma omp parallel
    {
        #pragma omp single
        mergesort(v, 0, N);
    }
}
Por qué la estructura es esta
  • #pragma omp parallel + single al inicio: la región paralela crea el pool de hilos, pero solo uno (cualquiera) llama a mergesort. Los demás esperan, listos para coger tareas que se creen dentro.
  • #pragma omp task → encola una tarea (la llamada recursiva). Cualquier hilo del pool la cogerá.
  • shared(v) → el array es compartido por todas las tareas. Cada tarea trabaja sobre un rango distinto de índices, así que no hay carrera.
  • #pragma omp taskwait → espera a que terminen las dos tareas hijas antes de llamar a merge (que necesita las dos mitades ordenadas).
  • UMBRAL de granularidad: si el subproblema es pequeño (< 10.000 elementos), no creamos tarea — el coste de planificación supera al de ordenar directamente.
Sin umbral, ¿qué pasa? Crearíamos miles de tareas de 1 elemento. El coste de crear y planificar tareas dominaría el tiempo. Con umbral, las hojas del árbol de recursión se resuelven secuencial y las ramas grandes en paralelo. Es la clave para que D&V paralelo gane.

7 · Programación dinámica (Floyd-Warshall)

Tabla con dependencias entre celdas. Paralelizamos las celdas independientes en cada frente.

Problema. Caminos mínimos entre todos los pares de nodos de un grafo de N vértices. Algoritmo: para cada vértice k, actualizar d[i][j] = min(d[i][j], d[i][k] + d[k][j]) para todo i,j. 
double d[N][N];   // matriz de distancias, inicializada con el grafo

for (int k = 0; k < N; k++) {
    #pragma omp parallel for collapse(2)
    for (int i = 0; i < N; i++) {
        for (int j = 0; j < N; j++) {
            double via_k = d[i][k] + d[k][j];
            if (via_k < d[i][j])
                d[i][j] = via_k;
        }
    }
}
Qué se puede paralelizar y qué no
  • El bucle externo en k es secuencial: cada iteración usa la matriz actualizada por la anterior. Si paralelizásemos k, romperíamos las dependencias.
  • Los bucles internos en i y j son paralelos: dentro de un mismo k, las celdas d[i][j] se calculan independientemente. La columna k y la fila k se leen pero no se modifican en este paso (propiedad de Floyd-Warshall).
  • collapse(2) → "trata los dos bucles anidados como uno solo de N·N iteraciones para repartir". Da mejor balance que paralelizar solo el externo.
  • via_k es local a cada iteración → privada por scope, no hay carrera.
Frente de onda. En general, en PD identificamos las celdas que dependen solo de celdas ya calculadas. Esas celdas son el frente de onda y se pueden procesar en paralelo. El bucle que avanza de un frente al siguiente debe quedarse secuencial.

8 · Trabajadores replicados (bolsa de tareas)

Maestro reparte tareas bajo demanda. Útil cuando las tareas tardan muy distinto.

Problema. Calcular el conjunto de Mandelbrot: para cada píxel (x,y), iterar una fórmula hasta divergir. Algunos píxeles convergen rápido, otros tardan miles de iteraciones. Carga muy desbalanceada. 
int rank, size;
MPI_Init(&argc, &argv);
MPI_Comm_rank(MPI_COMM_WORLD, &rank);
MPI_Comm_size(MPI_COMM_WORLD, &size);

int NUM_TAREAS = 1000;
int TAG_TAREA = 1, TAG_RESULTADO = 2, TAG_FIN = 3;

if (rank == 0) {
    // ============ MAESTRO ============
    int siguiente = 0;
    int entregadas = 0;

    // 1. Entregar una tarea inicial a cada trabajador
    for (int w = 1; w < size; w++) {
        MPI_Send(&siguiente, 1, MPI_INT, w, TAG_TAREA, MPI_COMM_WORLD);
        siguiente++;
    }

    // 2. Cuando uno entrega resultado, le doy otra tarea
    while (entregadas < NUM_TAREAS) {
        int resultado;
        MPI_Status st;
        MPI_Recv(&resultado, 1, MPI_INT, MPI_ANY_SOURCE,
                 TAG_RESULTADO, MPI_COMM_WORLD, &st);
        entregadas++;
        int worker = st.MPI_SOURCE;

        if (siguiente < NUM_TAREAS) {
            MPI_Send(&siguiente, 1, MPI_INT, worker, TAG_TAREA, MPI_COMM_WORLD);
            siguiente++;
        } else {
            // Ya no hay tareas: aviso de fin a ese trabajador
            MPI_Send(NULL, 0, MPI_INT, worker, TAG_FIN, MPI_COMM_WORLD);
        }
    }
} else {
    // ============ TRABAJADOR ============
    while (1) {
        int tarea;
        MPI_Status st;
        MPI_Recv(&tarea, 1, MPI_INT, 0, MPI_ANY_TAG, MPI_COMM_WORLD, &st);

        if (st.MPI_TAG == TAG_FIN) break;

        int resultado = calcular_mandelbrot_linea(tarea);
        MPI_Send(&resultado, 1, MPI_INT, 0, TAG_RESULTADO, MPI_COMM_WORLD);
    }
}

MPI_Finalize();
El patrón maestro-trabajador, paso a paso
  • El maestro no calcula nada, solo coordina. Mantiene un contador siguiente de la próxima tarea por entregar y entregadas de las completadas.
  • Arranque: el maestro da una tarea a cada trabajador para que arranquen todos a la vez.
  • Bucle principal: recibe el primer resultado que llegue (de quien sea). MPI_ANY_SOURCE permite no saber quién va a contestar antes; st.MPI_SOURCE nos dice quién fue.
  • Reparto bajo demanda: al trabajador que acaba de entregar, le doy la siguiente tarea pendiente. Si ya no quedan, le mando un mensaje con TAG_FIN.
  • El trabajador recibe con MPI_ANY_TAG porque puede llegarle TAG_TAREA o TAG_FIN. Mira st.MPI_TAG para decidir si continúa o sale.
Estructura / campoSignificado
MPI_Status stEstructura que rellena MPI tras un Recv. Contiene info del mensaje real.
st.MPI_SOURCERank de quien envió.
st.MPI_TAGEtiqueta con la que se envió.
MPI_ANY_SOURCEConstante: acepta de cualquier remitente.
MPI_ANY_TAGConstante: acepta cualquier etiqueta.
¿Por qué balance perfecto? Ningún trabajador se queda parado mientras haya tareas: en cuanto termina, pide otra. Las tareas caras y baratas se reparten automáticamente. El precio es la sobrecarga de comunicación con el maestro, que solo merece la pena si las tareas son lo bastante grandes.

9 · Branch & bound (mochila 0/1)

Búsqueda en árbol con poda. Cota global que se mejora y se difunde.

Problema. Mochila 0/1: tenemos N objetos con peso wi y valor vi, capacidad W. Maximizar el valor sin pasarse de peso. Cada nodo del árbol decide "meter o no meter" el siguiente objeto. 
// Esquema (pseudo-código MPI maestro-trabajador con cota global)

int cota_global = 0;   // mejor valor encontrado hasta ahora

void explorar_subarbol(Nodo n) {
    if (n.cota_optimista <= cota_global) return;   // PODA

    if (n.es_hoja()) {
        if (n.valor > cota_global) {
            cota_global = n.valor;
            // Difundir la mejora a otros procesos:
            for (int p = 0; p < size; p++)
                if (p != rank)
                    MPI_Send(&cota_global, 1, MPI_INT, p,
                             TAG_COTA, MPI_COMM_WORLD);
        }
        return;
    }

    explorar_subarbol(n.con_objeto_actual());
    explorar_subarbol(n.sin_objeto_actual());
}

// Cada proceso, periódicamente, comprueba mensajes de cota entrantes:
int flag;
MPI_Status st;
MPI_Iprobe(MPI_ANY_SOURCE, TAG_COTA, MPI_COMM_WORLD, &flag, &st);
if (flag) {
    int nueva_cota;
    MPI_Recv(&nueva_cota, 1, MPI_INT, st.MPI_SOURCE, TAG_COTA,
             MPI_COMM_WORLD, MPI_STATUS_IGNORE);
    if (nueva_cota > cota_global) cota_global = nueva_cota;
}
Las claves de B&B paralelo
  • Cota optimista de cada nodo (función del problema): un valor superior al mejor alcanzable en ese subárbol. Si es ≤ cota global actual, ese subárbol no puede mejorar la solución → se poda.
  • Cota global compartida: cuando un proceso encuentra una solución mejor, la difunde para que los demás puedan podar antes.
  • MPI_Iprobe(source, tag, comm, flag, status) → mira si hay un mensaje pendiente sin bloquear. flag queda en 1 si hay, 0 si no. Si hay, se hace Recv normal.
  • Usamos Iprobe para no detener la búsqueda esperando mensajes que tal vez no lleguen. Es la versión "asíncrona" de comprobar el buzón.
  • Anomalía: a veces el paralelo encuentra rápido una cota muy buena y poda enormes subárboles que el secuencial recorre. Resultado: speed-up > p (superlineal). También puede pasar al revés (slowdown).
Esquema combinado. B&B paralelo es trabajadores replicados (cola de subárboles por explorar) + cota global compartida. Cada worker coge subárboles de la cola, explora, comparte mejoras de cota, añade nuevos subárboles a la cola al ramificar.

Cómo se reconoce el esquema en el examen

Lee el enunciado y subraya las palabras de la columna izquierda. Si el enunciado dice "cada celda depende de la anterior", es síncrono o PD, no paralelismo de datos. Si dice "tareas de coste muy desigual", piensa en trabajadores replicados, no en un reparto estático. La regla de oro: identifica primero las dependencias entre los cálculos; el esquema sale solo.
