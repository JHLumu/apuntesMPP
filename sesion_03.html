OpenMP

El estándar dominante para memoria compartida: directivas que añades al código secuencial y un compilador que las traduce a hilos. Aparente sencillez con trampas a las que merece la pena anticiparse.

El modelo fork–join

Un programa OpenMP empieza con un único hilo, el maestro. Cuando encuentra una región paralela, crea un equipo de hilos esclavos. Todos ejecutan el bloque a la vez; al final del bloque, los esclavos se destruyen (o se duermen) y vuelve a haber un único hilo. Ese ciclo se llama fork–join.

Las regiones paralelas se anidan, pero entre ellas el programa vuelve a ser secuencial. Si no usas regiones, OpenMP no hace nada.

Mantén esta imagen en la cabeza al leer código OpenMP. Casi todos los errores comunes (variables compartidas que no deberían serlo, sincronizaciones de más) se entienden mejor situándolos en este diagrama.

Forma general de una directiva

Todas las directivas siguen el mismo patrón:

#pragma omp nombre-directiva [cláusulas]
   bloque estructurado

El bloque estructurado es una sentencia (típicamente un { ... }) con una sola entrada arriba y una sola salida abajo: nada de goto saltando fuera ni return abandonando la región. Esa restricción permite al compilador insertar la lógica de fork/join con seguridad.

Las cuatro directivas que se usan el 90% del tiempo

1 · parallel — crear el equipo

Lanza un grupo de hilos que ejecutan el bloque cada uno por entero. Por defecto, el número de hilos lo decide la variable de entorno OMP_NUM_THREADS o la cláusula num_threads.

#pragma omp parallel num_threads(4)
{
    int tid = omp_get_thread_num();   // 0..3
    printf("Hola desde hilo %d\n", tid);
}   // barrera implícita al cerrar la llave

Cláusulas más usadas: if (decide en tiempo de ejecución si paralelizar), private, shared, default, firstprivate, reduction, num_threads.

2 · for — repartir un bucle

Dentro de una región paralela, reparte las iteraciones del bucle entre los hilos del equipo. No crea hilos por sí solo; presupone que ya existen.

#pragma omp parallel
{
    #pragma omp for
    for (int i = 0; i < n; i++)
        a[i] = b[i] * 2;
}

Como esta combinación es muy frecuente existe la forma abreviada #pragma omp parallel for.

Restricciones del bucle: la inicialización es una asignación, el incremento una suma/resta, la condición una comparación con variable entera sin signo. Variables y límites deben ser enteros. Sin barrera al final si añades nowait.

3 · sections — repartir bloques distintos

Cuando lo que quieres paralelizar no es un bucle sino tareas diferentes que pueden hacerse a la vez.

#pragma omp parallel sections
{
    #pragma omp section
    procesar_imagenes();

    #pragma omp section
    indexar_textos();

    #pragma omp section
    sincronizar_db();
}

4 · single — solo uno lo hace

Dentro de una región paralela, marca un bloque que debe ejecutar un único hilo (no necesariamente el maestro). Útil para inicializaciones, E/S, o partes que no se pueden paralelizar.

#pragma omp parallel
{
    #pragma omp single
    inicializar();   // solo un hilo entra aquí

    #pragma omp for     // todos vuelven a trabajar
    for (i = 0; i < n; i++) ...
}

Parientes: master (similar pero forzando el hilo maestro y sin barrera), ordered (el bloque se ejecuta en el orden secuencial del bucle).

Variables: ¿quién ve qué?

Es la fuente principal de errores en OpenMP. Hay tres «estados» principales para una variable dentro de una región paralela:

shared

Una sola copia, visible para todos los hilos. Por defecto, las variables declaradas antes de la región son compartidas. Si dos hilos escriben, condición de carrera.

private

Cada hilo tiene su propia copia, no inicializada. Las variables declaradas dentro de la región paralela son privadas automáticamente. La variable original conserva su valor al salir, sin verse afectada.

Variantes útiles

  • firstprivate(x): cada hilo tiene su copia, inicializada con el valor que tenía x al entrar.
  • lastprivate(x): en bucles, la copia que sale es la de la última iteración (la que habría salido en el orden secuencial).
  • default(shared|none): si pones none, el compilador exige declarar el sharing de cada variable. Es una costumbre defensiva muy recomendable en programas grandes.
  • threadprivate(lista): declara variables globales como privadas a cada hilo, persistentes entre regiones paralelas.
CLÁUSULA ESTRELLA

reduction(op : lista)

Cada hilo tiene su copia privada de la variable inicializada al elemento neutro de op. Al final de la región, todas las copias se combinan con op y el resultado va a la variable original. Sin reduction, calcular una suma o un máximo en paralelo exigiría una sección crítica que serializaría el bucle.

double suma = 0.0;
#pragma omp parallel for reduction(+: suma)
for (int i = 0; i < n; i++)
    suma += a[i];

Operadores soportados: +, -, *, &, |, ^, &&, ||, min, max.

Sincronización

Cuatro construcciones para coordinar hilos. Aprende a distinguirlas:

critical — sección crítica

El bloque se ejecuta en exclusión mutua: solo un hilo a la vez. Se puede dar un nombre para distinguir varias secciones críticas distintas que no deben bloquearse entre sí.

#pragma omp critical(actualizar_total)
{
    total += valor_local;
    contador++;
}

atomic — actualización atómica

Versión muy ligera de critical para operaciones simples: x++, x += a, etc. El compilador suele traducirlo a una instrucción atómica del hardware. Mucho más rápido que critical cuando aplica.

#pragma omp atomic
contador++;

barrier — barrera explícita

Ningún hilo pasa hasta que todos hayan llegado. Las directivas for, sections y single ya tienen barrera implícita al final, así que solo necesitas barrier manualmente en casos especiales.

La regla del nowait

Cuando varias directivas for/sections/single van seguidas dentro de la misma región paralela, las barreras implícitas cuestan. Si la siguiente directiva no depende de los resultados de la anterior, ponle nowait a la primera.

#pragma omp parallel
{
    #pragma omp for nowait
    for (i = 0; i < n; i++) a[i] = f(i);

    #pragma omp for          // esta sí espera
    for (i = 0; i < m; i++) b[i] = g(i);
}

Distribución de iteraciones: schedule

Por defecto, for reparte las iteraciones en bloques iguales (static). Pero si las iteraciones tienen costes desiguales —típico en problemas no regulares— eso desequilibra el trabajo. La cláusula schedule da control:

PolíticaCómo reparteCuándo conviene
static[,chunk] Trozos fijos asignados a priori, round-robin Iteraciones de coste uniforme (la mayoría de los casos)
dynamic[,chunk] Cada hilo coge un trozo cuando acaba el suyo Iteraciones de coste muy variable; añade overhead
guided[,chunk] Como dynamic, pero los trozos van disminuyendo Equilibrio entre overhead y balanceo
auto Decide el compilador / runtime Cuando no quieres pensarlo
runtime Se toma de la variable OMP_SCHEDULE Experimentación: cambias la política sin recompilar

Tareas: task y taskwait

A partir de OpenMP 3.0, además del paralelismo «de bucle» existe el paralelismo de tareas. Una tarea es una unidad de trabajo que el runtime encola; cualquier hilo del equipo puede tomarla cuando esté libre.

void recorrer(node *n) {
    if (n == NULL) return;
    #pragma omp task
    recorrer(n->izq);
    #pragma omp task
    recorrer(n->der);
    procesar(n);
    #pragma omp taskwait   // espera a las dos subtareas
}

// Llamada inicial
#pragma omp parallel
#pragma omp single
recorrer(raiz);

Esto es lo que abre la puerta a paralelizar recursión (divide y vencerás) y árboles irregulares.

Ejemplo completo: suma de vectores

#include <stdio.h>
#include <omp.h>

int main() {
    const int N = 1000000;
    double a[N], b[N], c[N];
    double suma = 0.0;

    // inicialización en paralelo
    #pragma omp parallel for
    for (int i = 0; i < N; i++) {
        a[i] = i * 1.0;
        b[i] = (N - i) * 1.0;
    }

    double t0 = omp_get_wtime();

    // cómputo principal: c = a + b y suma global
    #pragma omp parallel for reduction(+: suma)
    for (int i = 0; i < N; i++) {
        c[i] = a[i] + b[i];
        suma += c[i];
    }

    double t1 = omp_get_wtime();
    printf("suma = %.2f, tiempo = %.4f s\n", suma, t1 - t0);
    return 0;
}

Para compilar: gcc -fopenmp -O2 prog.c -o prog. Para ejecutar con 8 hilos: OMP_NUM_THREADS=8 ./prog.

Práctica de recuperación

Algunas preguntas requieren pensar antes de revelar. Si fallas, vuelve a la sección correspondiente: el error productivo refuerza el aprendizaje más que el repaso pasivo.

¿Qué diferencia hay entre #pragma omp parallel y #pragma omp parallel for?
parallel solo crea el equipo de hilos: todos ejecutan el bloque entero. parallel for es la combinación abreviada que además reparte las iteraciones del bucle siguiente entre esos hilos.
¿Cuándo usas private y cuándo firstprivate?
private: cada hilo necesita su copia pero no le importa con qué valor empieza (típicamente la inicializa el propio bucle). firstprivate: cada hilo necesita su copia inicializada con el valor que tenía la variable al entrar a la región.
¿Cuál es la trampa de usar critical dentro de un bucle paralelo?
Que serializa los hilos: solo uno puede estar dentro de la sección crítica a la vez. Si la sección crítica es el grueso del trabajo, el paralelismo desaparece. Usa reduction o atomic si la operación es simple.
¿Para qué sirve nowait?
Elimina la barrera implícita al final de un for, sections o single. Sirve cuando la directiva siguiente no depende de los resultados de la anterior: ahorras el coste de la sincronización.
Tienes un bucle donde unas iteraciones tardan diez veces más que otras. ¿Qué schedule es razonable?
dynamic o guided. static repartiría los chunks a priori y los hilos con iteraciones largas terminarían tarde mientras otros esperan. guided suele ser un buen punto medio: empieza con chunks grandes y los reduce.
¿Qué hace reduction(max: m) en un bucle?
Cada hilo tiene su copia local de m inicializada al valor más pequeño posible. Calcula su máximo local. Al final, OpenMP combina todas las copias quedándose con el máximo global y lo escribe en m. Todo sin sincronización explícita.
¿Por qué default(none) es una buena práctica?
Obliga al compilador a fallar si se te olvida declarar el sharing de cualquier variable. En programas grandes evita errores difíciles de detectar: una variable que pensabas privada y resulta compartida, o viceversa.
Repaso de la sesión 2: ¿qué problema fundamental de la memoria compartida resuelve critical?
Las condiciones de carrera: si dos hilos escriben la misma variable sin coordinarse, el resultado es impredecible. critical serializa el acceso garantizando exclusión mutua.
¿Qué imprime este código con cuatro hilos?
#pragma omp parallel
printf("hola\n");
Cuatro veces «hola», sin orden garantizado. Cada hilo ejecuta el printf entero. Si quieres que solo uno escriba, encierra el printf en #pragma omp single.
