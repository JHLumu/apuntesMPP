OpenMP

El estándar dominante para memoria compartida: directivas que añades al código secuencial y un compilador que las traduce a hilos. Aparente sencillez con trampas a las que merece la pena anticiparse.

Antes de OpenMP: el problema que viene a resolver

Para entender por qué OpenMP se ve como se ve, hay que situarlo entre dos extremos del paralelismo. La sesión 2 dejó este eje planteado; aquí lo cerramos.

Paralelismo explícito

El programador especifica cómo cooperan los procesos: crea hilos, sincroniza, gestiona variables compartidas. El compilador tiene poco que hacer. Es lo que ofrece pthread en C o el modelo de threads de Java. Máximo control, máximo trabajo.

Paralelismo implícito

El programador escribe código secuencial y el compilador paralelizante inserta las instrucciones de paralelización. Tiene que analizar dependencias de datos —y cuando duda, no paraleliza. Mínimo esfuerzo, eficiencia limitada por la conservadora política del compilador.

OpenMP se sitúa entre los dos. El programador anota el código secuencial con directivas que le indican al compilador qué se puede paralelizar y cómo. El compilador, ayudado por esas pistas, genera el código paralelo. Es el compromiso pragmático: el programador aporta el conocimiento que el compilador no puede inferir solo, sin tener que escribir pthread_create a mano.

Un compilador paralelizante puro no paraleliza a[i] = b[i] + a[i-1] porque detecta una dependencia entre iteraciones. Si el programador sabe que en su caso i-1 nunca apunta a una posición conflictiva, puede forzarlo. Esa es la idea: el programador firma con su pragma lo que el compilador no podía garantizar solo.

El modelo fork–join

Un programa OpenMP empieza con un único hilo, el maestro. Cuando encuentra una región paralela, crea un equipo de hilos esclavos. Todos ejecutan el bloque a la vez; al final del bloque hay una barrera implícita: los esclavos se destruyen (o se duermen) y vuelve a haber un único hilo. Ese ciclo —duplicar y reunir— se llama fork–join.

Las regiones paralelas se pueden anidar, pero entre ellas el programa vuelve a ser secuencial. Si no usas regiones, OpenMP no hace nada por ti.

Implicaciones que no son evidentes a primera vista:

  • Crear y destruir hilos cuesta. Una región paralela muy pequeña puede ser más lenta que la secuencial. En la práctica, los runtimes mantienen un pool de hilos durmiendo y los despiertan, pero aun así el overhead existe.
  • Cuanto más tiempo pase el programa dentro de regiones paralelas, mejor. Lo dice la ley de Amdahl: el speed-up está limitado por la parte serie. Granularidad gruesa > granularidad fina.
  • El maestro también trabaja. Si lanzas 4 hilos, hay 4 hilos, no 4 esclavos + 1 maestro. El maestro participa en el reparto.
Mantén esta imagen en la cabeza al leer código OpenMP. Casi todos los errores comunes (variables compartidas que no deberían serlo, sincronizaciones de más, regiones paralelas que dan peor rendimiento que el código secuencial) se entienden mejor situándolos en este diagrama.

Forma general de una directiva

Todas las directivas siguen el mismo patrón:

#pragma omp nombre-directiva [cláusulas]
   bloque estructurado

El bloque estructurado es una sentencia (típicamente un { ... }) con una sola entrada arriba y una sola salida abajo: nada de goto saltando fuera ni return abandonando la región. Esa restricción permite al compilador insertar la lógica de fork/join con seguridad: sabe dónde empieza el reparto y dónde está la barrera de cierre.

Compilación con gcc o icc:

$ gcc -O3 programa.c -fopenmp -o programa
$ icc -O3 programa.c -qopenmp -o programa     # -openmp en icc antiguas
$ OMP_NUM_THREADS=8 ./programa

Si compilas sin -fopenmp, los #pragma omp se ignoran silenciosamente y el código se ejecuta secuencial. Esa es una de las gracias del modelo: el mismo código fuente vale para serie y paralelo. Las funciones de librería como omp_get_thread_num() sí dejan de existir, por eso suelen aparecer dentro de #ifdef _OPENMP:

#include <omp.h>
int main() {
    int iam = 0, np = 1;
    #pragma omp parallel private(iam, np)
    {
    #if defined(_OPENMP)
        np  = omp_get_num_threads();
        iam = omp_get_thread_num();
    #endif
        printf("Hello from thread %d out of %d\n", iam, np);
    }
}

Con 4 hilos imprime cuatro líneas «Hello…», en orden no garantizado. Esa indeterminación es la primera lección práctica: la salida no es un buen indicador de corrección, porque dos ejecuciones pueden barajar las líneas distinto siendo ambas correctas.

Las directivas que se usan el 90 % del tiempo

1 · parallel — crear el equipo

Lanza un grupo de hilos que ejecutan el bloque cada uno por entero. Por defecto, el número de hilos lo decide la variable de entorno OMP_NUM_THREADS o la cláusula num_threads; si no se fija, el runtime suele crear tantos hilos como cores tenga la máquina.

#pragma omp parallel num_threads(4)
{
    int tid = omp_get_thread_num();   // 0..3
    printf("Hola desde hilo %d\n", tid);
}   // barrera implícita al cerrar la llave

Cláusulas frecuentes: if(expr) (paraleliza solo si expr ≠ 0; muy útil para evitar el overhead cuando n es pequeño), num_threads(k), y todas las de alcance de datos (private, shared, default, firstprivate, reduction, copyin).

Cuando una región paralela aparece dentro de otra (anidamiento), cada hilo del nivel exterior se convierte en maestro de un nuevo equipo, si el anidamiento está habilitado (omp_set_nested(1) o OMP_NESTED=true).

2 · for — repartir un bucle

Dentro de una región paralela, reparte las iteraciones del bucle entre los hilos del equipo. No crea hilos por sí solo; presupone que ya existen.

#pragma omp parallel
{
    #pragma omp for
    for (int i = 0; i < n; i++)
        a[i] = b[i] * 2;
}

Como esta combinación es muy frecuente existe la forma abreviada #pragma omp parallel for, que admite todas las cláusulas de ambos salvo nowait.

El compilador exige que el bucle sea «canónico» para poder calcular el número de iteraciones antes de empezar:

  • Inicialización: una asignación simple (i = 0).
  • Condición: una comparación de la variable de iteración (entera, sin signo en el formulario clásico de las diapositivas) con un valor invariable, usando <, <=, > o >=.
  • Incremento: una suma o resta de una cantidad constante (i++, i += 2).
  • Tres partes con valores enteros, y dentro del bucle la variable de iteración no puede modificarse.

Esto descarta directamente bucles while, bucles con número variable de iteraciones y recorridos de listas enlazadas. Para esos, OpenMP propone tareas (más abajo).

3 · sections — repartir bloques distintos

Cuando lo que quieres paralelizar no es un bucle sino tareas diferentes que pueden hacerse a la vez. Cada section la ejecuta un único hilo del equipo.

#pragma omp parallel sections
{
    #pragma omp section
    procesar_imagenes();

    #pragma omp section
    indexar_textos();

    #pragma omp section
    sincronizar_db();
}   // barrera implícita salvo nowait

Si hay menos secciones que hilos, algunos hilos se quedan sin trabajo (esperan en la barrera). Si hay más, los hilos van cogiendo secciones a medida que acaban las anteriores.

4 · single, master y ordered — ejecución selectiva

Dentro de una región paralela, hay momentos en que solo un hilo debe ejecutar algo: imprimir un resultado, leer de fichero, asignar una variable global. Hay tres mecanismos parecidos pero distintos:

DirectivaQuién ejecutaBarrera al finalUso típico
single El primer hilo que llega (no necesariamente el maestro) Sí (salvo nowait) Inicializaciones, E/S, asignaciones únicas
master Obligatoriamente el hilo maestro No Reservar trabajo al hilo 0, sin sincronización
ordered Hilos por orden de iteración secuencial Implícita por iteración Imprimir o registrar en orden dentro de un for 
#pragma omp parallel
{
    #pragma omp single
    inicializar();   // solo un hilo entra aquí; los demás esperan

    #pragma omp for     // todos vuelven a trabajar
    for (i = 0; i < n; i++) ...
}

Variables: ¿quién ve qué?

Es la fuente principal de errores en OpenMP. Una variable, al entrar a una región paralela, puede tener varios alcances:

El alcance decide qué pasa con la variable al entrar y al salir de la región. Pocas reglas, pero a memorizar bien.

Las cláusulas, una por una

  • shared(lista): una sola copia visible para todos. Si varios hilos escriben, hay condición de carrera: necesita protección (critical, atomic, reduction).
  • private(lista): cada hilo tiene su copia, sin inicializar. Al salir, la variable original conserva su valor previo a la región (no se ve afectada por lo que hicieran los hilos).
  • firstprivate(lista): como private, pero cada copia se inicializa con el valor que tuviera la variable al entrar. Útil cuando los hilos necesitan partir de un valor común.
  • lastprivate(lista): como private, pero al salir la variable original recibe el valor que tendría en la última iteración secuencial (o la última sección). Útil cuando el bucle calcula algo cuyo último valor te interesa.
  • default(shared|none): si pones none, el compilador obliga a declarar el alcance de cada variable. Práctica defensiva muy recomendable.
  • threadprivate(lista): declara variables globales como privadas a cada hilo, persistentes entre regiones paralelas. Cada hilo tiene su copia que sobrevive al join.
  • copyin(lista): usado con threadprivate, copia el valor del maestro a las réplicas locales al entrar.
CLÁUSULA ESTRELLA

reduction(op : lista)

Cada hilo tiene su copia privada de la variable, inicializada al elemento neutro de op (0 para +, 1 para *, -INF para max…). Cada hilo opera sobre su copia local sin contención. Al final de la región, todas las copias se combinan con op y el resultado se acumula a la variable original. Sin reduction, calcular una suma o un máximo en paralelo exigiría una sección crítica que serializaría el bucle.

double suma = 0.0;
#pragma omp parallel for reduction(+: suma)
for (int i = 0; i < n; i++)
    suma += a[i];

Operadores soportados: +, -, *, &, |, ^, &&, ||, min, max (los dos últimos desde OpenMP 3.1).

Ejemplo canónico: cálculo de π por integración

Aproximamos π integrando 4/(1 + x²) entre 0 y 1, con la regla del rectángulo de N subintervalos.

double local, pi = 0.0, w = 1.0 / N;
long i;

#pragma omp parallel private(i, local)
{
    #pragma omp single
    pi = 0.0;                          // uno solo lo resetea

    #pragma omp for reduction(+: pi)
    for (i = 0; i < N; i++) {
        local = (i + 0.5) * w;          // centro del rectángulo
        pi += 4.0 / (1.0 + local * local);
    }
}
pi *= w;

Por qué funciona: cada hilo tiene su local (privada) y su copia local de pi (gracias a reduction). El bucle no tiene dependencias entre iteraciones. Sin la cláusula reduction, tendríamos que poner pi += … en una sección crítica y el bucle se serializaría.

Sincronización

Cinco construcciones para coordinar hilos. Aprende a distinguirlas — los exámenes preguntan cuál usar en cada caso.

critical — sección crítica

El bloque se ejecuta en exclusión mutua: solo un hilo a la vez. Se puede dar un nombre para distinguir varias secciones críticas distintas que no deben bloquearse entre sí (dos críticas con nombres distintos pueden ejecutarse a la vez).

#pragma omp critical(actualizar_total)
{
    total += valor_local;
    contador++;
}

Sin nombre, todas las críticas comparten la misma llave global: si tienes dos críticas distintas sin nombre en distintas partes del código, se serializan entre sí innecesariamente.

atomic — actualización atómica

Versión muy ligera de critical para operaciones simples: x++, ++x, x--, --x, x op= expr. El compilador suele traducirlo a una instrucción atómica del hardware (típicamente un compare-and-swap o un fetch-and-add). Mucho más rápido que critical cuando aplica.

#pragma omp atomic
contador++;

Limitación importante: solo cubre la expresión que sigue, no un bloque arbitrario. Si necesitas «leer, decidir, escribir», sigue siendo critical.

barrier — barrera explícita

Ningún hilo pasa hasta que todos hayan llegado. Las directivas for, sections y single ya tienen barrera implícita al final, así que solo necesitas barrier manualmente en casos especiales —por ejemplo, en medio de un bloque parallel sin for.

La regla del nowait

Cuando varias directivas for/sections/single van seguidas dentro de la misma región paralela, las barreras implícitas cuestan. Si la siguiente directiva no depende de los resultados de la anterior, ponle nowait a la primera.

#pragma omp parallel
{
    #pragma omp for nowait
    for (i = 0; i < n; i++) a[i] = f(i);

    #pragma omp for          // esta sí espera
    for (i = 0; i < m; i++) b[i] = g(i);
}   // barrera implícita del parallel

El primer bucle puede acabar a destiempo: el hilo que termine antes puede entrar al segundo bucle sin esperar. Solo es seguro si a y b son independientes.

Llaves de librería: omp_*_lock

Cuando necesitas exclusión mutua programática (por ejemplo, una llave por elemento de un array, o una llave que un hilo intenta tomar y, si no puede, hace otra cosa), las directivas no llegan. Existen funciones de librería:

omp_lock_t lck;
omp_init_lock(&lck);

#pragma omp parallel shared(lck) private(id)
{
    id = omp_get_thread_num();

    omp_set_lock(&lck);          // bloquea hasta conseguir la llave
    printf("hilo %d en sección\n", id);
    omp_unset_lock(&lck);

    while (!omp_test_lock(&lck))  // no bloqueante: si no la consigue,
        skip(id);                // hace otra cosa entretanto
    work(id);
    omp_unset_lock(&lck);
}
omp_destroy_lock(&lck);

El detalle importante es omp_test_lock: intenta tomar la llave; si la consigue devuelve verdadero, si no, falso (sin bloquear). Eso permite patrones de «si está libre actualizo, si no sigo y reintento más tarde».

Distribución de iteraciones: schedule

Por defecto, for reparte las iteraciones en bloques iguales (static). Pero si las iteraciones tienen costes desiguales —típico en problemas irregulares (matrices triangulares, búsquedas, refinamientos)— eso desequilibra el trabajo. La cláusula schedule da control.

Reparto de 16 iteraciones entre 4 hilos. dynamic y guided deciden en runtime: cada hilo coge un trozo al acabar el anterior.

PolíticaCómo reparteCuándo conviene
static[,chunk] Trozos fijos asignados a priori, round-robin. Sin chunk, un bloque grande por hilo. Iteraciones de coste uniforme (la mayoría de los casos). Overhead mínimo.
dynamic[,chunk] Cada hilo coge un trozo cuando acaba el suyo. Sin chunk, equivale a chunk = 1. Iteraciones de coste muy variable; coste de gestión no despreciable.
guided[,chunk] Como dynamic, pero los trozos van disminuyendo (empieza con bloques grandes y termina con pequeños). Equilibrio entre overhead y balanceo. Buena opción por defecto si las iteraciones son irregulares.
auto Decide el compilador / runtime. Cuando no quieres pensarlo (raramente óptimo).
runtime Se toma de la variable de entorno OMP_SCHEDULE. Experimentación: cambias la política sin recompilar.

Tareas: task y taskwait

A partir de OpenMP 3.0 (2008), además del paralelismo «de bucle» existe el paralelismo de tareas. El modelo cambia de thread-centric (creo hilos y reparto bucles) a task-centric (declaro unidades de trabajo y el runtime las planifica).

Una tarea es una unidad de trabajo que el runtime encola; cualquier hilo del equipo puede tomarla cuando esté libre. Las tareas existen porque hay paralelismo que las directivas anteriores no expresan bien:

  • Recorridos de listas cuya longitud no se conoce al entrar (no se puede #pragma omp for sobre un while).
  • Recursión: divide y vencerás, árboles, backtracking. El paralelismo es irregular y sólo se descubre dentro de la propia ejecución.
  • Anidamiento sin barreras innecesarias: al usar parallel dentro de parallel, las barreras implícitas pueden ser muy costosas.

Modelo task-centric: la directiva task crea una unidad de trabajo y la entrega al runtime; los hilos del equipo se la reparten dinámicamente.

Sintaxis

#pragma omp task [clausulas]
  bloque

#pragma omp taskwait      // espera a que terminen las hijas

Cláusulas: if(expr) (si falsa, la tarea se ejecuta inmediatamente, sin encolar), untied, private, firstprivate, shared, default(shared|none).

Un conjunto de tareas se ve forzado a completarse en: (a) una barrera implícita de hilos, (b) una barrera explícita (#pragma omp barrier) o (c) un #pragma omp taskwait.

Tied vs untied

Por defecto una tarea está tied (ligada) al hilo que la empieza a ejecutar: ese hilo la termina, aunque pueda hacer pausas en los task scheduling points (los puntos donde el runtime puede cambiar de tarea: #pragma omp task, taskwait, barreras, fin de tarea). Si declaras untied, cualquier hilo puede continuar la tarea desde donde otro la dejó. Untied es más flexible pero peligroso: no uses threadprivate ni omp_get_thread_num() dentro de tareas untied, porque cambian de hilo a mitad.

Ejemplo: recorrer una lista

Versión secuencial:

while (p) {
    do_independent_work(p);
    p = p->next;
}

No se puede hacer con #pragma omp for (la condición no es entera). Con tareas:

p = listhead;
#pragma omp parallel
{
    #pragma omp single nowait          // un hilo genera tareas...
    {
        while (p) {
            #pragma omp task firstprivate(p)  // ...que cualquiera ejecuta
            do_independent_work(p);
            p = p->next;
        }
    }   // fin del single (sin barrera)
}   // barrera del parallel: espera a todas las tareas

El firstprivate(p) es crítico: cada tarea recibe una copia del valor actual de p. Sin él, todas las tareas compartirían el mismo puntero y, cuando se ejecutaran, estarían apuntando al final de la lista.

Ejemplo: Fibonacci recursivo

long fib(int n) {
    long a, b;
    if (n < 2) return n;

    #pragma omp task shared(a)
    a = fib(n - 1);
    #pragma omp task shared(b)
    b = fib(n - 2);

    #pragma omp taskwait          // hasta que terminen ambas
    return a + b;
}

// Llamada inicial
#pragma omp parallel
#pragma omp single
result = fib(30);

Cuidado práctico: si n es pequeño, crear tareas cuesta más que ejecutar el código serie. La heurística estándar es un umbral de corte: a partir de cierto n, llamar a una versión secuencial. Lo veremos en divide y vencerás.

Las tareas se generan dentro de una región paralela, pero normalmente desde un solo punto del código (un single que recorre y genera). Si todos los hilos generaran las mismas tareas, las duplicarías.

Cuidados prácticos

Aleatoriedad: rand() no es thread-safe

Si paralelizas un Monte Carlo o una metaheurística, vas a generar números aleatorios. rand() mantiene estado global: dos hilos llamando a rand a la vez pueden serializarse (el sistema garantiza secuencialidad) o, peor, devolver la misma secuencia. Solución: usar rand_r(&seed), donde la semilla es local al hilo.

#pragma omp parallel private(seed)
{
    seed = 12345 + omp_get_thread_num();   // semilla por hilo
    #pragma omp for
    for (i = 0; i < N; i++) {
        int r = rand_r(&seed);             // thread-safe
        ...
    }
}

False sharing

Si dos hilos escriben en variables distintas pero contiguas en memoria (que comparten línea de caché), el hardware fuerza a invalidar y recargar la línea constantemente. El programa es correcto, pero brutalmente lento. Síntoma: paralelizas y va peor que en serie. Mitigación: separar los datos en memoria o usar arrays con padding.

Variables de bucle

En #pragma omp parallel for, la variable de iteración del bucle es automáticamente privada. Pero si tienes bucles anidados, las variables interiores no lo son:

#pragma omp parallel for            // i es privada ✓
for (i = 0; i < N; i++)
    for (j = 0; j < M; j++)        // j es SHARED por defecto: ¡bug!
        a[i][j] = ...;

Solución: private(j) en el pragma, o declarar j dentro del bucle.

Ejemplo completo y comentado: suma y producto escalar

#include <stdio.h>
#include <omp.h>

int main() {
    const int N = 1000000;
    double *a = malloc(N * sizeof(double));
    double *b = malloc(N * sizeof(double));
    double *c = malloc(N * sizeof(double));
    double dot = 0.0;

    // 1. inicialización en paralelo (cada hilo escribe en sus propias posiciones)
    #pragma omp parallel for
    for (int i = 0; i < N; i++) {
        a[i] = i * 1.0;
        b[i] = (N - i) * 1.0;
    }

    double t0 = omp_get_wtime();

    // 2. cómputo principal: c = a + b, y producto escalar a·b
    //    Dos cosas a la vez: suma vectorial (sin contención) y reducción.
    #pragma omp parallel for reduction(+: dot)
    for (int i = 0; i < N; i++) {
        c[i] = a[i] + b[i];        // shared, sin conflicto: cada i va a sitio distinto
        dot += a[i] * b[i];        // reducción: cada hilo acumula local, se combinan al final
    }

    double t1 = omp_get_wtime();
    printf("dot = %.2f, t = %.4f s\n", dot, t1 - t0);

    free(a); free(b); free(c);
    return 0;
}

$ gcc -fopenmp -O3 prog.c -o prog
$ OMP_NUM_THREADS=8 ./prog

Guía resuelta: ejercicios de la Semana 3

Los ejercicios oficiales se basan en cuatro ficheros del paquete ejemplosOpenMP.tar.gz (directorio CCCC/). Aquí los resolvemos razonando, no solo dando la respuesta.

Ejercicio 1 · ejemplo_for.c — comparar schedules

Enunciado: probar distintos schedule (estático/dinámico, sin tamaño/tamaño = 2) y razonar el comportamiento observado.

Qué probar (cuatro variantes):

#pragma omp parallel for schedule(static)
#pragma omp parallel for schedule(static, 2)
#pragma omp parallel for schedule(dynamic)
#pragma omp parallel for schedule(dynamic, 2)

Sugerencia: que el bucle imprima i y omp_get_thread_num(), así ves a qué hilo va cada iteración.

Comportamiento esperado:

  • static (sin tamaño) con P hilos y N iteraciones: cada hilo coge un bloque contiguo de aproximadamente N/P iteraciones. Hilo 0 → 0..N/P−1, etc. Reparto fijo desde el inicio.
  • static, 2: trozos de tamaño 2, en round-robin. Hilo 0 → {0,1}, hilo 1 → {2,3}, hilo 2 → {4,5}, hilo 3 → {6,7}, hilo 0 → {8,9}…
  • dynamic (sin tamaño = dynamic, 1): cada hilo coge una iteración del pool cuando acaba. El orden de impresión es no determinista entre ejecuciones porque depende del scheduler del SO.
  • dynamic, 2: igual pero por parejas. Menos overhead que dynamic, 1, mismo carácter no determinista.

Razonamiento: con iteraciones de coste uniforme, static es el más rápido porque no tiene overhead de gestión. dynamic es útil cuando las iteraciones tienen costes muy distintos: si la iteración 0 cuesta lo mismo que la 100, ningún hilo queda esperando. El precio es un coste de coordinación: cada vez que un hilo pide trabajo, hay sincronización con el pool. Con chunk = 2 reduces ese coste a la mitad respecto a dynamic, 1, a cambio de balance ligeramente peor.

Ejercicio 2 · ejemplo_sections.c — combinar directivas

Enunciado: agrupar las directivas parallel y sections en una única.

Solución: sustituir

#pragma omp parallel
{
    #pragma omp sections
    {
        #pragma omp section
        bloque_A();
        #pragma omp section
        bloque_B();
    }
}

por la forma combinada:

#pragma omp parallel sections
{
    #pragma omp section
    bloque_A();
    #pragma omp section
    bloque_B();
}

Razonamiento: es equivalente funcionalmente. La forma combinada es preferida cuando dentro de la región paralela solo hay las sections; ahorra un nivel de anidamiento. Limitación: la combinada no admite nowait (la barrera del parallel ya estaría allí de todas formas).

Ejercicio 3 · comparar sections, single y critical

Enunciado: razonar el comportamiento de los tres.

ConstrucciónCuántos hilos lo ejecutanEn paralelo entre síBarrera al final
sectionsTantos hilos como secciones (uno por sección)Sí, distintas secciones en paraleloSí (salvo nowait)
singleUno solo (el primero que llega)—Sí (salvo nowait)
criticalTodos, pero uno cada vezNo, serializaNo

Cuándo se usa cada uno:

  • sections: distintos trabajos que se pueden hacer en paralelo. Reparto estático: si tienes 3 secciones y 8 hilos, 5 se quedan parados.
  • single: un único trabajo que debe hacer un hilo cualquiera. Los demás esperan en la barrera al final.
  • critical: el mismo trabajo lo hacen todos, pero con exclusión mutua. Útil para acumular en una variable compartida o cualquier sección que no pueda ejecutarse simultáneamente.

Confusión típica: «si quiero que algo solo lo haga un hilo, ¿single o critical?». Respuesta: single. critical hace que todos los hilos pasen, uno a uno; single hace que uno solo entre y los demás se salten el bloque.

Ejercicio 4 · Fibonacci paralelo sin tareas

Enunciado: indicar cómo paralelizar fib(n) = fib(n-1) + fib(n-2) sin usar #pragma omp task.

Idea: las dos llamadas recursivas son independientes (no hay dependencias entre ellas hasta sumarlas). El primer nivel de la recursión expone dos trabajos paralelos. Eso encaja perfectamente en sections:

long fib(int n) {
    long a, b;
    if (n < 2) return n;

    #pragma omp parallel sections     // dos secciones en paralelo
    {
        #pragma omp section
        a = fib(n - 1);              // llamada secuencial dentro

        #pragma omp section
        b = fib(n - 2);
    }
    return a + b;
}

Razonamiento crítico: esta solución funciona, pero tiene dos problemas serios:

  1. Solo el primer nivel se paraleliza. Las llamadas internas fib(n-1) y fib(n-2) son secuenciales. Si quisieras paralelismo más profundo, necesitarías habilitar paralelismo anidado (omp_set_nested(1)) y aceptar que cada nivel crea un nuevo equipo de hilos —explosión combinatoria de hilos.
  2. Desbalance: fib(n-1) es mucho más costoso que fib(n-2) (de hecho la diferencia es exponencial). Una sección tarda muchísimo más que la otra; un hilo se queda esperando.

Por eso las tareas son la respuesta natural: cada nodo del árbol de recursión se convierte en una tarea, el runtime las planifica sobre un equipo fijo de hilos, sin anidamiento explosivo, con balanceo dinámico. Esta es la motivación canónica del modelo de tareas.

Errores frecuentes (y cómo detectarlos)

SíntomaCausa probableDiagnóstico
Resultado cambia entre ejecuciones Condición de carrera: variable shared que debería ser private o reduction Ejecuta con 1 hilo (OMP_NUM_THREADS=1): si funciona, es paralelismo. Revisa la cláusula de cada variable.
Versión paralela más lenta que la serie Granularidad demasiado fina o false sharing Mide el tamaño de la región; agrupa más trabajo por hilo. Si la región es grande, busca false sharing.
El programa se cuelga Llave no liberada o barrera dentro de if Por cada omp_set_lock debe haber un unset_lock en todas las salidas del bloque. Las barreras deben encontrarlas todos los hilos.

Práctica de recuperación

Algunas preguntas requieren pensar antes de revelar. Si fallas, vuelve a la sección correspondiente: el error productivo refuerza el aprendizaje más que el repaso pasivo.

¿Qué diferencia hay entre #pragma omp parallel y #pragma omp parallel for?
parallel solo crea el equipo de hilos: todos ejecutan el bloque entero. parallel for es la combinación abreviada que además reparte las iteraciones del bucle siguiente entre esos hilos. Sin el for, todos los hilos ejecutarían el bucle completo independientemente.
¿Cuándo usas private y cuándo firstprivate?
private: cada hilo necesita su copia pero no le importa con qué valor empieza (típicamente la inicializa el propio bucle). firstprivate: cada hilo necesita su copia inicializada con el valor que tenía la variable al entrar a la región.
¿Cuál es la trampa de usar critical dentro de un bucle paralelo?
Que serializa los hilos: solo uno puede estar dentro de la sección crítica a la vez. Si la sección crítica es el grueso del trabajo, el paralelismo desaparece. Usa reduction si es una agregación, atomic si es una operación simple, o reparte el trabajo de modo que no haga falta sincronizar.
¿Para qué sirve nowait?
Elimina la barrera implícita al final de un for, sections o single. Sirve cuando la directiva siguiente no depende de los resultados de la anterior: ahorras el coste de la sincronización. Sin nowait, todos los hilos esperan al más lento.
Tienes un bucle donde unas iteraciones tardan diez veces más que otras. ¿Qué schedule es razonable?
dynamic o guided. static repartiría los chunks a priori y los hilos con iteraciones largas terminarían tarde mientras otros esperan. guided suele ser un buen punto medio: empieza con chunks grandes (amortiza el overhead) y los reduce (mejora el balanceo final).
¿Qué hace reduction(max: m) en un bucle?
Cada hilo tiene su copia local de m inicializada al valor más pequeño posible. Calcula su máximo local. Al final, OpenMP combina todas las copias quedándose con el máximo global y lo escribe en m. Todo sin sincronización explícita por iteración.
¿Por qué default(none) es una buena práctica?
Obliga al compilador a fallar si se te olvida declarar el alcance de cualquier variable. En programas grandes evita errores difíciles de detectar: una variable que pensabas privada y resulta compartida, o viceversa.
¿Para qué sirve #pragma omp single nowait dentro de un parallel que genera tareas?
Para que un solo hilo recorra la estructura generando tareas (las demás generaciones serían duplicados), pero los otros hilos no esperen en la barrera del single y empiecen a coger tareas del pool inmediatamente.
Diferencia entre tarea tied y untied.
Tied (por defecto): el hilo que empieza la tarea es el que la termina, aunque pueda hacer pausas para atender otras tareas. Untied: cualquier hilo del equipo puede retomar la tarea desde donde otro la dejó. Untied da más flexibilidad al scheduler pero rompe con threadprivate y omp_get_thread_num().
¿Por qué Fibonacci con sections escala mal y con task escala bien?
Con sections solo paralelizas el nivel superior; las llamadas internas son secuenciales y, además, las dos ramas tienen costes muy distintos (desbalance). Con task, todas las llamadas recursivas se convierten en tareas que el runtime planifica dinámicamente sobre un equipo fijo de hilos: aprovechas el paralelismo de todo el árbol y compensas el desbalance.
Repaso de la sesión 2: ¿qué problema fundamental de la memoria compartida resuelve critical?
Las condiciones de carrera: si dos hilos escriben la misma variable sin coordinarse, el resultado es impredecible. critical serializa el acceso garantizando exclusión mutua.
¿Qué imprime este código con cuatro hilos?
#pragma omp parallel
printf("hola\n");
Cuatro veces «hola», sin orden garantizado. Cada hilo ejecuta el printf entero. Si quieres que solo uno escriba, encierra el printf en #pragma omp single.
En un bucle #pragma omp parallel for con un bucle for (j=…) anidado, ¿hace falta declarar algo sobre j?
Sí: declarar private(j), o mejor declarar j dentro del bucle exterior (con for (int j…)). La variable de iteración del bucle anotado con el pragma es privada automáticamente, pero las de los bucles internos no.
