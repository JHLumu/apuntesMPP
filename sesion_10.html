El grafo de dependencias

Hasta ahora las tareas eran o independientes (relajado) o se sincronizaban globalmente (síncrono). Pero muchas veces hay un patrón intermedio: cada tarea depende de unas pocas, y al terminar habilita a otras. Esto se captura con un grafo dirigido acíclico (DAG).

DEFINICIÓN

Grafo de dependencias

Grafo dirigido acíclico donde los nodos representan tareas y una arista de A a B significa que B no puede empezar hasta que A termine. Opcionalmente, los nodos se etiquetan con el coste de la tarea y las aristas con el coste de la comunicación.

El grafo da información esencial: el camino crítico (la cadena más larga) marca el tiempo mínimo posible. La anchura máxima del grafo indica cuánto paralelismo se puede aprovechar simultáneamente.

Conceptos clave del grafo

  • Camino crítico: secuencia más larga de tareas con dependencias. Aunque tengas infinitos procesadores, no puedes ir más rápido que el camino crítico.
  • Grado de paralelismo: número máximo de tareas que pueden ejecutarse simultáneamente (anchura máxima del grafo en algún nivel).
  • Granularidad: tamaño promedio de los nodos. Granularidad grande → más cómputo, menos overhead de gestión.

Paralelismo en árbol

Caso particular del grafo: cuando la estructura es un árbol. Aparece en reducciones, ordenaciones, búsquedas y muchos algoritmos divide-y-vencerás.

Ejemplo canónico: reducción en árbol

Sumar n números con p procesos. En cada paso, la mitad de los procesos envían su parcial a la otra mitad, que acumula. Repetir log p veces.

Reducción binaria. En cada nivel se duplica la distancia entre los procesos que se comunican, lo que requiere log p fases pero solo p−1 sumas en total.

Sistemas triangulares: ejemplo de coste y eficiencia

Resolver Lx = b con L triangular inferior. Por sustitución hacia delante:

for (i = 0; i < n; i++) {
    x[i] = b[i];
    for (j = 0; j < i; j++)
        x[i] -= L[i][j] * x[j];
    x[i] /= L[i][i];
}

Para calcular x[i] hace falta x[0], ..., x[i-1]. Cadena de dependencias. Coste secuencial: Σ(2 + 2i) ≈ n² + n.

Paralelizándolo por filas con difusión del valor calculado, el coste paralelo es aproximadamente 5n (n pasos, cada uno con cómputo y comunicación de 5 operaciones).

  • Speed-up: S = (n² + n) / 5n ≈ n / 5.
  • Eficiencia: E = S/p = n / (5p). Con p = n (un proceso por fila), E = 1/5 = 20%.

Las bajas prestaciones se deben a que el camino crítico (la cadena de dependencias) impide solapar todo: cada paso solo hay un dato «nuevo» que comunicar. Los problemas con cadenas largas escalan mal.

Computación pipeline

Cuando el grafo tiene la estructura de una cadena lineal de etapas, cada una pasando datos a la siguiente, se llama pipeline. Es la versión software de la segmentación que hace la propia CPU con sus instrucciones.

El pipeline tiene tres regímenes: arranque (las etapas se van llenando), régimen permanente (todas trabajan a la vez) y vaciado (se procesa lo que queda).

Análisis del pipeline

Con k etapas iguales de coste t cada una, y n datos a procesar:

  • Tiempo secuencial: cada dato pasa por las k etapas: tsec = n · k · t.
  • Tiempo pipeline: arranque (k − 1) + procesado de n datos + vaciado (incluido). Total ≈ (n + k − 1) · t.
  • Speed-up: S = n·k / (n + k − 1). Cuando n ≫ k, se aproxima a k.
CUÁNDO RENTABILIZA EL PIPELINE

Necesitas muchos datos en cola

El pipeline solo brilla cuando procesas muchos elementos: un único elemento no se acelera, porque pasa por todas las etapas en serie. Es el patrón natural cuando el problema es un flujo continuo de datos: vídeo, audio, sensores, gráficos por computador.

Limitaciones del pipeline

  • Etapa más lenta = cuello de botella: si una etapa tarda el doble que las demás, dicta el ritmo del régimen permanente. Hay que balancear los tiempos de las etapas.
  • Pocos elementos = poco speed-up: el arranque y el vaciado dominan.
  • Difícil ante feedback: si una etapa necesita salida de la siguiente, no es un DAG y no se puede segmentar igual.

Ejemplos de pipeline

Procesado de imagen

Captura → corrección de color → filtro de bordes → reconocimiento → guardado. Si tienes 100 imágenes a procesar, cada etapa puede trabajar en una distinta al mismo tiempo.

Compilación

Lexer → parser → análisis semántico → generación de código. Diseñado tradicionalmente como pipeline conceptual; en un compilador moderno son fases secuenciales para un único fichero, pero si compilas N ficheros pueden segmentarse.

Multiplicación matriz-vector sistólica

Patrón clásico en arquitecturas dedicadas: la matriz fluye por un anillo de procesos, cada uno multiplica su elemento y suma. Pipeline en hardware.

Clases de equivalencia: un ejemplo de paralelismo en árbol

Dado un conjunto de elementos agrupados en clases por una relación de equivalencia (representada como bosque de árboles donde cada elemento apunta a su «padre»), encontrar el representante de la clase de cada elemento.

Algoritmo paralelo: cada nodo lee el valor de su padre y se actualiza. Repetir. Después de log h pasos (h = altura), todos los nodos apuntan a la raíz.

El número de comunicaciones por nodo es como mucho una por paso, y el patrón es fijo: cada nodo siempre habla con el mismo padre. Es paralelismo en árbol con patrón estático.

Práctica de recuperación

Esta sesión introduce conceptos cuantitativos importantes: camino crítico, speed-up del pipeline, eficiencia con cadenas. Asegúrate de que recuerdas las fórmulas con ejemplos numéricos.

¿Qué es un grafo de dependencias?
Grafo dirigido acíclico (DAG) donde los nodos son tareas y las aristas representan que una tarea no puede empezar hasta que termine la otra. Los nodos pueden etiquetarse con coste de cómputo y las aristas con coste de comunicación.
¿Qué es el camino crítico?
La cadena más larga de tareas con dependencias en el grafo. Marca el tiempo mínimo absoluto que tardará el algoritmo, aunque tengas infinitos procesadores. Su longitud determina el límite teórico del speed-up.
¿Cuántos pasos necesita una reducción en árbol con p procesos?
log p pasos. En cada paso, la mitad de los procesos activos comunican su parcial a la otra mitad, que los suma. En total se hacen p − 1 sumas, pero distribuidas en log p fases.
Si el coste secuencial de un sistema triangular es n² + n y el paralelo es 5n, ¿cuál es la eficiencia con p = n procesos?
Speed-up = (n²+n)/(5n) ≈ n/5. Eficiencia = S/p = (n/5)/n = 1/5 = 20%. Eficiencia baja porque la cadena de dependencias impide el paralelismo masivo.
¿Qué es la computación pipeline?
Esquema donde el cómputo se divide en una cadena lineal de etapas. Cada etapa procesa un dato, lo pasa a la siguiente y empieza con el siguiente dato. En régimen permanente todas las etapas trabajan a la vez.
Para un pipeline de k etapas iguales y n datos, ¿cuál es el speed-up aproximado?
S = n·k / (n + k − 1). Cuando n ≫ k, se aproxima a k. Cuando n es del orden de k, el arranque y vaciado pesan y el speed-up queda muy por debajo.
¿Cuál es la limitación más típica del pipeline?
El cuello de botella de la etapa más lenta: dicta el ritmo del régimen permanente. Si una etapa tarda 2 ms y las demás 1 ms, el sistema produce un resultado cada 2 ms. Hay que balancear los tiempos.
Repaso de la sesión 9: ¿cómo se relacionan el grafo de dependencias y el paralelismo síncrono?
El síncrono es un caso particular: grafo «por niveles» donde todas las tareas del mismo nivel son independientes entre sí y dependen solo del nivel anterior. La barrera de fase es la sincronización implícita entre niveles.
¿Por qué el pipeline no acelera el procesado de un único dato?
Porque ese dato pasa por las k etapas secuencialmente. El pipeline gana paralelismo entre diferentes datos en distintas etapas a la vez. Sin múltiples datos, no hay paralelismo que extraer.
