Híbrida
y manycores

Combinar paradigmas no es elegancia académica: es lo que dicta la arquitectura real. Un cluster con multicores pide MPI+OpenMP, y si además hay GPU se suma CUDA.

Por qué la programación pura ya no basta

Imagina un cluster de 200 nodos con 32 cores por nodo y una GPU por nodo. Tienes varias opciones:

OpciónCómo aprovecha el hardwareProblema
Solo MPI, 6400 procesos Cada core un proceso MPI Excesivas comunicaciones intra-nodo y consumo de memoria (cada proceso replica estructuras)
Solo OpenMP Solo aprovecha un nodo Desperdicia 199 nodos
Solo CUDA Aprovecha la GPU del nodo donde se lanza CPUs y otros nodos ociosos
MPI + OpenMP + CUDA Un proceso MPI por nodo, OpenMP entre cores, kernels CUDA en la GPU Más complejo de programar, pero el único que aprovecha todo

Cada nivel de paralelismo ataca un nivel del hardware: MPI entre nodos, OpenMP entre cores de un nodo, SIMD/CUDA dentro del core o en la GPU.

Paralelismo anidado: el caso más simple

Antes de mezclar APIs, conviene entender el anidamiento dentro de OpenMP: una región paralela dentro de otra región paralela.

omp_set_nested(1);   // habilitar anidamiento

#pragma omp parallel num_threads(4)
{
    // 4 hilos exteriores
    #pragma omp parallel num_threads(3)
    {
        // cada uno de los 4 lanza 3 hilos interiores = 12 hilos
        trabajar();
    }
}

Útil cuando hay dos niveles naturales de paralelismo: por ejemplo, varias instancias de un problema que se resuelven en paralelo, y dentro de cada instancia hay bucles paralelizables.

Habilitar anidamiento no significa siempre que sea buena idea. Si los hilos exteriores ya saturan los cores, los interiores compiten por el mismo hardware. Regla práctica: anida cuando en el nivel exterior tengas pocas tareas grandes y heterogéneas.

MPI + OpenMP: el patrón estándar

La combinación más común en HPC. Un proceso MPI por nodo (o por socket), y OpenMP dentro para usar los cores.

Un proceso MPI por nodo. La comunicación entre nodos pasa por la red; dentro de cada nodo, los hilos OpenMP comparten memoria y se comunican mucho más rápido.

Inicialización: MPI_Init_thread

Cuando vas a usar MPI con hilos hay que declarar el nivel de uso:

int provided;
MPI_Init_thread(&argc, &argv, MPI_THREAD_FUNNELED, &provided);
  • MPI_THREAD_SINGLE: sin hilos. Equivalente a MPI_Init.
  • MPI_THREAD_FUNNELED: hay hilos OpenMP pero solo el maestro llama a MPI. El más usado.
  • MPI_THREAD_SERIALIZED: cualquier hilo puede llamar a MPI, pero no a la vez.
  • MPI_THREAD_MULTIPLE: cualquier hilo en cualquier momento. Máxima flexibilidad, máximo coste.

Ejemplo: multiplicación de matrices

// Cada proceso MPI recibe unas filas y multiplica con OpenMP
int filas_locales = N / nproc;

MPI_Scatter(A, filas_locales * N, MPI_DOUBLE,
            A_local, filas_locales * N, MPI_DOUBLE,
            0, MPI_COMM_WORLD);

MPI_Bcast(B, N * N, MPI_DOUBLE, 0, MPI_COMM_WORLD);

#pragma omp parallel for collapse(2)
for (int i = 0; i < filas_locales; i++)
    for (int j = 0; j < N; j++) {
        double s = 0;
        for (int k = 0; k < N; k++)
            s += A_local[i*N + k] * B[k*N + j];
        C_local[i*N + j] = s;
    }

MPI_Gather(C_local, filas_locales * N, MPI_DOUBLE,
           C, filas_locales * N, MPI_DOUBLE,
           0, MPI_COMM_WORLD);

GPU con CUDA

Las GPU son aceleradores con miles de cores muy simples. El paradigma es SIMD masivo: lanzas un kernel y se ejecuta sobre una grid de hilos, todos haciendo lo mismo sobre datos distintos.

El kernel descompone el problema en una grid de bloques; cada bloque en hilos. La GPU asigna automáticamente esos bloques a los Streaming Multiprocessors disponibles.

Esqueleto típico de programa CUDA

// Kernel: lo que ejecuta CADA hilo
__global__ void sumar_vectores(float *a, float *b, float *c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n)
        c[i] = a[i] + b[i];
}

int main() {
    float *a, *b, *c;          // host
    float *d_a, *d_b, *d_c;    // device (GPU)
    int n = 1 << 20;
    size_t bytes = n * sizeof(float);

    cudaMalloc(&d_a, bytes);
    cudaMalloc(&d_b, bytes);
    cudaMalloc(&d_c, bytes);

    cudaMemcpy(d_a, a, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(d_b, b, bytes, cudaMemcpyHostToDevice);

    int B = 256;
    int G = (n + B - 1) / B;
    sumar_vectores<<<G, B>>>(d_a, d_b, d_c, n);

    cudaMemcpy(c, d_c, bytes, cudaMemcpyDeviceToHost);

    cudaFree(d_a); cudaFree(d_b); cudaFree(d_c);
    return 0;
}

Para compilar: nvcc prog.cu -o prog.

CUIDADO CARO

Las transferencias host ↔ device son lentas

El bus PCIe es órdenes de magnitud más lento que la memoria de la GPU. Si copias datos de ida y vuelta para hacer poco cómputo, la GPU resulta más lenta que la CPU. Regla pragmática: transfiere una vez, ejecuta muchos kernels, recoge una vez.

Memoria de una GPU: jerarquía

No es una memoria homogénea como la RAM de un PC. Hay varios niveles con tamaños y velocidades muy distintos.

  • Registros (por hilo): rapidísimos, pocos.
  • Memoria compartida (por bloque): controlada por el programador con __shared__. Permite que los hilos de un bloque cooperen sin tocar memoria global. Decisiva para rendimiento.
  • Memoria global: la VRAM grande, accesible por todos los hilos. Lenta comparada con la compartida.
  • Memoria de constantes: cacheada, de solo lectura.
  • Memoria de texturas: cacheada, optimizada para acceso espacial.

Los accesos a memoria global deben ser coalescentes: hilos consecutivos deben leer direcciones consecutivas para que el hardware los fusione en una sola transacción.

Intel Xeon Phi (MIC)

Alternativa a las GPU. Coprocesador x86 con muchos cores (hasta 61) y unidades vectoriales muy anchas (512 bits). La gran ventaja respecto a CUDA: reutilizas código x86, no aprendes un lenguaje nuevo. OpenMP y MPI funcionan directamente.

Tres modos de ejecución:

  • Nativo: el programa se compila para el Phi y se ejecuta enteramente en él, sin host.
  • Simétrico: procesos MPI tanto en host como en Phi, vistos como nodos más.
  • Offload: el programa corre en host y descarga regiones concretas al Phi con directivas (similar al modelo CPU+GPU).

Intel descontinuó la línea Xeon Phi, pero las ideas continuaron en las extensiones AVX-512 de los Xeon convencionales.

Paralelismo multinivel

El concepto general: cada nivel de paralelismo (anidado, híbrido, heterogéneo) usa un mecanismo distinto. Un programa real puede tener:

  • Nivel 1 — explícito: MPI entre nodos (el programador escribe los send).
  • Nivel 2 — explícito: OpenMP entre cores (el programador pone #pragma omp).
  • Nivel 3 — implícito: BLAS multithreaded en cada operación matricial (la librería ya está paralelizada).
  • Nivel 4 — implícito: el compilador vectoriza con AVX automáticamente.

Lo difícil del multinivel no es escribirlo: es medir qué configuración (cuántos procesos × cuántos hilos × cuántas threads vectoriales) da las mejores prestaciones en un sistema concreto. Las prácticas del curso piden exactamente eso.

Sesión 06 · Análisis

El análisis de prestaciones de un programa híbrido es más complejo porque hay que estudiar speed-up y eficiencia variando dos o tres parámetros (procesos, hilos, configuración GPU) a la vez.

Práctica de recuperación

Hay preguntas que combinan conceptos de varias sesiones deliberadamente. Discriminar OpenMP de MPI de CUDA es exactamente lo que se evalúa en el examen.

En un cluster con multicores, ¿por qué no usar simplemente un proceso MPI por core?
Cada proceso MPI tiene su propia memoria y replica estructuras: gasta más RAM. Y las comunicaciones intra-nodo entre tantos procesos sobrecargan el sistema. Es más eficiente un proceso MPI por nodo (o socket) que use OpenMP entre los cores.
¿Qué hace MPI_Init_thread y qué nivel se elige normalmente?
Es la inicialización de MPI cuando vas a usar hilos. Le dices qué nivel necesitas. Lo más común es MPI_THREAD_FUNNELED: hay hilos OpenMP pero solo el maestro hace llamadas MPI.
¿Qué es un kernel CUDA?
Una función que se ejecuta en la GPU, lanzada desde la CPU con kernel<<<grid, bloque>>>(args). Una sola llamada lanza miles de hilos, cada uno con su threadIdx y blockIdx para identificar qué dato le toca.
¿Por qué la transferencia host↔device puede arruinar el rendimiento de un programa CUDA?
El bus PCIe es mucho más lento que la memoria interna de la GPU. Si por cada cómputo pequeño copias datos ida y vuelta, el tiempo de transferencia domina y la GPU resulta más lenta que la CPU.
¿Qué es la memoria compartida (__shared__) en CUDA?
Memoria muy rápida asociada a un bloque de hilos, gestionada explícitamente por el programador. Permite que los hilos de un mismo bloque cooperen sin pasar por la memoria global. Su buen uso es la clave de los algoritmos GPU eficientes.
¿Qué significa "acceso coalescente" a memoria global de una GPU?
Que hilos consecutivos (dentro de un warp) leen posiciones consecutivas de memoria. Cuando se cumple, el hardware combina los accesos en una sola transacción. Cuando no, cada hilo paga su propia transacción y el rendimiento cae en picado.
¿Cuál es la gran ventaja de Xeon Phi sobre CUDA?
Es x86: el código secuencial existente compila y se ejecuta directamente. OpenMP y MPI funcionan sin cambios. CUDA, en cambio, exige reescribir las partes paralelas en su modelo específico.
Repaso de la sesión 3: ¿qué hace collapse(2) en un #pragma omp parallel for?
Combina dos bucles anidados en uno solo a efectos de reparto entre hilos. Aumenta la granularidad disponible y, en bucles cuadrados pequeños, evita que solo se paralelice el bucle exterior.
¿Qué es el paralelismo multinivel implícito? Pon un ejemplo.
El que no escribe el programador sino que aporta una librería o el compilador. Ejemplos: BLAS multihilo dentro de una llamada a dgemm, vectorización automática AVX que aplica el compilador a un bucle.
