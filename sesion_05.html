Híbrida
y manycores

Combinar paradigmas no es elegancia académica: es lo que dicta la arquitectura real. Un cluster con multicores pide MPI+OpenMP, y si además hay GPU se suma CUDA.

Los sistemas computacionales no son monolíticos

La sesión 1 introdujo los tipos de máquinas; aquí los volvemos a juntar bajo otra óptica: la mayoría de los sistemas reales son a la vez híbridos, heterogéneos y jerárquicos.

Híbridos

Combinan distintas características: memoria compartida + memoria distribuida, multicore + GPU, varios sistemas en red…

Heterogéneos

Sus componentes no son iguales: memorias de distinta capacidad y estructura de acceso, nodos a distinta velocidad, redes de conexión de prestaciones distintas…

Jerárquicos

Se estructuran en capas: varios sistemas en red → cada uno un cluster o cc-NUMA → cada cluster con varios nodos → cada nodo con varios sockets → cada socket con varios cores.

Ejemplos de sistemas reales

  • Portátil o sobremesa moderno: dual/quad/hexa-core, posible hyperthreading, GPU integrada o discreta con cientos a miles de cores. Híbrido (CPU+GPU), heterogéneo (jerarquía de memorias L1/L2/L3 + VRAM), jerárquico (host lanza kernels a la GPU).
  • Ben-Arabí (antiguo Centro de Supercomputación de Murcia): 128 cores cc-NUMA (Ben) + 816 cores en cluster de 102 nodos (Arabí). Híbrido (MC + MD), heterogéneo (Ben y Arabí a distinta velocidad y memoria) y jerárquico.
  • Marenostrum 5 (BSC, 314 PFlops pico): partición general (Intel Sapphire Rapids, 6408 nodos) + partición acelerada (Sapphire Rapids + 4 NVIDIA Hopper por nodo en 1120 nodos, 230 PFlops). Combina MC + MD + GPU.
  • Pirineus (CESGA): 1344 cores en memoria compartida en nodos hexacore. Tendencia general: combinar MC + MD + coprocesadores (GPU, Xeon Phi, FPGA) — consultar el TOP500.
Para introspeccionar el sistema (cores, sockets, niveles de cache, NUMA…) la herramienta de referencia es hwloc (open-mpi.org/projects/hwloc). Una lstopo rápida te dice cuánto paralelismo tienes y a qué granularidad lo puedes explotar.

Por qué la programación pura ya no basta

Imagina un cluster de 200 nodos con 32 cores por nodo y una GPU por nodo. Tienes varias opciones:

OpciónCómo aprovecha el hardwareProblema
Solo MPI, 6400 procesos Cada core un proceso MPI Excesivas comunicaciones intra-nodo y consumo de memoria (cada proceso replica estructuras)
Solo OpenMP Solo aprovecha un nodo Desperdicia 199 nodos
Solo CUDA Aprovecha la GPU del nodo donde se lanza CPUs y otros nodos ociosos
MPI + OpenMP + CUDA Un proceso MPI por nodo, OpenMP entre cores, kernels CUDA en la GPU Más complejo de programar, pero el único que aprovecha todo

Cada nivel de paralelismo ataca un nivel del hardware: MPI entre nodos, OpenMP entre cores de un nodo, SIMD/CUDA dentro del core o en la GPU. Las posibilidades del PDF oficial son:

  • Paralelismo anidado en OpenMP.
  • Paralelismo multinivel: hilos OpenMP llamando a rutinas paralelas (BLAS/MKL multithreaded).
  • Programación híbrida MPI + OpenMP: procesos MPI con hilos OpenMP dentro.
  • Combinar los tres: MPI + OpenMP + MKL.
  • Computación heterogénea, adaptativa, grid/cloud/P2P…

Paralelismo anidado: el caso más simple

Antes de mezclar APIs, conviene entender el anidamiento dentro de OpenMP: una región paralela dentro de otra región paralela. El thread inicial crea un equipo de esclavos; cada esclavo crea, a su vez, otro equipo del que es maestro.

Modelo de la diapositiva oficial (p. 13 de Hibrida.pdf): un nivel exterior con dos hilos, cada uno creando un equipo de tres en el nivel interior.

omp_set_nested(1);   // habilitar anidamiento (o OMP_NESTED=true)

#pragma omp parallel num_threads(2)
{
    #pragma omp parallel num_threads(3)
    {
        // 2 × 3 = 6 hilos totales en el nivel interior
        trabajar();
    }
}

Útil para:

  • Mejorar el uso de una memoria organizada jerárquicamente.
  • Asignar pocas tareas costosas: un hilo exterior por cada tarea, y dentro un equipo distinto que resuelve esa tarea.
Habilitar anidamiento no significa siempre que sea buena idea. Si los hilos exteriores ya saturan los cores, los interiores compiten por el mismo hardware. Regla práctica: anida cuando en el nivel exterior tengas pocas tareas grandes y heterogéneas.

MPI + OpenMP: el patrón estándar

La combinación más común en HPC. Un proceso MPI por nodo (o por socket), y OpenMP dentro para usar los cores.

Un proceso MPI por nodo. La comunicación entre nodos pasa por la red; dentro de cada nodo, los hilos OpenMP comparten memoria y se comunican mucho más rápido.

Inicialización: MPI_Init_thread

Cuando vas a usar MPI con hilos hay que declarar el nivel de uso:

int provided;
MPI_Init_thread(&argc, &argv, MPI_THREAD_FUNNELED, &provided);
  • MPI_THREAD_SINGLE: sin hilos. Equivalente a MPI_Init.
  • MPI_THREAD_FUNNELED: hay hilos OpenMP pero solo el maestro llama a MPI. El más usado.
  • MPI_THREAD_SERIALIZED: cualquier hilo puede llamar a MPI, pero no a la vez.
  • MPI_THREAD_MULTIPLE: cualquier hilo en cualquier momento. Máxima flexibilidad, máximo coste.

El parámetro provided devuelve el nivel realmente garantizado por la implementación; comprueba que es ≥ al que pediste.

Hello híbrido (PDF p. 15)

int main(int argc, char *argv[]) {
    int nthreads, nprocs, idpro, idthr, namelen;
    char processor_name[MPI_MAX_PROCESSOR_NAME];

    MPI_Init(&argc, &argv);
    MPI_Comm_size(MPI_COMM_WORLD, &nprocs);
    MPI_Comm_rank(MPI_COMM_WORLD, &idpro);
    MPI_Get_processor_name(processor_name, &namelen);

    #pragma omp parallel private(idthr) firstprivate(idpro, processor_name)
    {
        idthr = omp_get_thread_num();
        printf("thread %d, proceso %d, procesador %s\n",
               idthr, idpro, processor_name);
        if (idthr == 0) {
            nthreads = omp_get_num_threads();
            printf("proceso %d, threads %d, procesos %d\n",
                   idpro, nthreads, nprocs);
        }
    }

    MPI_Finalize();
}

Por qué firstprivate(idpro, processor_name): estas variables se inicializaron antes del parallel y queremos que cada hilo las vea con su valor (que es común al proceso MPI). Sin la cláusula serían shared por defecto —técnicamente correcto— pero firstprivate documenta intención y evita malentendidos.

Multiplicación de matrices en MPI + OpenMP

Versión del PDF p. 16-20: las filas de A se reparten entre procesos MPI; cada proceso paraleliza con OpenMP el bucle exterior de su trozo.

Reparto inicial

MPI_Init(&argc, &argv);
MPI_Comm_size(MPI_COMM_WORLD, &np);
MPI_Comm_rank(MPI_COMM_WORLD, &nodo);

if (nodo == 0) {
    N          = atoi(argv[1]);
    NUMTHREADS = atoi(argv[2]);
}
MPI_Bcast(&N,          1, MPI_INT, 0, MPI_COMM_WORLD);
MPI_Bcast(&NUMTHREADS, 1, MPI_INT, 0, MPI_COMM_WORLD);
omp_set_num_threads(NUMTHREADS);

if (nodo == 0) {                              // reparte A por filas
    for (i = 1; i < np; i++)
        MPI_Send(&a[i*lda*N/np], fal*ca, MPI_DOUBLE,
                 i, 20, MPI_COMM_WORLD);
    MPI_Bcast(b, fb*cb, MPI_DOUBLE, 0, MPI_COMM_WORLD);
} else {
    MPI_Recv(a, fal*ca, MPI_DOUBLE, 0, 20,
             MPI_COMM_WORLD, &estado);
    MPI_Bcast(b, fb*cb, MPI_DOUBLE, 0, MPI_COMM_WORLD);
}

Cómputo local con OpenMP

void mm(double *a, int fa, int ca, int lda, …) {
    #pragma omp parallel
    {
        printf("Proceso %d, nodo %s, thread %d\n",
               nodo, maquina, omp_get_thread_num());

        #pragma omp for private(i,j,k,s) schedule(static)
        for (i = 0; i < fa; i++)
            for (j = 0; j < cb; j++) {
                s = 0.;
                for (k = 0; k < ca; k++)
                    s += a[i*lda + k] * b[k*ldb + j];
                c[i*ldc + j] = s;
            }
    }
}

Recolección y medida

MPI_Barrier(MPI_COMM_WORLD);
ti = MPI_Wtime();

mm(a, fal, ca, lda, b, fb, cb, ldb, c, fcl, cc, ldc, nodo, nombre);

MPI_Barrier(MPI_COMM_WORLD);
tf = MPI_Wtime();

if (nodo == 0) {
    for (i = 1; i < np; i++)
        MPI_Recv(&c[i*ldc*N/np], fcl*cc, MPI_DOUBLE,
                 i, 30, MPI_COMM_WORLD, &estado);
} else {
    MPI_Send(c, fcl*cc, MPI_DOUBLE, 0, 30, MPI_COMM_WORLD);
}

Por qué dos MPI_Barrier: garantizan que ti y tf miden exactamente la fase de cómputo del proceso más lento, no la mezcla con la recolección. Mediciones honestas en programas paralelos requieren acotar bien el tramo.

Una versión más limpia usaría MPI_Scatter para repartir A y MPI_Gather para recoger C. La versión del PDF mantiene el bucle de Send/Recv para que se vea explícitamente quién manda qué. En la práctica, prefiere colectivas.

Paralelismo multinivel con MKL

El paralelismo anidado y el híbrido son casos particulares de paralelismo multinivel. A veces un nivel es explícito (OpenMP) y otro implícito (BLAS multithreaded como MKL). Es posible apilarlos: MPI + OpenMP + MKL.

Funciones de MKL

  • mkl_set_num_threads(n) o variable MKL_NUM_THREADS: número de hilos para las rutinas MKL.
  • mkl_set_dynamic(1) o MKL_DYNAMIC=true: permite al runtime decidir cuántos hilos usar (≤ valor establecido).
  • mkl_set_dynamic(0) o MKL_DYNAMIC=false: fuerza exactamente el número indicado. Necesario cuando quieres una configuración concreta.

Multiplicación OpenMP + MKL

omp_set_nested(1);                // nivel exterior OpenMP
omp_set_num_threads(nthomp);
mkl_set_dynamic(0);
mkl_set_num_threads(nthmkl);       // nivel interior MKL

#pragma omp parallel
{
    // 1. determinar mi submatriz de A según tid
    int tid     = omp_get_thread_num();
    int filas   = N / nthomp;
    int inicio  = tid * filas;

    // 2. dgemm de MKL multiplica esa submatriz por B
    //    cada llamada usa nthmkl hilos internos
    cblas_dgemm(CblasRowMajor, CblasNoTrans, CblasNoTrans,
                filas, N, N, 1.0, &A[inicio*N], N,
                B, N, 0.0, &C[inicio*N], N);
}

El programa usa nthomp × nthmkl hilos totales. Con 16 cores, configuraciones típicas a comparar: 16×1, 8×2, 4×4, 2×8, 1×16. La óptima depende del tamaño y del sistema: en multiplicaciones grandes MKL escala bien internamente; en muchas multiplicaciones pequeñas paralelas conviene más OpenMP en el nivel exterior.

DECISIÓN CLAVE

¿Cuántos hilos en cada nivel?

El producto nthomp × nthmkl debe coincidir aproximadamente con el número de cores físicos. Pasarse → contención. Quedarse corto → desaprovechas hardware. La distribución entre niveles se decide empíricamente: prueba combinaciones y mide.

Computación heterogénea

Cuando los componentes del sistema no son iguales (procesadores a distinta velocidad, nodos con distinta memoria), un reparto homogéneo del trabajo desequilibra: los procesadores lentos retrasan al conjunto. Estrategias del PDF:

  • Programa homogéneo en sistema heterogéneo: asignar más procesos a los procesadores rápidos.
  • Programa heterogéneo: la cantidad de trabajo por proceso es proporcional a la velocidad del procesador asignado.
  • Multinivel adaptativo: distinto número de hilos por subgrupo según la velocidad del nodo o el volumen de cómputo.
  • OpenMP + GPU: decidir en cada parte del programa si se asigna a GPU o multicore, según el coste y el tipo de operación.
  • Bolsa de tareas: cola común de la que cada procesador coge trabajo cuando termina lo anterior. Auto-balanceo. (Lo veremos en la sesión 12.)

La GPGPU es heterogénea por definición: distintos tipos de memoria (registros, shared, global, constant, texture) con coste de acceso muy distinto.

Sistemas manycore

Cuatro familias principales de coprocesadores especializados:

TecnologíaQué esProgramación
GPU (Graphic Processing Units) Originalmente para gráficos; muchos cores pequeños. GPGPU usa GPU para cómputo general. CUDA (NVIDIA), OpenCL, OpenACC
MIC (Many Integrated Cores) Alternativa de Intel: Xeon Phi. Cores x86 simples con SIMD muy ancho. OpenMP, MPI directamente; offload con directivas

En este curso miramos en detalle GPU (con CUDA) y Xeon Phi.

GPGPU y CUDA

Las GPU son aceleradores con miles de cores muy simples. El paradigma es SIMT (Single Instruction, Multiple Thread): lanzas un kernel y se ejecuta sobre una grid de hilos, todos haciendo lo mismo sobre datos distintos.

Hitos del software

  • 2006: NVIDIA presenta CUDA con la GeForce 8800. Arquitectura unificada hardware+software. Modelos compatibles: GeForce, Tesla, Quadro.
  • 2009: OpenCL — intento de estándar abierto multi-fabricante.
  • 2012: OpenACC — directivas estilo OpenMP para acelerar código en CPU+GPU.
  • Librerías optimizadas para GPU: álgebra lineal (cuBLAS), ordenación, transformada de Fourier (cuFFT)…

Arquitectura hardware

Una GPU se compone de N Streaming Multiprocessors (SMs), y cada SM contiene M Streaming Processors (SPs):

GPU = N × Streaming Multiprocessors; cada SM = M × Streaming Processors compartiendo una Instruction Unit. Por eso todos los SP de un mismo SM ejecutan la misma instrucción en cada instante: SIMT.

Modelo de programación: kernel, grid, block, thread

  • Las partes paralelizadas para ejecutar en GPU se llaman kernels.
  • Cada kernel descompone el problema y lo mapea sobre una grid (vector 1D, 2D o 3D de thread blocks). Cada block tiene su blockIdx (X, Y, Z).
  • Cada thread block es a su vez 1D/2D/3D de threads, identificados por threadIdx (X, Y, Z).
  • Los hilos usan blockIdx, threadIdx, blockDim, gridDim para calcular qué dato les toca procesar.
  • Los hilos son gestionados por el hardware del SM: creación y cambios de contexto con coste despreciable.

El kernel descompone el problema en una grid de bloques; cada bloque en hilos. El runtime asigna esos bloques a los SM disponibles según vayan quedando libres.

Ejemplo: saxpy — y = α x + y (PDF p. 11)

// Versión secuencial
void saxpy_serial(int n, float *y, float alpha, float *x) {
    for (int i = 0; i < n; i++)
        y[i] = alpha * x[i] + y[i];
}
saxpy_serial(n, y, 2.0, x);

// Versión paralela CUDA: kernel
__global__
void saxpy_parallel(int n, float *y, float alpha, float *x) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) y[i] = alpha * x[i] + y[i];
}

// Llamada al kernel desde el host
int nblocks = (n + 255) / 256;          // redondeo hacia arriba
saxpy_parallel<<<nblocks, 256>>>(n, y, 2.0, x);

Lectura del kernel:

  • __global__ marca la función como kernel: se ejecuta en GPU, se llama desde CPU.
  • blockIdx.x * blockDim.x + threadIdx.x es la fórmula canónica para convertir (bloque, hilo) en un índice global lineal.
  • El if (i < n) es imprescindible: si n no es múltiplo de 256, sobran hilos en el último bloque y no deben tocar memoria fuera de rango.

Esqueleto completo con gestión de memoria

int main() {
    float *x, *y;              // host
    float *d_x, *d_y;          // device (GPU)
    int n = 1 << 20;
    size_t bytes = n * sizeof(float);

    x = malloc(bytes);  y = malloc(bytes);
    // ... inicializar x, y en CPU ...

    cudaMalloc(&d_x, bytes);
    cudaMalloc(&d_y, bytes);
    cudaMemcpy(d_x, x, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(d_y, y, bytes, cudaMemcpyHostToDevice);

    saxpy_parallel<<<(n + 255) / 256, 256>>>(n, d_y, 2.0, d_x);

    cudaMemcpy(y, d_y, bytes, cudaMemcpyDeviceToHost);
    cudaFree(d_x); cudaFree(d_y);
    free(x); free(y);
}

Compilación con NVIDIA: nvcc prog.cu -o prog.

CUIDADO CARO

Las transferencias host ↔ device son lentas

El bus PCIe es órdenes de magnitud más lento que la memoria de la GPU. Si copias datos de ida y vuelta para hacer poco cómputo, la GPU resulta más lenta que la CPU. Regla pragmática: transfiere una vez, ejecuta muchos kernels, recoge una vez.

Modelo de memoria de la GPU

No es homogénea como la RAM de un PC. Hay varios niveles con tamaños, velocidades y ámbitos muy distintos. Optimizar un programa CUDA es, casi siempre, mover datos al nivel correcto.

Los accesos a memoria global deben ser coalescentes: hilos consecutivos (dentro de un warp de 32 hilos) deben leer direcciones consecutivas para que el hardware los fusione en una sola transacción. Si no, cada hilo paga su transacción y el rendimiento cae en picado.

Intel Xeon Phi (MIC)

Alternativa a las GPU. Coprocesador x86 con muchos cores y unidades vectoriales muy anchas. La gran ventaja respecto a CUDA: reutilizas código x86, no aprendes un lenguaje nuevo. OpenMP y MPI funcionan directamente.

Características de hardware

  • Hasta 61 cores a 1.0–1.3 GHz, basados en x86 (Pentium) con direccionamiento de 64 bits.
  • Cada core con una VPU (Vector Processing Unit) con 32 registros de 512 bits (SIMD).
  • Cada core soporta 4 threads por hardware (hyperthreading de 4 vías).
  • Interconexión de bus en anillo bidireccional.
  • Hasta 6 GB de GDDR5.

Software: MPSS y compiladores offload

  • Intel MPSS (Manycore Platform Software Stack): permite que los programas se ejecuten y se comuniquen con el coprocesador.
  • Compiladores Offload (Intel C/C++ y Fortran 2013): generan código que se ejecuta solo en host, solo en coprocesador, o parejas de binarios que se comunican entre ellos.
  • Entornos de programación paralela: los mismos que en host — Intel TBB, Pthreads, OpenMP, MPI.

Funcionamiento interno

El coprocesador tiene memoria flash con BIOS, bootloader, firmware y un kernel embebido. Ejecuta un microkernel de Linux que se carga desde el host al arrancarlo. No tiene acceso a sistema de ficheros permanente: usa RAM (residente en memoria del coprocesador) o NFS montado desde el host.

Modos de ejecución

ModoDónde corre el códigoCuándo conviene
Nativo Enteramente en el Phi (lanzable desde host o coprocesador) Kernel computacional puro, sin interacción con host
Simétrico Procesos MPI en host y en Phi, vistos como nodos cualesquiera Aprovechar simultáneamente CPU y MIC
Offload Programa principal en host; regiones marcadas se descargan al Phi Acelerar partes específicas, similar al modelo CPU+GPU

Ejemplo offload (PDF p. 18)

// Versión secuencial offload
double reduction(double *data, int size) {
    double ret = 0.0;
    #pragma offload target(mic) in(data:length(size))
    for (int i = 0; i < size; i++) ret += data[i];
    return ret;
}

// Versión con OpenMP dentro del offload
double omp_reduction(double *data, int size) {
    double ret = 0.0;
    #pragma offload target(mic) in(data:length(size))
    {
        #pragma omp parallel for reduction(+: ret)
        for (int i = 0; i < size; i++) ret += data[i];
    }
    return ret;
}

in(data:length(size)) indica que data es entrada y especifica cuántos elementos copiar al Phi. Hay también out(...) y inout(...) según el sentido de la transferencia.

MPI en Phi

Tres modos: Nativo (todos los rangos en el coprocesador), Simétrico (rangos en host y Phi), Offload (rangos en host que descargan trabajo al Phi vía compilador Intel).

Intel descontinuó la línea Xeon Phi, pero las ideas continuaron en las extensiones AVX-512 de los Xeon convencionales.

Ejercicio 1 · describir el código de multiplicación de matrices MPI+OpenMP

Enunciado: describir el ejemplo de Hibrida.pdf, comentando cómo está implementado el paralelismo en cada nivel y qué hace cada función.

Descripción por niveles:

  • Nivel MPI (entre nodos): el proceso 0 actúa como coordinador. Lee N (tamaño) y NUMTHREADS de los argumentos y los difunde con dos MPI_Bcast. Reparte la matriz A por filas: N/np filas a cada proceso mediante un bucle de MPI_Send (el resto del proceso 0 también queda con su trozo). La matriz B la difunde entera por MPI_Bcast, ya que todos la necesitan completa para multiplicar.
  • Nivel OpenMP (dentro de cada proceso): tras la barrera de sincronización (para que la medida sea honesta), cada proceso ejecuta la función mm, que abre una región #pragma omp parallel. Dentro hay un #pragma omp for con schedule(static) y private(i,j,k,s) que reparte el bucle exterior i (filas de la submatriz local) entre los hilos del equipo.
  • Recolección: tras una segunda barrera, el proceso 0 recibe los trozos de C de los demás con MPI_Recv en orden, ensamblando la matriz resultado.

Función mm(...):

  • Parámetros: punteros a A, B, C con sus dimensiones lógicas y físicas (fa, ca, lda, etc.); el rango y nombre del nodo para depuración.
  • Tres bucles anidados clásicos del producto matricial. Solo el exterior i se paraleliza con OpenMP; j y k quedan secuenciales dentro de cada hilo. private(i,j,k,s) evita condiciones de carrera sobre los índices y el acumulador.
  • schedule(static) reparte las filas en bloques contiguos iguales: adecuado porque todas las iteraciones cuestan lo mismo (todas multiplican una fila de A por la matriz B).

Parámetros de invocación: dos enteros desde la línea de comandos: tamaño N y número de hilos OpenMP. El número de procesos MPI lo fija mpirun -np. Combinaciones típicas a probar: 1 proceso × 16 hilos, 2 × 8, 4 × 4, 8 × 2, 16 × 1.

Ejercicio 2 · describir EjemploXeonPhi.cpp

Enunciado: describir cálculos, funciones, constructores OpenMP, directivas para ejecución en el MIC y parámetros del programa de ejemplo.

El programa modelo (suma de matrices) sigue el patrón:

  • Cálculo: suma elemento a elemento de dos matrices A y B en una tercera C. Algoritmo trivialmente paralelizable; sirve para mostrar la mecánica del offload sin distraer con álgebra.
  • Funciones: una rutina de inicialización (en host), una rutina suma_mic que contiene la directiva offload y dentro paraleliza con OpenMP, y código de verificación/medida en host.
  • Constructores OpenMP: #pragma omp parallel for sobre el bucle de filas, normalmente con schedule(static) por uniformidad de las iteraciones; podría usarse collapse(2) si se quiere fusionar los dos bucles exteriores para más paralelismo.
  • Directivas MIC:
    • #pragma offload target(mic) abre la región que se descarga al coprocesador.
    • in(A:length(N*N)), in(B:length(N*N)): copia las matrices al Phi al entrar.
    • out(C:length(N*N)): copia la matriz resultado del Phi al host al salir.
    • target(mic:0) selecciona el coprocesador 0 si hay varios; sin número, el sistema elige.
  • Parámetros: típicamente tamaño N por línea de comandos y número de hilos OpenMP (vía OMP_NUM_THREADS o omp_set_num_threads). En el modo offload conviene fijar variables específicas para los hilos del Phi: MIC_OMP_NUM_THREADS=240 aprovecharía 60 cores × 4 threads/core.

Compilación: con el compilador Intel (icc o icpc) habilitando -qopenmp. No necesita flags especiales para el offload: las directivas se interpretan si el compilador soporta MIC; si no, se ignoran y el código corre todo en host.

Decisiones de diseño en programación multinivel

Lo difícil del multinivel no es escribirlo, es medir qué configuración da las mejores prestaciones. Algunas reglas heurísticas:

  • Procesos MPI × hilos OpenMP ≈ cores físicos. Pasarse provoca contención; quedarse corto desaprovecha hardware.
  • Un proceso MPI por socket NUMA. En máquinas multi-socket, los hilos de un mismo proceso deben compartir el mismo banco de memoria local; cruzar sockets duplica latencia.
  • Granularidad gruesa en MPI, fina en OpenMP. Las comunicaciones MPI son caras; los fork/join de OpenMP, baratos. Comunica poco y a lo grande entre procesos; paraleliza con libertad dentro.
  • Si hay GPU, llévala a saturar. Una GPU ociosa con CPU a tope es desperdicio: vale la pena rediseñar el reparto del trabajo.
  • Mide siempre. El óptimo cambia con el tamaño del problema y el sistema. Las prácticas piden tablas de configuraciones; no las inventes.
Sesión 06 · Análisis

El análisis de prestaciones de un programa híbrido es más complejo: hay que estudiar speed-up y eficiencia variando dos o tres parámetros (procesos, hilos, configuración GPU) a la vez.

Práctica de recuperación

Hay preguntas que combinan conceptos de varias sesiones deliberadamente. Discriminar OpenMP de MPI de CUDA es exactamente lo que se evalúa en el examen.

¿Qué significa que un sistema sea «híbrido, heterogéneo y jerárquico»?
Híbrido: combina características distintas (MC + MD, multicore + GPU). Heterogéneo: sus componentes no son iguales (memorias, velocidades, redes). Jerárquico: se estructura en capas (red → cluster → nodo → socket → core). La mayoría de los sistemas reales son las tres cosas a la vez.
En un cluster con multicores, ¿por qué no usar simplemente un proceso MPI por core?
Cada proceso MPI tiene su propia memoria y replica estructuras: gasta más RAM. Las comunicaciones intra-nodo entre tantos procesos sobrecargan el sistema. Es más eficiente un proceso MPI por nodo (o socket) que use OpenMP entre los cores compartiendo memoria.
¿Qué hace MPI_Init_thread y qué nivel se elige normalmente?
Es la inicialización de MPI cuando vas a usar hilos. Le dices qué nivel necesitas. Lo más común es MPI_THREAD_FUNNELED: hay hilos OpenMP pero solo el maestro llama a MPI. El parámetro provided devuelve el nivel realmente garantizado; verifícalo.
¿Cómo se distribuye el trabajo en MPI+OpenMP+MKL en una multiplicación de matrices?
MPI reparte las filas de A entre nodos. Dentro de cada nodo, OpenMP reparte su submatriz entre hilos. Cada hilo llama a dgemm de MKL, que a su vez se paraleliza con sus propios hilos internos. El número total de hilos es nthomp × nthmkl; conviene que coincida con los cores físicos.
¿Para qué sirve mkl_set_dynamic(0)?
Para forzar exactamente el número de hilos que indiques con mkl_set_num_threads. Sin desactivar el dinámico, MKL puede usar menos hilos de los pedidos. Si quieres comparar configuraciones, fija el número.
¿Qué es un kernel CUDA?
Una función marcada con __global__ que se ejecuta en la GPU, lanzada desde la CPU con kernel<<<grid, bloque>>>(args). Una sola llamada lanza miles de hilos, cada uno con su threadIdx y blockIdx para identificar qué dato le toca.
¿Qué relación hay entre un SM y un SP en una GPU NVIDIA?
Una GPU se compone de N Streaming Multiprocessors (SM). Cada SM contiene M Streaming Processors (SP) que comparten una Instruction Unit: por eso todos ejecutan la misma instrucción al mismo tiempo (modelo SIMT). Las cifras N y M dependen del modelo.
¿Por qué la transferencia host↔device puede arruinar el rendimiento de un programa CUDA?
El bus PCIe es mucho más lento que la memoria interna de la GPU. Si por cada cómputo pequeño copias datos ida y vuelta, el tiempo de transferencia domina y la GPU resulta más lenta que la CPU. Patrón correcto: transferir, lanzar muchos kernels, recoger.
¿Qué es la memoria compartida (__shared__) en CUDA?
Memoria muy rápida asociada a un bloque de hilos, gestionada explícitamente por el programador. Permite que los hilos de un mismo bloque cooperen sin pasar por la memoria global. Su buen uso es la clave de los algoritmos GPU eficientes.
¿Qué significa «acceso coalescente» a memoria global de una GPU?
Que hilos consecutivos (dentro de un warp de 32) leen posiciones consecutivas de memoria. Cuando se cumple, el hardware combina los accesos en una sola transacción. Cuando no, cada hilo paga su propia transacción y el rendimiento cae en picado.
¿Cuál es la gran ventaja de Xeon Phi sobre CUDA?
Es x86: el código secuencial existente compila y se ejecuta directamente. OpenMP y MPI funcionan sin cambios. CUDA, en cambio, exige reescribir las partes paralelas en su modelo específico (kernels, grids, gestión explícita de memoria).
Diferencia entre los modos «nativo», «simétrico» y «offload» de Xeon Phi.
Nativo: todo el programa corre en el Phi. Simétrico: rangos MPI tanto en host como en Phi, tratados como nodos cualesquiera. Offload: el programa corre en host y descarga regiones marcadas con #pragma offload target(mic) al coprocesador.
Repaso de la sesión 3: ¿qué hace collapse(2) en un #pragma omp parallel for?
Combina dos bucles anidados en uno solo a efectos de reparto entre hilos. Aumenta la granularidad disponible y, en bucles cuadrados pequeños, evita que solo se paralelice el bucle exterior.
¿Qué es el paralelismo multinivel implícito? Pon un ejemplo.
El que no escribe el programador sino que aporta una librería o el compilador. Ejemplos: BLAS multihilo dentro de una llamada a dgemm; vectorización automática AVX que aplica el compilador a un bucle.
