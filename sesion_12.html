Trabajadores
replicados

Cuando el trabajo no se puede repartir a priori porque su tamaño y forma son impredecibles. Esquemas dinámicos: bolsa de tareas, maestro-esclavo, recorridos de árboles.

El problema del trabajo impredecible

Hasta ahora hemos asumido que se conocía a priori cuánto trabajo había y cómo se podía repartir. En muchos problemas reales ese supuesto no se cumple:

  • El árbol de búsqueda crece de forma impredecible (juegos, optimización combinatoria).
  • Las tareas tienen costes muy distintos y desconocidos.
  • Se generan nuevas tareas durante la ejecución (ejemplo: simulación de partículas que colisionan).

El reparto estático falla: si distribuyes la mitad inicial del árbol, puede que una mitad se resuelva en un segundo y la otra requiera horas. Hace falta balanceo dinámico: los procesos piden trabajo según lo terminan.

Recorrido de árboles: backtracking

Búsqueda exhaustiva con vuelta atrás. Para cada decisión, se prueban todas las opciones; si llega a un callejón sin salida, se retrocede.

Ejemplo clásico: N reinas

Colocar N reinas en un tablero N×N sin que se ataquen. Recorrido en profundidad probando posiciones, retrocediendo cuando una posición es inválida.

void reinas(int *col, int fila, int n) {
    if (fila == n) {
        contar_solucion(col);
        return;
    }
    for (int c = 0; c < n; c++) {
        if (es_valida(col, fila, c)) {
            col[fila] = c;
            reinas(col, fila + 1, n);   // rama
        }
    }
}

Paralelización

Cada rama del árbol de búsqueda puede explorarse en paralelo. Con OpenMP tareas:

void reinas_par(int *col, int fila, int n, int nivel_corte) {
    if (fila == n) { contar_solucion(col); return; }

    if (fila < nivel_corte) {
        // paralelizar: cada rama es una tarea
        for (int c = 0; c < n; c++) {
            if (es_valida(col, fila, c)) {
                int *col_local = copiar(col);
                col_local[fila] = c;
                #pragma omp task firstprivate(col_local)
                reinas_par(col_local, fila + 1, n, nivel_corte);
            }
        }
    } else {
        // secuencial cuando el árbol ya es pequeño
        for (int c = 0; c < n; c++) {
            if (es_valida(col, fila, c)) {
                col[fila] = c;
                reinas_par(col, fila + 1, n, nivel_corte);
            }
        }
    }
}

Decisiones de diseño: hasta qué profundidad paralelizar (umbral de corte), si copiar el estado (necesario en backtracking si las tareas son concurrentes), cómo combinar resultados.

Branch & Bound

Variante del backtracking para problemas de optimización. Mantiene una cota global (la mejor solución conocida hasta el momento) y poda las ramas cuyo límite superior es peor que la cota actual.

Reto en paralelo: la cota compartida

La cota debe ser visible para todos los procesos:

  • En memoria compartida: variable global con atomic o sección crítica para actualizar.
  • En memoria distribuida: hay que difundir las mejoras. Patrones típicos: anuncio asíncrono cuando se mejora la cota, sondeo periódico.

Riesgo: trabajo especulativo

Sin coordinación, los procesos pueden explorar ramas que serían podadas si conocieran la cota actualizada de otro proceso. Hay un equilibrio entre coste de comunicación y trabajo especulativo: comunicar cada mejora es caro; comunicar poco aumenta el trabajo inútil.

Trabajadores replicados: el patrón

Generalización: tienes una bolsa de tareas. Los procesos cogen tareas, las procesan y posiblemente generan nuevas tareas, que devuelven a la bolsa. Cuando la bolsa está vacía y todos los procesos están ociosos, se termina.

Patrón de bolsa de tareas: balanceo automático porque cada trabajador coge nueva tarea cuando ha terminado. Adecuado para cargas impredecibles.

Implementación maestro-esclavo

La forma más común. Un proceso (maestro) gestiona la bolsa; los demás (esclavos) le piden tareas y le devuelven resultados.

// Maestro (rank 0)
while (hay_tareas() || trabajadores_ocupados > 0) {
    MPI_Recv(&msg, ..., MPI_ANY_SOURCE, ...);
    int esclavo = msg.source;

    if (es_resultado(msg)) {
        procesar_resultado(msg);
        trabajadores_ocupados--;
    }

    if (hay_tareas()) {
        tarea_t t = sacar_tarea();
        MPI_Send(&t, ..., esclavo, ...);
        trabajadores_ocupados++;
    } else {
        MPI_Send(&fin, ..., esclavo, ...);
    }
}

// Esclavo (rank > 0)
while (1) {
    MPI_Send(&pet, ..., 0, ...);   // pedir trabajo
    MPI_Recv(&msg, ..., 0, ...);
    if (msg.tipo == FIN) break;
    resultado_t r = procesar(msg.tarea);
    MPI_Send(&r, ..., 0, ...);    // devolver resultado
}

Ventajas

  • Balanceo de carga automático: los esclavos rápidos procesan más tareas.
  • Tolerante a heterogeneidad: si los procesos están en máquinas distintas con velocidades distintas, no importa.
  • Sencillo de implementar y depurar.

Desventajas

  • El maestro es un cuello de botella: con muchos esclavos, no le da tiempo a atender a todos.
  • El maestro está ocioso casi todo el tiempo (solo gestiona, no calcula): malgasta un proceso.
  • Si la red es lenta, las peticiones penalizan al esclavo.

Esquemas descentralizados

Para escalar más, se eliminan los cuellos de botella distribuyendo la bolsa.

Bolsa distribuida

Cada proceso tiene su propia mini-bolsa. Cuando se le acaba, pide trabajo a un vecino (work stealing). Es lo que hace internamente el runtime de tareas de OpenMP y librerías como Cilk o Intel TBB.

Algoritmos de detección de terminación

Sin maestro, decidir cuándo se ha acabado todo es un problema en sí mismo: un proceso puede creer que ha terminado, pero otro a punto de generar nuevas tareas para él. Algoritmos clásicos:

  • Dijkstra-Scholten: token que circula. Si vuelve sin actividad, se acabó.
  • Conteo distribuido: cada proceso cuenta tareas creadas y completadas; cuando la suma global es 0, fin.
Estos algoritmos son sutiles. Implementarlos correctamente es un buen ejercicio, pero en la práctica conviene usar librerías que ya los traen (la propia OpenMP, MPI's distributed graph topology, frameworks como Charm++).

Práctica de recuperación

Esta sesión cierra los esquemas algorítmicos. Las preguntas mezclan ideas de toda esta parte del curso.

¿Qué es el balanceo dinámico de carga?
Esquema en el que los procesos piden trabajo según lo van terminando, en lugar de recibir un reparto fijo al principio. Imprescindible cuando los costes de las tareas son impredecibles o se generan tareas durante la ejecución.
¿Cuál es la diferencia entre backtracking y branch & bound?
Ambos exploran árboles. Backtracking busca soluciones (válidas o todas las soluciones). Branch & bound resuelve problemas de optimización: mantiene una cota de la mejor solución actual y poda ramas que no pueden mejorarla.
¿Por qué la cota global en branch & bound paralelo es problemática?
Tiene que ser visible para todos los procesos. Si no se difunde, los procesos exploran ramas que ya podrían podarse (trabajo especulativo). Si se difunde con demasiada frecuencia, se penalizan las comunicaciones. Hay un equilibrio.
Describe el patrón maestro-esclavo en cuatro líneas.
Un proceso (maestro) mantiene una bolsa de tareas. Los demás (esclavos) le piden trabajo, lo procesan y le devuelven resultados. El maestro asigna nueva tarea con cada resultado. Termina cuando bolsa vacía y nadie procesa.
¿Cuáles son las dos limitaciones principales del maestro-esclavo?
(1) El maestro es cuello de botella con muchos esclavos (no puede atender a todos). (2) Un proceso entero se dedica solo a gestionar, no calcula: malgasta hardware. Se mitigan con bolsas distribuidas y work stealing.
¿Qué es el work stealing?
Técnica de balanceo distribuido: cada proceso tiene su propia mini-bolsa de tareas. Cuando se queda sin trabajo, «roba» tareas de la bolsa de un vecino. Es lo que hace internamente OpenMP con sus tareas, Cilk, Intel TBB.
¿Por qué la detección de terminación en sistemas descentralizados es no trivial?
Sin maestro central, un proceso puede creer que terminó pero otro está a punto de generar tareas para él. Hace falta un algoritmo distribuido (Dijkstra-Scholten, conteo distribuido) que garantice que la suma global de tareas pendientes es cero.
¿Por qué N reinas es un problema natural para trabajadores replicados?
El árbol de búsqueda crece de forma muy desigual: algunas ramas se podan enseguida, otras llegan hasta la profundidad N. Repartir estáticamente las ramas iniciales daría un desequilibrio enorme. El balanceo dinámico iguala los tiempos.

Quiz final entreverado del curso

Ocho preguntas que mezclan deliberadamente todas las sesiones. Discriminar entre conceptos parecidos —MPI vs OpenMP, datos vs tareas, síncrono vs relajado— es la habilidad clave que el examen evalúa. Responde primero, comprueba después.

PREGUNTA 1
En un cluster de 50 nodos, cada uno con 8 cores y 1 GPU, ¿qué combinación de paradigmas aprovecha todo el hardware?
  • Solo MPI con 400 procesos
  • MPI entre nodos + OpenMP entre cores + CUDA en la GPU
  • Solo OpenMP con 400 hilos
  • Solo CUDA
El triple híbrido: un proceso MPI por nodo, OpenMP entre los 8 cores del nodo, CUDA para las partes regulares en la GPU. Cada paradigma ataca un nivel del hardware.
PREGUNTA 2
Tienes una suma de 10⁹ datos. El tiempo paralelo aproximado es (n/p)·tc + log p · (ts+tw). ¿Qué término domina con p = 8?
  • (n/p)·tc, porque n es enorme y log 8 = 3
  • log p · (ts+tw), siempre domina la comunicación
  • Los dos por igual
  • No se puede saber sin medir
Con n = 10⁹ y p = 8, el cómputo n/p = 1.25·10⁸ es muy superior a log p · (ts+tw) salvo en redes patológicas. El paralelismo será casi ideal.
PREGUNTA 3
¿Qué directiva OpenMP usarías para evitar la condición de carrera al actualizar un contador único desde varios hilos?
  • #pragma omp parallel for
  • #pragma omp atomic antes de la actualización
  • #pragma omp single
  • #pragma omp barrier
atomic es la opción más ligera para una operación simple como un incremento. critical también funcionaría pero es más costoso. single haría que solo un hilo incrementara —no es lo que queremos—.
PREGUNTA 4
En MPI, ¿qué función envía un mismo valor desde un proceso a todos los demás del comunicador?
  • MPI_Scatter
  • MPI_Bcast
  • MPI_Allreduce
  • MPI_Send en bucle
MPI_Bcast es broadcast: uno a todos con el mismo dato. MPI_Scatter reparte trozos distintos a cada uno. MPI_Send en bucle funciona pero es muy ineficiente: el broadcast usa árboles para difundir en log p.
PREGUNTA 5
Ley de Amdahl: si tu programa tiene un 10% inherentemente secuencial, ¿cuál es el speed-up máximo posible con infinitos procesadores?
  • 100
  • 10
  • Infinito
  • 90
El speed-up está acotado por 1/f. Con f = 0.10, el límite es 1/0.10 = 10. Aunque tuvieras 1000 procesadores, la parte secuencial impediría superar speed-up 10.
PREGUNTA 6
Quieres resolver el problema de las N reinas para N = 14. El árbol de búsqueda crece de forma desigual y se podan muchas ramas. ¿Qué esquema algorítmico es más apropiado?
  • Paralelismo de datos puro
  • Pipeline
  • Trabajadores replicados con bolsa de tareas
  • Programación dinámica
El recorrido del árbol es impredecible: el balanceo estático fallaría (unas ramas grandes, otras minúsculas). La bolsa de tareas permite que los procesos cojan trabajo según terminan, igualando los tiempos.
PREGUNTA 7
En una multiplicación de matrices distribuida, A se reparte por filas y B se replica en todos los procesos. ¿Qué operación colectiva MPI usas para repartir A?
  • MPI_Bcast
  • MPI_Scatter
  • MPI_Reduce
  • MPI_Allgather
MPI_Scatter reparte trozos distintos del root a cada proceso (cada uno recibe sus filas). Para B, que se replica, sí se usa MPI_Bcast. Para recoger C al final, MPI_Gather.
PREGUNTA 8
¿Cuál es la diferencia clave entre divide y vencerás y programación dinámica?
  • DV es recursivo y PD es iterativo
  • DV usa memoización y PD no
  • En DV los subproblemas son independientes; en PD se solapan y se reutilizan
  • DV es para minimización y PD para maximización
DV: subproblemas independientes, cada uno aparece una vez, ideal para paralelizar. PD: subproblemas solapados, se calculan una vez y se reutilizan vía tabla. Esa reutilización es lo que da eficiencia a la PD pero complica su paralelización (hay dependencias).

Cómo seguir consolidando lo aprendido

Has llegado al final de las doce sesiones. Pero el aprendizaje no acaba aquí; en realidad, el efecto de espaciado dice que apenas empieza:

  • Espacia el repaso. Vuelve a estas páginas en 2-3 días, luego en una semana, luego en tres semanas. Cada vez que recuerdes lo que olvidaste, la memoria se consolida.
  • Practica con las prácticas. Aplicar los conceptos en código es la mejor consolidación: implementa un algoritmo nuevo cada semana usando un esquema distinto.
  • Mezcla los temas al estudiar. No repases una sesión entera; mezcla preguntas de varias. La interferencia entrena la discriminación.
  • Genera tus propios ejemplos. Cada vez que leas un esquema, inventa un problema concreto que encaje en él. La elaboración es la técnica con más respaldo empírico para aprendizaje profundo.
  • Explícaselo a alguien. Lo que se enseña se aprende mejor. Si no tienes a quien, escríbelo como si lo explicaras.
Los estudios sobre técnicas de aprendizaje son consistentes en algo: lo que parece más cómodo —releer, marcar en amarillo, copiar apuntes— es lo menos efectivo. Lo incómodo —recordar de memoria, equivocarse, esperar a olvidar antes de repasar— es lo que funciona.
