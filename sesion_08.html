Paralelismo
de datos

El esquema más sencillo y, en muchos casos, el más eficiente: aplicar la misma operación a muchos datos a la vez. Es el caballo de batalla del cómputo numérico.

Idea central

«Muchos datos tratados de una forma igual o similar». No hay tareas distintas que coordinar: todos los procesos hacen el mismo trabajo sobre porciones distintas del problema.

Es el modelo natural de:

  • Algoritmos numéricos sobre arrays y matrices.
  • Procesado de imagen píxel a píxel.
  • Simulaciones de partículas independientes.
  • Cualquier «mapa» de una función sobre una colección.

También es el modelo que mejor se adapta a las GPU y a las unidades vectoriales SIMD.

Dos variantes según el modelo de memoria

Memoria compartida — paralelismo de datos

Los hilos comparten el array. Solo hay que decidir qué porción procesa cada uno. Paralelización a menudo implícita: el compilador o el runtime reparten las iteraciones (#pragma omp parallel for).

Memoria distribuida — particionado de datos

Cada proceso tiene su trozo en memoria propia. La distribución del array es explícita: el programador decide qué dato vive en qué proceso, y comunica los datos compartidos cuando hace falta.

La distinción «paralelismo de datos» vs «particionado de datos» es propia del curso, refleja el mismo principio en dos modelos de memoria. Otros textos usan «paralelismo de datos» de forma más general.

Ejemplo canónico: suma de n datos

Calcular la suma de un array. Algo trivial en secuencial, base de muchos algoritmos paralelos.

Versión implícita (memoria compartida)

double suma = 0.0;
#pragma omp parallel for reduction(+: suma)
for (int i = 0; i < n; i++)
    suma += a[i];

OpenMP reparte el bucle entre hilos, cada uno acumula en su copia privada de suma, y al final combina con +. El programador no especifica cómo se reparte ni cómo se combina: lo decide el runtime.

Versión explícita (paso de mensajes)

// Cada proceso tiene su trozo a_local de tamaño n/p
double suma_local = 0.0;
for (int i = 0; i < n_local; i++)
    suma_local += a_local[i];

double suma_total;
MPI_Reduce(&suma_local, &suma_total, 1, MPI_DOUBLE,
           MPI_SUM, 0, MPI_COMM_WORLD);

El reparto es responsabilidad del programador (con MPI_Scatter o asignación inicial). La reducción se hace explícitamente con MPI_Reduce.

Análisis del tiempo

Para cada uno de los p procesos: cómputo = (n/p) · tc. La reducción en árbol cuesta log p · (ts + tw). Total:

MODELO DEL TIEMPO

t(n, p) ≈ (n/p)·tc + log p · (ts + tw)

El primer término es decreciente en p (buen escalado del cómputo). El segundo crece como log p. Mientras el cómputo domine, el speed-up es casi ideal. Para n pequeño, la reducción manda y paralelizar puede no compensar.

Producto escalar y producto matriz-vector

Producto escalar

Estructura prácticamente idéntica a la suma:

// dot(a, b) = a[0]*b[0] + a[1]*b[1] + ... + a[n-1]*b[n-1]
double dot = 0.0;
#pragma omp parallel for reduction(+: dot)
for (int i = 0; i < n; i++)
    dot += a[i] * b[i];

Producto matriz-vector

Dos bucles. Se paraleliza el externo, sobre las filas:

// r = A · x (A es n×n, x es n×1)
#pragma omp parallel for
for (int i = 0; i < n; i++) {
    double s = 0;
    for (int j = 0; j < n; j++)
        s += A[i][j] * x[j];
    r[i] = s;
}

Cada iteración del bucle externo es independiente, así que no hace falta ni reduction ni critical. Cada hilo escribe en una posición distinta de r.

Multiplicación de matrices

El problema modelo del bloque. Aparece en todos los ejemplos del curso.

Versión implícita básica

#pragma omp parallel for private(i, j, k)
for (i = 0; i < n; i++)
    for (j = 0; j < n; j++) {
        c[i][j] = 0;
        for (k = 0; k < n; k++)
            c[i][j] += a[i][k] * b[k][j];
    }

El número de operaciones en coma flotante es 2n³. Pero el tiempo se puede mejorar mucho sin cambiar el orden algorítmico, jugando con el acceso a memoria:

  • Usar una variable escalar s para acumular y escribir c[i][j] una sola vez al final, evitando recargas.
  • Reordenar bucles para mejorar localidad (versión ikj en lugar de ijk).
  • Multiplicar por bloques (tiling): aprovecha la caché.
  • Vectorizar con AVX o usar BLAS (dgemm).

Versión por bloques

Particiona A, B, C en bloques de tamaño m × m. Cada elemento del resultado por bloques es:

DESCOMPOSICIÓN POR BLOQUES

Cij = Σk Aik · Bkj

Mismo patrón que la multiplicación elemental pero con bloques. Ventajas: (1) cada bloque cabe en caché y se reusa intensamente, (2) las multiplicaciones de bloques son tareas grandes, fáciles de mapear a procesos.

Distribución de datos en memoria distribuida

Cuando los datos no caben en una sola máquina o queremos paralelizar entre nodos, hay que distribuir los arrays entre procesos. Hay varias formas estándar.

Para vectores y matrices unidimensionales

Distribución por bloques vs cíclica. Existen también híbridas («bloque-cíclica») que combinan ambas ventajas.

Para matrices bidimensionales

Las opciones son más ricas:

  • Por filas: cada proceso tiene un bloque de filas consecutivas. Cómodo para productos matriz-vector con vector replicado.
  • Por columnas: análogo.
  • Por bloques 2D: malla de procesos P = Pr × Pc; cada proceso tiene un subbloque rectangular. Mejor para multiplicación de matrices.

Fórmulas para distribución por bloques 2D

Si tenemos matriz A de m × n y malla de procesos P = pr × pc:

  • Tamaño de bloque por fila: br = ⌈m / pr⌉.
  • Tamaño de bloque por columna: bc = ⌈n / pc⌉.
  • El elemento global ai,j va al proceso (i div br, j div bc).
  • Sus índices locales son (i mod br, j mod bc).

Ejercicio guiado: aplicar las fórmulas

Matriz A de 14 × 11, malla de procesos P = 3 × 2. Calcular destino e índices locales para tres elementos.

Tamaño de bloque por fila: br = ⌈14 / 3⌉ = 5.
Tamaño de bloque por columna: bc = ⌈11 / 2⌉ = 6.

Elemento globalProceso destino (fila, col)Índices locales

Estos cálculos son ejercicios típicos de examen. Practica con tu propio ejemplo: dibuja la malla, marca los bordes de los bloques, y comprueba que las fórmulas dan lo que ves.

Cuándo el paralelismo de datos no encaja

No todos los problemas son adecuados. Síntomas de incompatibilidad:

  • Iteraciones con coste muy desigual: el reparto estático produce desequilibrio. Necesitas balanceo dinámico (sesión 12).
  • Dependencias entre iteraciones: si a[i] depende de a[i-1], no se puede paralelizar directamente. Hay que reformular (pipeline o algoritmos relajados).
  • Naturaleza estructural irregular: árboles, grafos arbitrarios. Mejor con esquemas de tareas o trabajadores replicados.

Práctica de recuperación

Esta sesión sienta el modelo base que las siguientes contrastarán. Comprueba que distingues el paralelismo de datos de los otros esquemas que vendrán.

Define paralelismo de datos en una sola frase.
Muchos datos tratados de la misma forma —o muy similar— por procesos que ejecutan el mismo código sobre porciones distintas del problema.
¿En qué se diferencia «paralelismo de datos» (memoria compartida) de «particionado de datos» (memoria distribuida)?
En memoria compartida los hilos comparten el array y el reparto suele ser implícito (lo hace OpenMP). En memoria distribuida cada proceso tiene su trozo en memoria propia y el reparto es explícito (el programador hace MPI_Scatter o equivalente).
¿Por qué para una matriz m × n en malla pr × pc el tamaño de bloque por fila es ⌈m/pr⌉?
Para garantizar que todos los procesos de una fila de la malla reciben filas de la matriz y que ninguno se queda con más de un bloque. El último proceso puede quedarse con menos si m no es múltiplo de pr.
¿Qué ventaja aporta la multiplicación de matrices por bloques sobre la versión clásica?
Mejor uso de caché: cada bloque cabe en caché L2/L3 y se reutiliza muchas veces antes de salir. Además, los bloques se mapean naturalmente a procesos en memoria distribuida. La complejidad asintótica es la misma (O(n³)) pero el tiempo real puede ser un orden de magnitud menor.
Diferencia entre distribución por bloques y cíclica.
Bloques: cada proceso recibe un trozo contiguo del array. Bueno para acceso secuencial y patrones de vecinos. Cíclica: los elementos se reparten alternados entre procesos. Bueno para balancear costes que dependen del índice (típico en algoritmos triangulares).
Para una suma de n datos con p procesos, el tiempo aproximado es (n/p)·tc + log p·(ts+tw). ¿Cuándo NO merece la pena paralelizar?
Cuando n es tan pequeño que (n/p)·tc < log p·(ts+tw): el coste de comunicar/coordinar supera al cómputo ahorrado. Hay un n umbral por debajo del cual el secuencial gana.
¿Sería útil paralelizar la suma total final de los productos parciales en un producto escalar distribuido?
No: solo son p valores. La suma cuesta p − 1 sumas, mientras que paralelizarla añade comunicaciones que cuestan log p · (ts+tw). Para los típicos valores de p, p − 1 sumas son baratísimas. Se hace secuencialmente en el root (o se usa MPI_Reduce, que ya optimiza internamente).
Repaso de la sesión 7: ¿qué fase de PCAM toma la decisión «cada proceso recibe n/p filas»?
Aglomeración. El particionado original genera una tarea por elemento (o por fila); la aglomeración las agrupa en bloques del tamaño que conviene al hardware.
¿Por qué el paralelismo de datos es el modelo natural de las GPU?
Porque las GPU son SIMD masivo: miles de hilos ejecutan la misma instrucción sobre datos distintos. Cuando tu problema encaja en ese molde —mapear una función sobre un array—, la GPU brilla. Para problemas irregulares, no.
