3 · Paralelismo síncrono vs asíncrono (relajado)

  • Síncrono: todos los procesos avanzan al mismo ritmo, con sincronización tras cada paso (barreras). Modelo natural en SIMD y en problemas de stencil: ecuación del calor, Jacobi iterativo, simulaciones físicas con dependencia de datos del paso anterior. Garantiza determinismo y resultados reproducibles, pero el más lento marca el ritmo del conjunto.
  • Relajado / asíncrono: no hay sincronización global; cada proceso usa los datos más recientes que tenga, aunque no sean los del último paso. Útil en optimización (Gauss-Seidel asíncrono, algoritmos heurísticos) y en computación irregular. Sacrifica reproducibilidad por velocidad.

4 · Secciones críticas y exclusión mutua

Una sección crítica es una región de código que solo puede ejecutar un hilo/proceso a la vez. Necesaria cuando varios elementos de proceso acceden a la misma variable compartida y al menos uno escribe (condición de carrera).

Mecanismos: #pragma omp critical (OpenMP), #pragma omp atomic (para operaciones simples y mucho más rápida), llaves de librería (omp_set_lock), MPI_Win_lock en memoria compartida MPI. La cláusula reduction evita la sección crítica explícita para acumulaciones: cada hilo opera sobre una copia y al final se combinan.

6 · Metodología de Foster (PCAM)

Cuatro fases para diseñar un algoritmo paralelo, en orden:

  1. Particionado: descomponer el problema en muchas tareas pequeñas (de datos o funcionales), sin pensar en arquitectura.
  2. Comunicación: identificar qué tareas dependen de qué otras y cómo se intercambian datos.
  3. Aglomeración: agrupar tareas pequeñas en bloques mayores para reducir overhead, manteniendo el paralelismo suficiente.
  4. Mapeo: asignar cada aglomerado a un procesador físico, buscando balance de carga.

¿Qué imprime / ejecuta este código con OpenMP?

double array[100];
int i;
omp_set_num_threads(X);
#pragma omp parallel private(i)
for (i = 0; i < 100; i++)
    array[i] = sin(i);
  • A. Con X=4, el trabajo acumulado es 100 iteraciones.
  • B. Con X=10, el programa fallará en un procesador con 4 cores.
  • C. Con 5 hilos, cada uno ejecuta 20 iteraciones.
  • D. Todas las anteriores son falsas. ✓
Trampa clave: no hay #pragma omp for, solo parallel. Cada hilo ejecuta el bucle entero de 100 iteraciones.
  • A · falsa: con 4 hilos el trabajo total es 4 × 100 = 400 iteraciones (ineficiente, además todos escriben en array[i] simultáneamente — race condition).
  • B · falsa: lanzar 10 hilos en 4 cores no falla; el SO los planifica time-sharing. Solo será más lento o equivalente.
  • C · falsa: precisamente porque no hay for, cada hilo hace las 100 iteraciones, no 20. Si tuviera parallel for sí sería cierto.
Recuerda: parallel crea el equipo, for reparte. Olvidar el for es uno de los errores OpenMP más típicos.

Bloque III · Preguntas cortas resueltas

1 · ¿Para qué y cuándo son necesarias las secciones críticas? Ejemplo de uso.

Una sección crítica es una región de código que solo puede ejecutar un hilo o proceso a la vez. Son necesarias cuando varios elementos de proceso comparten una variable y al menos uno la modifica: sin protección habría condiciones de carrera (resultados no deterministas, dependientes del orden temporal de los accesos).

Ejemplo en OpenMP: contar elementos que cumplen un predicado.

int contador = 0;
#pragma omp parallel for
for (int i = 0; i < n; i++) {
    if (cumple(v[i])) {
        #pragma omp critical
        contador++;
    }
}

Para esta operación tan simple, #pragma omp atomic sería más rápido. Y aún mejor sería reduction(+: contador), que evita por completo la sincronización: cada hilo cuenta en su copia local y se combinan al final.

2 · ¿Qué diferencia hay entre un sistema de paso de mensajes y uno de memoria compartida?

  • Memoria compartida: los procesos/hilos comparten un mismo espacio de direcciones. Se comunican leyendo y escribiendo variables comunes (comunicación implícita). Necesitan primitivas de sincronización (secciones críticas, barreras, locks) para evitar condiciones de carrera. Limitado a la memoria física de un nodo. Modelo: OpenMP, Pthreads.
  • Paso de mensajes: cada proceso tiene su propia memoria local privada. La comunicación es explícita mediante operaciones send / recv. No hay condiciones de carrera (cada uno gestiona su memoria), pero hay coste de comunicación y posibles deadlocks. Escala a clusters con miles de nodos. Modelo: MPI.

La programación híbrida (sesión 5) combina ambos: procesos MPI entre nodos, hilos OpenMP dentro de cada nodo.

3 · Explica el paralelismo síncrono: ideas principales y ejemplo.

Paralelismo síncrono: todos los elementos de proceso avanzan al mismo ritmo, sincronizándose en puntos predefinidos (barreras). Cada paso (o iteración) usa los resultados del paso anterior y solo del anterior. El más lento marca el ritmo.

Ideas clave:

  • Barreras explícitas o implícitas tras cada fase.
  • Determinismo: el resultado no depende del orden temporal de ejecución, solo del orden lógico.
  • Suele requerir doble buffer: una matriz con los valores del paso k y otra con los del paso k+1, para evitar leer datos parcialmente actualizados.

Ejemplo canónico: ecuación del calor en 2D por diferencias finitas (Jacobi). Cada punto interior se actualiza al promedio de sus cuatro vecinos del paso anterior — discretización de ∂u/∂t = ∇²u.

for (paso = 0; paso < T; paso++) {
    #pragma omp parallel for collapse(2)
    for (int i = 1; i < N-1; i++)         // fila (sin bordes)
        for (int j = 1; j < N-1; j++)     // columna (sin bordes)
            nuevo[i][j] = 0.25 * (viejo[i-1][j] + viejo[i+1][j]
                              + viejo[i][j-1] + viejo[i][j+1]);

    // Barrera implícita del parallel for: todos terminaron el paso
    intercambiar(&viejo, &nuevo);                  // swap de punteros, O(1)
}

Variables del código

VariableAlcanceSignificado
pasosharedIteración temporal (0..T−1). El bucle exterior NO se paraleliza: cada paso depende del anterior.
Tshared (constante)Número de pasos temporales a simular.
Nshared (constante)Dimensión de la malla (N × N).
i, jprivate automáticaÍndices de fila y columna. Bajo collapse(2), OpenMP «aplana» los dos bucles en uno y los reparte juntos, lo que da más paralelismo cuando N es pequeño.
viejo, nuevoshared (lectura/escritura disjuntas)Doble buffer: viejo guarda el estado del paso anterior (solo lectura); nuevo se rellena con el nuevo estado (escritura, sin solape con la lectura). Tras cada paso, se intercambian los punteros y nuevo pasa a ser el nuevo viejo.

Por qué este patrón funciona

  1. Sin doble buffer habría carrera: si actualizásemos u[i][j] in-place, un hilo rápido escribiría valores que otro hilo lento todavía está leyendo como «del paso anterior».
  2. El bucle temporal no se paraleliza: hay dependencia entre pasos consecutivos (el paso k usa los valores del k−1).
  3. Sí se paraleliza el bucle espacial: dentro de un paso, todos los puntos pueden calcularse en paralelo porque solo leen de viejo (sin condiciones de carrera).
  4. La barrera al final del parallel for garantiza que todos los hilos terminan el paso antes de que intercambiar reasigne los punteros.

El contrapunto es el paralelismo relajado (Gauss-Seidel asíncrono): cada hilo usa los valores más recientes que encuentre, sin esperar y sin doble buffer. Converge en menos iteraciones porque cada actualización propaga información antes; a cambio se pierde determinismo (el resultado de cada paso depende del orden de ejecución).

Bloque IV · Ejercicios prácticos resueltos

Ejercicio 1 · OpenMP — paralelizar bucle 2D y analizar coste

Enunciado: escalar cada elemento de una matriz a[N][M] por 0.5, en (1) paralelismo implícito sobre array unidimensional, (2) paralelismo explícito. Indicar el coste con p hilos.

Proceso de construcción (lo que vas pensando)

  1. ¿Las iteraciones son independientes? Sí: a[i][j] *= 0.5 solo toca la posición (i, j), no lee ni escribe ninguna otra. No hay dependencias entre iteraciones. → Paralelismo de datos puro.
  2. ¿Qué bucle paralelizo? El exterior (más iteraciones gruesas) o ambos combinados con collapse(2). Pero el enunciado pide aplanar a 1D primero, lo que nos da un único bucle de N·M iteraciones.
  3. ¿Por qué pasar a 1D? Una double **a es un array de punteros a filas; las filas son contiguas cada una, pero entre filas hay un salto arbitrario. Aplanando a double *a de tamaño N*M, la matriz queda en un único bloque de memoria. Esto facilita: vectorización del compilador, transmisión MPI con un único Send, mejor uso de la caché por prefetching.
  4. ¿Hace falta protección? No: cada hilo escribe en posiciones distintas (idx es la variable de iteración, automáticamente privada en parallel for).
  5. ¿Qué schedule? static: las iteraciones cuestan todas lo mismo (una multiplicación), reparto fijo sin overhead de gestión.

(1) Paralelismo implícito sobre array 1D:

double *a = malloc(N * M * sizeof(double));   // almacenamiento contiguo
// ... inicialización: a[i*M + j] equivale a a[i][j] ...

#pragma omp parallel for schedule(static)
for (int idx = 0; idx < N * M; idx++)
    a[idx] *= 0.5;

Coste con p hilos

La operación a[idx] *= 0.5 es O(1) por elemento; hay N·M elementos. Sin comunicación entre hilos:

(2) Paralelismo explícito: el programador calcula los rangos a mano. Equivalente lógico al schedule(static) anterior.

#pragma omp parallel
{
    int tid    = omp_get_thread_num();                // quién soy (0..nth-1)
    int nth    = omp_get_num_threads();               // cuántos somos
    int total  = N * M;                                // trabajo total
    int chunk  = (total + nth - 1) / nth;             // tamaño de bloque (techo)
    int ini    = tid * chunk;                          // mi posición de inicio
    int fin    = (ini + chunk < total) ? ini + chunk : total;   // fin sin desbordar

    for (int idx = ini; idx < fin; idx++)
        a[idx] *= 0.5;
}

Variables del código explícito

chunkint · privateTamaño nominal del bloque por hilo. El + nth − 1 antes de dividir es el truco para redondear hacia arriba y que el último hilo coja también el resto cuando total no es múltiplo de nth.

Las variables declaradas dentro de la región parallel son privadas automáticamente: cada hilo crea su propia copia en su pila. No hace falta declarar private(tid, nth, …).

La diferencia con (1) es pedagógica: en explícito ves el reparto; en implícito confías en el runtime de OpenMP. En la práctica, prefiere implícito (más legible y menos propenso a errores aritméticos en el cálculo de los límites).

Ejercicio 2 · MPI — cálculo de π en paralelo

Enunciado: paralelizar con MPI el cálculo de π por la regla del rectángulo de n subintervalos: π ≈ h · Σi=1..n f(xi) con h = 1/n, xi = h · (i − 0.5), f(x) = 4 / (1 + x²).

Proceso de construcción (lo que vas pensando)

  1. ¿Qué se paraleliza? El bucle de la suma (iteraciones independientes: cada xi se evalúa por separado y se acumula). Es paralelismo de datos con reducción.
  2. ¿Cómo reparto las iteraciones entre procesos? Dos opciones equivalentes:
    • Por bloques: el proceso k coge [k·n/nproc, (k+1)·n/nproc). Sencillo.
    • Cíclico: el proceso k coge i = k+1, k+1+nproc, k+1+2·nproc…. Balanceo perfecto cuando n no es múltiplo de nproc.
    Aquí usamos cíclico por elegancia.
  3. ¿Qué datos comparten todos? El número total de subintervalos n. Lo lee el proceso 0 y se reparte con MPI_Bcast. Como aquí es una constante del programa, el Bcast es opcional; lo incluimos para que el patrón sea general.
  4. ¿Cómo se combinan los resultados parciales? Cada proceso calcula su suma local; al final se suman todas con MPI_Reduce usando operador MPI_SUM hacia el root.
  5. ¿Quién imprime? Solo el root: tras la reducción es el único que tiene el resultado completo.
  6. Esqueleto resultante: MPI_Init → Comm_rank/Comm_size → Bcast(n) → bucle local → Reduce → root imprime → MPI_Finalize. Este es el patrón canónico de reducción global.
#include <mpi.h>
#include <stdio.h>
#include <math.h>

double f(double a) { return 4.0 / (1.0 + a * a); }     // integrando

int main(int argc, char *argv[]) {
    int rank, nproc, n = 100, i;
    double h, local_sum = 0.0, total_sum, pi, x;
    const double PI25DT = 3.141592653589793;     // π con 15 cifras (referencia)

    MPI_Init(&argc, &argv);                          // 1. arranca MPI
    MPI_Comm_rank(MPI_COMM_WORLD, &rank);            // 2. ¿quién soy? (0..nproc-1)
    MPI_Comm_size(MPI_COMM_WORLD, &nproc);           // 3. ¿cuántos somos?

    MPI_Bcast(&n, 1, MPI_INT, 0, MPI_COMM_WORLD);     // 4. todos conocen n
    h = 1.0 / (double) n;                            // 5. ancho de cada rectángulo

    // 6. reparto cíclico: proceso k toma i = k+1, k+1+nproc, ...
    for (i = rank + 1; i <= n; i += nproc) {
        x = h * ((double) i - 0.5);                  // centro del rectángulo i
        local_sum += f(x);                              // aporte de este rectángulo
    }

    // 7. suma todas las local_sum en el proceso 0
    MPI_Reduce(&local_sum, &total_sum, 1, MPI_DOUBLE,
               MPI_SUM, 0, MPI_COMM_WORLD);

    if (rank == 0) {                                  // 8. solo el root tiene total_sum
        pi = h * total_sum;
        printf("pi ≈ %.16f, error = %.16f\n",
               pi, fabs(pi - PI25DT));
    }

    MPI_Finalize();                                  // 9. cierra MPI
    return 0;
}

Comentarios para argumentar en el examen

  • Reparto cíclico (i = rank+1, +=nproc) en lugar de por bloques contiguos: balanceo perfecto cuando n no es múltiplo de nproc. Si fuera por bloques se usaría local_n = n / nproc y trozos contiguos [rank·local_n, (rank+1)·local_n).
  • MPI_Reduce en lugar de un bucle de Recv + suma: usa árbol logarítmico, mucho más eficiente y conciso (O(log p) vs O(p)).
  • Solo el root imprime: el resultado total solo lo tiene el proceso 0 tras MPI_Reduce. Si todos lo necesitaran (por ejemplo, para una segunda fase de cómputo), sería MPI_Allreduce.
  • Coste de comunicaciones: 1 Bcast de un entero + 1 Reduce de un double. Ambas O(log p). Despreciable comparado con el cómputo si n es grande.
  • Variables locales vs globales: ojo con confundir local_sum y total_sum. La primera la calcula cada proceso por separado; la segunda solo es válida en el root tras la reducción.

Ejercicio 3 · Pipeline de 3 etapas (paso de mensajes)

Enunciado: pseudocódigo con paso de mensajes para un pipeline que procesa a[n]: etapa 1 filtra impares, etapa 2 multiplica por 2, etapa 3 imprime.

Proceso de construcción (lo que vas pensando)

  1. ¿Cuántos procesos? Uno por etapa. Aquí 3: P0 filtra, P1 transforma, P2 imprime.
  2. ¿Quién tiene los datos al inicio? El array a está en P0 (la primera etapa siempre es la que recibe la entrada).
  3. ¿Cómo fluyen los datos? En cadena: P0 → P1 → P2. Cada etapa recibe del anterior, hace su trabajo y envía al siguiente. La última etapa no envía a nadie.
  4. ¿Cómo se sabe que el flujo se acabó? Con un centinela: un valor especial que nunca aparece en datos reales (aquí FIN = −1) que marca el final. Cada etapa lo propaga a la siguiente al recibirlo.
  5. ¿Qué tipo de envío? Bloqueante (MPI_Send/MPI_Recv) por claridad. En producción se usa no bloqueante (Isend/Irecv) para que cada etapa empiece a procesar el siguiente dato mientras envía el actual.
  6. ¿Cómo se logra el paralelismo? En régimen permanente (cuando el pipeline está lleno) las 3 etapas trabajan a la vez sobre datos distintos del flujo. Aceleración teórica = número de etapas.

Tres procesos, dos canales de comunicación. Mientras P1 procesa el dato k, P0 ya puede mandar el k+1 y P2 imprimir el k−1. Si las tres etapas son del mismo coste, hay paralelismo real ≈ 3.

// Constante centinela para indicar fin de flujo
#define FIN -1

// ============ PROCESO 0 (etapa 1: filtrar impares) ============
if (rank == 0) {
    for (int i = 0; i < n; i++)
        if (a[i] % 2 != 0)
            MPI_Send(&a[i], 1, MPI_INT, 1, TAG, MPI_COMM_WORLD);

    int fin = FIN;                                // señaliza fin
    MPI_Send(&fin, 1, MPI_INT, 1, TAG, MPI_COMM_WORLD);
}

// ============ PROCESO 1 (etapa 2: multiplicar por 2) ============
if (rank == 1) {
    int x;
    while (1) {
        MPI_Recv(&x, 1, MPI_INT, 0, TAG,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
        if (x == FIN) {
            MPI_Send(&x, 1, MPI_INT, 2, TAG, MPI_COMM_WORLD);
            break;
        }
        x *= 2;
        MPI_Send(&x, 1, MPI_INT, 2, TAG, MPI_COMM_WORLD);
    }
}

// ============ PROCESO 2 (etapa 3: imprimir) ============
if (rank == 2) {
    int x;
    while (1) {
        MPI_Recv(&x, 1, MPI_INT, 1, TAG,
                 MPI_COMM_WORLD, MPI_STATUS_IGNORE);
        if (x == FIN) break;
        printf("%d\n", x);
    }
}

Razonamiento clave para el examen

  • El centinela FIN es el patrón estándar para terminar un pipeline: cuando el flujo se acaba, cada etapa propaga el FIN a la siguiente y termina. Sin centinela, las etapas posteriores quedarían eternamente en Recv (deadlock).
  • Las comunicaciones son punto a punto y bloqueantes: MPI_Send/MPI_Recv. En un pipeline real conviene no bloqueantes (MPI_Isend) para solapar comunicación con cómputo, pero el pseudocódigo se entiende mejor con bloqueantes.
  • Solapamiento: si el flujo es continuo, las 3 etapas trabajan a la vez. Aceleración ideal con r etapas: × r en régimen permanente. Las 2 primeras y 2 últimas iteraciones son fases transitorias (no todas las etapas activas).
  • Cuándo sirve pipeline: cuando las etapas pueden hacerse en orden, tienen costes similares y hay muchos elementos en el flujo. Si una etapa es mucho más lenta, será el cuello de botella (la replicarías).
  • Cuello de botella en el ejemplo: la etapa 3 (imprimir) suele ser la más lenta por I/O. En un caso real, podríamos mantener un buffer en P2 y volcarlo en lotes.

Bloque IV bis · Cómo se construye un programa paralelo

Esta sección sintetiza el modelo mental de cada tecnología: la cadena de decisiones que vas tomando al diseñar un programa. Si la interiorizas, los ejercicios del examen se resuelven casi solos.

OpenMP — modelo mental

Árbol de decisión para escoger directiva. Empieza por «¿es un for canónico?» y desciende según la naturaleza del paralelismo.

Decisiones que hay que tomar, en orden

  1. Identificar el paralelismo: ¿qué partes son independientes? Si no lo son, ¿se pueden reordenar las operaciones para que lo sean?
  2. Elegir la directiva: parallel for para bucles, sections para ramas fijas, task para recursión/listas, single para «solo uno hace algo» dentro de un paralelo.
  3. Clasificar las variables:
    • Solo lectura → shared por defecto, sin problema.
    • Cada hilo escribe en posiciones distintas → shared, sin problema.
    • Variables locales del bucle → declararlas dentro del parallel o usar private(...).
    • Acumuladores (suma, max, conteo) → reduction(+:var), reduction(max:var)…
    • Escritura compartida ineludible → critical o atomic.
  4. Elegir scheduling: static si las iteraciones son uniformes; dynamic/guided si son irregulares.
  5. Sincronizar: ¿hace falta una barrier manual? ¿Puedo ahorrar tiempo con nowait?

Ejemplo guiado: suma de un array

// Punto de partida secuencial
double suma = 0.0;
for (int i = 0; i < n; i++) suma += v[i];

Aplicamos la cadena de decisiones:

  1. «¿Qué paralelizo?» → el bucle: cada v[i] se lee una vez, suma parece compartido pero solo se acumula.
  2. «¿Bucle for canónico?» → sí. → parallel for.
  3. «¿Variables?» → v es shared (todos leen); n shared (constante); i private automática; suma es el problema — todos quieren escribir.
  4. «¿Suma sobre variable compartida?» → patrón clásico de acumulación. → reduction(+:suma).
double suma = 0.0;
#pragma omp parallel for reduction(+: suma)
for (int i = 0; i < n; i++) suma += v[i];

Si no conociéramos reduction, tendríamos que usar critical y el bucle se serializaría. Si fuera una operación complicada (no asociativa-conmutativa) no podríamos usar reduction y habría que repensar el algoritmo.

MPI — modelo mental

Decisiones que hay que tomar, en orden

  1. Esqueleto siempre igual: MPI_Init → Comm_rank → Comm_size → tu lógica → MPI_Finalize. Memorízalo: aparece en todos los programas.
  2. Decidir el rol de cada proceso: típicamente el 0 es maestro lógico (lee entrada, distribuye, recoge, imprime); los demás son workers. Esa asimetría se implementa con if (rank == 0) { … } else { … }.
  3. ¿Cómo reparto los datos iniciales?
    • Mismo dato a todos → MPI_Bcast.
    • Trozo distinto a cada uno (array) → MPI_Scatter.
    • Distribución manual cuando el reparto es desigual → Send/Recv en bucle.
  4. ¿Cómo reparto el cómputo entre procesos? Cada uno determina su trabajo a partir de rank y nproc:
    • Por bloques: local_n = n / nproc; ini = rank * local_n; fin = ini + local_n;
    • Cíclico: for (i = rank; i < n; i += nproc) ...
  5. ¿Cómo se combinan los resultados?
    • Una sola cantidad agregada (suma, max…) en root → MPI_Reduce.
    • Una sola cantidad agregada en todos → MPI_Allreduce.
    • Cada proceso aporta un trozo, root los junta → MPI_Gather.
    • Procesos vecinos intercambian (stencil) → MPI_Sendrecv.
  6. ¿Quién imprime? Solo el root. El resto de procesos no deberían escribir a stdout salvo para depurar.
  7. Cuidado con deadlocks: dos Ssend mutuos se cuelgan. Solución: Sendrecv o no bloqueantes.

Las cinco fases típicas. Casi todos los programas MPI del temario encajan en este esqueleto, cambiando qué se distribuye, cómo se calcula y cómo se combina.

Ejemplo guiado: producto escalar paralelo

Queremos calcular Σ a[i] · b[i] con n procesos. Aplicamos la cadena:

  1. «¿Cómo reparto?» → cada proceso recibe local_n = n/nproc elementos consecutivos de a y b.
  2. «¿Cada proceso qué calcula?» → su producto escalar local local_dot = Σ a_local[i]*b_local[i].
  3. «¿Cómo se combinan?» → reducción suma al root con MPI_Reduce(MPI_SUM).
int local_n = n / nproc;
double *a_local = malloc(local_n * sizeof(double));
double *b_local = malloc(local_n * sizeof(double));

MPI_Scatter(a, local_n, MPI_DOUBLE, a_local, local_n, MPI_DOUBLE, 0, MPI_COMM_WORLD);
MPI_Scatter(b, local_n, MPI_DOUBLE, b_local, local_n, MPI_DOUBLE, 0, MPI_COMM_WORLD);

double local_dot = 0.0;
for (int i = 0; i < local_n; i++)
    local_dot += a_local[i] * b_local[i];

double total_dot;
MPI_Reduce(&local_dot, &total_dot, 1, MPI_DOUBLE, MPI_SUM, 0, MPI_COMM_WORLD);

if (rank == 0) printf("a·b = %f\n", total_dot);

CUDA — modelo mental

Decisiones que hay que tomar, en orden

  1. ¿Tiene sentido usar GPU? Necesitamos miles o millones de elementos independientes a procesar. Si son cientos o hay mucha lógica condicional, la CPU gana.
  2. Identificar el kernel: ¿qué operación se aplica idénticamente a muchos elementos? Esa es la función a escribir con __global__.
  3. Mapear datos a hilos: ¿cuántos hilos lanzo? Uno por elemento es lo habitual. Ese hilo calculará su índice a partir de blockIdx.x, blockDim.x, threadIdx.x.
  4. Dimensionar grid y bloque: tamaño de bloque típico 128, 256 ó 512 (múltiplo de 32, el tamaño del warp). Número de bloques: ceil(n/B) = (n + B − 1) / B.
  5. Gestionar memoria: las cinco fases canónicas (malloc → memcpy H→D → kernel → memcpy D→H → free).
  6. ¿Cooperación entre hilos? Si los hilos de un mismo bloque necesitan compartir datos (típico en reducciones intra-bloque o stencils), usar __shared__ y sincronizar con __syncthreads().
  7. Comprobar fronteras: cuando n no es múltiplo del tamaño de bloque, los hilos sobrantes deben saltarse el trabajo con if (i < n).

Ejemplo guiado: y = α x + y (SAXPY) — el «hola mundo» de CUDA

// Kernel: lo que ejecuta CADA hilo
__global__
void saxpy_kernel(int n, float alpha, float *x, float *y) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;     // mi índice global
    if (i < n) y[i] = alpha * x[i] + y[i];             // guarda fronteriza
}

int main() {
    int    n     = 1 << 20;                  // 2^20 elementos
    size_t bytes = n * sizeof(float);
    float  alpha = 2.0f;

    // Buffers en host
    float *x = (float*) malloc(bytes);
    float *y = (float*) malloc(bytes);
    // ... inicializar x e y ...

    // 1. Reservar memoria en device
    float *d_x, *d_y;
    cudaMalloc(&d_x, bytes);
    cudaMalloc(&d_y, bytes);

    // 2. Copiar host → device
    cudaMemcpy(d_x, x, bytes, cudaMemcpyHostToDevice);
    cudaMemcpy(d_y, y, bytes, cudaMemcpyHostToDevice);

    // 3. Lanzar kernel
    int B = 256;                            // tamaño de bloque
    int G = (n + B - 1) / B;                // número de bloques (techo)
    saxpy_kernel<<<G, B>>>(n, alpha, d_x, d_y);

    // 4. Copiar resultado device → host
    cudaMemcpy(y, d_y, bytes, cudaMemcpyDeviceToHost);

    // 5. Liberar memoria
    cudaFree(d_x); cudaFree(d_y);
    free(x); free(y);
    return 0;
}

Fórmula clave del kernel

Esta línea aparece en casi todo kernel 1D y conviene tenerla automatizada en la cabeza:

int i = blockIdx.x * blockDim.x + threadIdx.x;

Cómo se lee: «de los blockIdx.x bloques que vienen antes, cada uno tiene blockDim.x hilos; dentro de mi bloque soy el hilo número threadIdx.x». El resultado es mi posición lineal global, que uso para indexar arrays.

Si trabajas en 2D (matrices), repites en las dos dimensiones: row = blockIdx.y*blockDim.y + threadIdx.y; col = blockIdx.x*blockDim.x + threadIdx.x;

Comparación rápida: el mismo problema en las tres tecnologías

Suma de un array a[n] resuelta con los tres modelos. Observa cómo cambian las decisiones:

OpenMP

double s = 0;
#pragma omp parallel for \
        reduction(+: s)
for (int i = 0; i < n; i++)
    s += a[i];

Una directiva. El runtime se encarga de todo: crear hilos, repartir iteraciones, combinar copias locales.

MPI

double ls = 0, gs;
for (int i = rank;
     i < n;
     i += nproc)
    ls += a[i];
MPI_Reduce(&ls,&gs,1,
   MPI_DOUBLE,MPI_SUM,0,
   MPI_COMM_WORLD);

Reparto manual con rank+nproc, suma local explícita, comunicación con Reduce.

CUDA

// Necesita reducción
// intra-bloque con
// __shared__ +
// __syncthreads y
// luego reducción
// entre bloques.
// Más complejo que
// los otros dos.

CUDA brilla en operaciones map (independientes). Las reducciones requieren patrón especializado o usar cuBLAS/Thrust.

Moraleja: cada tecnología optimiza un tipo de paralelismo. OpenMP es el más expresivo para un nodo multicore con código compartido. MPI es el más explícito pero el único que escala fuera del nodo. CUDA es el más rápido cuando hay miles de elementos a procesar independientemente, a cambio de complicar reducciones y operaciones con dependencias.

Race condition oculta Bucle paralelizado donde varios hilos escriben en la misma posición compartida Necesita critical, atomic o reduction. Si no se puede arreglar y se usa reduction, suele ser la opción «correcta» aunque parezca sutil.

Esquema mal asignado «Mergesort con programación dinámica», «Floyd con divide y vencerás» Mergesort, quicksort, FFT → Divide y Vencerás. Floyd, alineamiento, Fibonacci con memoización → Programación Dinámica.

Bloque VII · Plantillas mentales para responder rápido

OpenMP: «¿qué directiva uso?»

  • Bucle for con iteraciones independientes → #pragma omp parallel for
  • Bucle con acumulación en una variable → añadir reduction(+: var)
  • Iteraciones con coste muy variable → schedule(dynamic) o guided
  • Dos o tres trozos de código diferentes en paralelo → parallel sections
  • Recorrer una estructura recursiva o un while → parallel + single que crea task
  • «Solo uno lo hace, los demás esperan» → single (o master si te da igual la barrera)
  • «Solo uno lo hace cada vez, todos pasan» → critical (o atomic si es x++ u operación simple)

MPI: «¿qué función uso?»

  • Un proceso a otro → Send / Recv
  • Riesgo de deadlock con Ssend mutuo → Sendrecv o no bloqueantes
  • De uno a todos (mismo dato) → Bcast
  • De uno a todos (trozos distintos) → Scatter
  • De todos a uno (cada uno aporta un trozo) → Gather
  • De todos a uno (combinando con op) → Reduce
  • Resultado en todos → versión Allreduce, Allgather, etc.
  • Solapar cómputo y comunicación → Isend/Irecv + Wait

CUDA: «¿qué pasos tiene un programa?»

  1. Reservar memoria en device: cudaMalloc(&d_x, bytes)
  2. Copiar host → device: cudaMemcpy(d_x, x, bytes, cudaMemcpyHostToDevice)
  3. Lanzar kernel: kernel<<<G, B>>>(d_x, ...)
  4. Copiar device → host: cudaMemcpy(y, d_y, bytes, cudaMemcpyDeviceToHost)
  5. Liberar: cudaFree(d_x)

Dentro del kernel: int i = blockIdx.x * blockDim.x + threadIdx.x; y comprobación if (i < n) antes de tocar memoria.

Bloque VIII · Mini-quiz de autoevaluación

Diez preguntas para comprobar que has interiorizado lo esencial. Hazlas sin mirar las respuestas; luego revisa.

¿Qué pasa si pones #pragma omp parallel antes de un bucle pero olvidas el for?
Cada hilo ejecuta el bucle completo, no se reparte. Si los hilos escriben en el mismo array hay race condition. El trabajo total es p × n iteraciones (más lento que la serie).
Programa con fracción serie s = 0.1. ¿Speed-up máximo asintótico (p → ∞)?
Smáx = 1/s = 1/0.1 = 10. Por mucho que aumentes los procesadores, el 10 % serie pone un techo de 10×. Es la moraleja de Amdahl.
Diferencia esencial entre MPI_Send y MPI_Ssend.
MPI_Send (estándar) deja a la implementación decidir entre usar buffer o sincronizar; el comportamiento depende del tamaño. MPI_Ssend (síncrono) garantiza que no retorna hasta que el receptor empieza a recibir.
¿Para qué sirve reduction(+: suma) en OpenMP?
Cada hilo tiene una copia local de suma inicializada a 0 (neutro de +). Acumula sin sincronización. Al final, OpenMP combina todas las copias con + y deja el total en la variable original. Evita la sección crítica que serializaría el bucle.
¿Cómo paralelizarías mergesort con OpenMP?
Con task: cada llamada recursiva genera dos tareas paralelas (mitad izquierda y derecha) seguidas de taskwait. Llamada inicial dentro de parallel + single. Conviene un umbral: por debajo de cierto tamaño, llamar a la versión secuencial para no crear tareas demasiado pequeñas.
Tienes una matriz double **a y quieres enviarla con MPI. Problema.
double **a es array de punteros: las filas no están contiguas en memoria. MPI_Send(a, …) mandaría los punteros, no los datos. Solución: usar double *a con indexado i*N + j, o construir un MPI_Datatype derivado, o copiar a un buffer contiguo.
¿En qué se diferencia un pipeline de un esquema síncrono?
El síncrono opera todos los elementos a la vez en cada paso temporal (barrera global). El pipeline tiene etapas distintas en cascada: en un instante dado, cada etapa procesa un elemento distinto del flujo. La aceleración del pipeline es el número de etapas; la del síncrono es el número de procesadores trabajando en paralelo dentro de cada paso.
¿Por qué transferir datos host↔device puede arruinar un programa CUDA?
El bus PCIe es órdenes de magnitud más lento que la memoria interna de la GPU. Si para cada cómputo pequeño copias ida y vuelta, el tiempo de transferencia domina. Patrón correcto: transferir una vez, ejecutar muchos kernels, recoger una vez.
Diferencia entre MPI_Reduce y MPI_Allreduce.
Ambos combinan valores de todos los procesos con un operador (suma, max…). Reduce deja el resultado solo en el root. Allreduce lo deja en todos los procesos. Internamente, Allreduce es más eficiente que Reduce + Bcast.
Si tuvieras que decidir entre OpenMP y MPI para un cluster de 64 nodos × 16 cores, ¿qué eliges?
Híbrido MPI + OpenMP. Solo OpenMP no llegaría a otros nodos (memoria compartida limitada al nodo). Solo MPI (1024 procesos) replicaría estructuras y haría comunicaciones intra-nodo innecesarias. Un proceso MPI por nodo + 16 hilos OpenMP cada uno es la configuración natural.
¿Qué hace MPI_Bcast(&n, 1, MPI_INT, 0, MPI_COMM_WORLD)?
Difunde el valor de n (1 entero) desde el proceso 0 a todos los procesos del comunicador MPI_COMM_WORLD. Es una operación colectiva: todos los procesos deben llamarla, no solo el root. Tras la llamada, todos tienen el mismo valor en n.
Indica las 4 fases de la metodología de Foster.
Particionado (descomponer en muchas tareas), Comunicación (identificar dependencias), Aglomeración (agrupar tareas pequeñas), Mapeo (asignar a procesadores físicos). Acrónimo: PCAM.

Sesión 6 · Análisis de algoritmos paralelos

¿Por qué medimos?

Un algoritmo paralelo se diseña para correr más rápido, no por estética. Para saber si lo conseguimos necesitamos tres preguntas: ¿cuánto se acelera?, ¿usamos bien los procesadores?, ¿escala cuando crecemos? Las tres tienen una métrica concreta.

Speed-up y eficiencia: qué dicen realmente

El speed-up S(p) = T1 / Tp compara el tiempo del mejor secuencial con el del paralelo en p procesadores. Cuidado con el numerador: si usas como T1 tu propio paralelo lanzado con un único hilo, estás haciendo trampa, porque ese código arrastra overhead. Lo correcto es el algoritmo secuencial óptimo.

La eficiencia E(p) = S(p) / p es el speed-up por procesador. Si E = 1, estás aprovechando cada uno al 100 %. En la práctica E < 1 casi siempre: hay comunicación, sincronización y partes serie. Una eficiencia de 0,7 con 64 procesadores ya es muy buena.

El coste es C(p) = p · Tp: el tiempo total agregado de procesador (como sumar horas de trabajo). Un algoritmo se llama coste-óptimo si C(p) = Θ(T1), es decir, no estás malgastando trabajo respecto al secuencial.

La ley de Amdahl, explicada con un ejemplo

Si una fracción s del tiempo es estrictamente serie y el resto (1−s) paraleliza perfectamente, el speed-up máximo es S(p) ≤ 1 / (s + (1−s)/p). La parte serie no se acelera; cuando p tiende a infinito, el segundo término se anula y queda S∞ = 1/s.

Ejemplo: si s = 5 % (= 0,05), el techo es 1/0,05 = 20. Aunque tengas mil procesadores, no pasarás de 20×. La lectura práctica: optimiza la parte serie antes de comprar más nodos.

El modelo ts + n·tw

Comunicar n datos cuesta tcom = ts + n·tw: una latencia fija ts (coste de arrancar el mensaje, ronda los microsegundos) más un ancho de banda inverso tw por dato. Consecuencia directa: mandar mil mensajes de un byte es muchísimo más caro que un mensaje de mil bytes. De ahí la regla de oro agrupa mensajes.

El tiempo total paralelo se modela como Tp = Tcómputo/p + Tcom(p) + Tocio. Para que el paralelismo gane, el cómputo tiene que dominar; si la comunicación crece tan rápido como el reparto del trabajo, la curva de speed-up se aplana o incluso baja al subir p (slowdown).

Escalabilidad e isoeficiencia

Un algoritmo es escalable si, al aumentar p, se puede mantener la eficiencia subiendo el tamaño del problema n. La función de isoeficiencia dice cuánto hay que aumentar n en función de p para que E se mantenga constante. Si crece lentamente (lineal en p), el algoritmo escala bien; si crece como p² o peor, no escala.

Sesión 7 · Metodología de diseño (PCAM de Foster)

¿Para qué sirve PCAM?

Diseñar un algoritmo paralelo "a ojo" lleva a soluciones acopladas a la máquina y difíciles de migrar. Foster propone separar el qué se puede paralelizar (Partición y Comunicación) del cómo lo ejecuto en mi hardware (Aglomeración y Mapeo). Las dos primeras fases son abstractas; las dos últimas son físicas.

Particionado: descomponer al máximo

Se identifican las tareas más pequeñas posibles, ignorando el número real de procesadores. Hay dos estilos:

  • Descomposición de dominio: troceamos los datos. Cada tarea trabaja sobre un subconjunto. Es la natural en mallas, vectores, matrices, imágenes.
  • Descomposición funcional: troceamos las operaciones. Cada tarea hace una función distinta sobre el mismo dato. Es la natural en pipelines y procesadores de eventos.

Comunicación: dependencias entre tareas

Se marca qué datos necesita cada tarea de cuáles otras. Distinguimos comunicación local (entre vecinos, típica de stencils) de global (todos con todos, típica de reducciones o FFT), y comunicación estática (patrón fijo, conocido en compilación) de dinámica (depende del estado en ejecución, p. ej. ramificación en B&B).

Aglomeración: agrupar para reducir overhead

Las tareas mínimas suelen ser demasiadas. Si cada una hace 10 µs de cómputo y necesita 50 µs de comunicación, perdemos. Por eso se agrupan tareas vecinas en bloques más grandes con tres objetivos: (1) bajar el ratio comunicación/cómputo, (2) reducir el número de mensajes, (3) preservar suficiente paralelismo para todos los procesadores.

Mapeo: asignación a procesadores físicos

El último paso decide qué aglomerado corre dónde. Las dos consideraciones clave son balance de carga (que ningún procesador termine mucho antes que otros) y localidad (que las tareas que se comunican mucho estén cerca). Si la carga es regular, mapeo estático cíclico o por bloques. Si es irregular o dinámica, mapeo por bolsa de tareas (lo veremos en la sesión 12).

Sesión 8 · Paralelismo de datos

La idea

Aplicar la misma operación a una colección grande de datos independientes. Es el esquema más simple y el que mejor escala, porque el paralelismo es masivo y el patrón de acceso es regular. Cubre desde y = αx + y (SAXPY) hasta multiplicaciones matriciales, filtros de imagen y simulaciones físicas paso a paso.

SIMD, SPMD, SIMT: matices

SIMD es hardware: una instrucción opera sobre varios registros a la vez (AVX en CPU, unidades vectoriales). SPMD es software: todos los procesos ejecutan el mismo programa pero cada uno opera sobre sus datos (modelo MPI). SIMT es la mezcla de NVIDIA: cada hilo CUDA tiene su propio contador de programa, pero la unidad de planificación (warp) ejecuta 32 hilos a la vez como SIMD. Tres niveles distintos, los tres son paralelismo de datos.

Balance de carga: el problema oculto

Si la operación cuesta lo mismo por dato (suma de vectores), una división por bloques es óptima. Si el coste varía por dato (Mandelbrot: cada píxel necesita un número distinto de iteraciones), un reparto por bloques deja procesadores ociosos. Soluciones: distribución cíclica (cada uno coge los datos i, i+p, i+2p…, mezcla zonas caras y baratas) o schedule dynamic en OpenMP / bolsa de tareas en MPI.

Comunicación en paralelismo de datos

Cuando hay fronteras (stencil: cada celda depende de sus vecinas), la descomposición regular implica intercambiar halos en cada paso: una fila/columna con los procesos vecinos. El cómputo crece con el área del bloque (n²), la comunicación con su perímetro (n), así que bloques grandes → mejor relación. Esto justifica por qué en una matriz se prefieren submatrices cuadradas antes que franjas largas.

Sesión 9 · Paralelismo síncrono y relajado

Síncrono: barreras en cada paso

En problemas iterativos (Jacobi, calor, juego de la vida) cada paso usa los valores del paso anterior. Para garantizarlo, todos los procesos calculan, intercambian fronteras y sincronizan con una barrera antes del siguiente paso. Es determinista, reproducible y fácil de razonar, pero el procesador más lento marca el ritmo.

Doble buffer

Para no pisar valores que aún se leen, se usan dos arreglos: U_old (lectura) y U_new (escritura). Al final del paso se intercambian (basta intercambiar punteros, no copiar). Esto desacopla lectura de escritura y permite paralelizar el cuerpo del bucle sin condiciones de carrera.

Relajado: renunciar a la sincronía global

Si el algoritmo converge aunque cada proceso lea valores algo desactualizados, podemos quitar la barrera. Gauss-Seidel asíncrono, métodos de relajación, metaheurísticas (algoritmos genéticos paralelos, enjambre) caben aquí. Ganamos tiempo (el más rápido no espera al más lento) pero perdemos determinismo: la trayectoria depende del orden real de ejecución. Solo se aplica cuando hay garantías teóricas de convergencia.

Solapar comunicación y cómputo

Truco clave incluso en el caso síncrono: lanzar la comunicación de fronteras con MPI_Isend/Irecv antes de calcular el interior del dominio (que no depende de los vecinos). Cuando termina ese cómputo interno, los halos ya han llegado y solo queda actualizar la frontera. La comunicación cuesta "gratis" porque se hace en paralelo con cómputo útil.

Sesión 10 · Grafo de dependencias y pipeline

Grafo de tareas (DAG)

Cuando las tareas tienen dependencias arbitrarias (no son todas iguales), se modelan como un grafo dirigido acíclico: nodos = tareas, arcos = "esta tarea necesita la salida de aquella". El tiempo paralelo mínimo está acotado inferiormente por el camino crítico: la cadena más larga de tareas dependientes. Aunque tengas infinitos procesadores, no bajas de la suma de tiempos en ese camino.

Implicación práctica: para acelerar, hay que acortar el camino crítico (paralelizar las tareas que están en él), no añadir procesadores a tareas fuera del camino.

Pipeline (segmentación)

Caso particular de grafo: cadena lineal de etapas E1 → E2 → … → Ek. Cada etapa procesa un elemento y lo pasa a la siguiente. En régimen permanente las k etapas trabajan a la vez sobre k elementos distintos del flujo.

  • Latencia: tiempo que tarda un elemento en cruzar el pipeline = suma de tiempos de etapa.
  • Throughput: elementos procesados por unidad de tiempo en régimen permanente = 1 / (tiempo de la etapa más lenta). El cuello de botella manda.
  • Speed-up ideal: k, alcanzable solo si las etapas están balanceadas. Si una etapa duplica las demás, replícala o redistribuye el trabajo.

¿Cuántas etapas?

Más etapas → más paralelismo potencial, pero también más comunicación entre etapas y más latencia inicial (hay que llenar el pipeline). El óptimo aparece cuando todas las etapas tardan parecido y la comunicación entre ellas no domina.

Sesión 11 · Divide y vencerás y programación dinámica

Divide y vencerás paralelo

Estructura recursiva: divide el problema, resuelve recursivamente las partes (en paralelo) y combina. Quicksort, mergesort, FFT, multiplicación de matrices de Strassen siguen este patrón. En OpenMP se expresa de forma directa con #pragma omp task; en MPI mediante reparto inicial entre procesos.

Granularidad y umbral de corte

El problema crítico es la granularidad: si seguimos subdividiendo hasta tamaños minúsculos, el coste de crear tareas y comunicar supera el de calcular. Por eso se fija un umbral: cuando el subproblema es más pequeño que ese tamaño, se resuelve secuencialmente sin crear nueva tarea. Es el equivalente paralelo de "para arrays pequeños, mejor inserción que quicksort".

Combinación: el cuello escondido

La fase de combinar (merge) es a menudo secuencial y puede dominar el tiempo total. Acelerarla (merge paralelo, reducción en árbol) es lo que define un buen algoritmo D&V paralelo.

Programación dinámica paralela

PD rellena una tabla en la que cada celda depende de otras anteriores. La clave es ver el frente de onda: el conjunto de celdas que pueden calcularse a la vez porque sus dependencias ya están listas. Ejemplos:

  • Alineamiento de secuencias (Needleman-Wunsch): cada celda (i,j) depende de (i−1,j), (i,j−1) y (i−1,j−1). Las antidiagonales son frentes de onda independientes.
  • Floyd-Warshall: para cada k, la matriz se actualiza en función de la columna y la fila k. Las filas dentro de un mismo k son independientes; el bucle externo en k es serie.

La regla general: el bucle paralelizable es el que recorre celdas de un mismo frente; el que las atraviesa de un frente al siguiente debe quedar serie.

Sesión 12 · Trabajadores replicados y branch & bound

Bolsa de tareas (replicated workers)

Patrón maestro-trabajadores con una cola común de tareas. Cada worker, en cuanto termina una tarea, pide la siguiente. Resuelve el problema de balance de carga cuando las tareas tienen costes muy distintos y desconocidos a priori: backtracking, ray tracing, búsquedas heurísticas. La cola puede ser centralizada (maestro reparte) o distribuida (work stealing: los workers se roban tareas entre sí cuando se quedan sin trabajo).

El problema de la terminación

¿Cuándo paramos? No basta con que un worker tenga la cola vacía: puede que otro le envíe una nueva tarea (porque está generando ramas en B&B, por ejemplo). Si paramos demasiado pronto, perdemos resultados. Soluciones clásicas:

  • Algoritmo del testigo de Dijkstra: un testigo circula entre los procesos; solo se considera terminado si completa una vuelta entera sin encontrar a nadie con trabajo pendiente y sin mensajes en vuelo.
  • Contadores de mensajes: cada proceso lleva cuenta de mensajes enviados y recibidos; cuando los totales globales cuadran y todas las colas están vacías, hay terminación.

Branch & bound paralelo

Variante de backtracking que mantiene una cota (mejor solución encontrada hasta el momento) y poda ramas cuyo cálculo demuestra que no pueden mejorarla. En paralelo:

  • Cada worker explora un subárbol y mantiene una cota global que se difunde cuando se mejora (broadcast asíncrono al resto de procesos).
  • Una buena cota inicial reduce muchísimo el trabajo total. Por eso a veces conviene hacer una búsqueda greedy serie primero, para sembrar una cota razonable, antes de lanzar la exploración paralela.

Anomalías de speed-up

Curiosidad importante: en B&B paralelo pueden aparecer aceleraciones superlineales (speed-up > p) y también slowdown, según el orden en que se exploran las ramas. Si un worker encuentra rápido una cota buena, los demás podan masivamente y el trabajo total cae por debajo del secuencial. Si los workers exploran ramas inútiles antes que el secuencial habría visitado, el trabajo total sube. Es el único esquema del temario donde estas anomalías son esperables.

Síntesis cruzada: cómo encajan los esquemas

Una forma útil de organizar el bloque IV en la cabeza:

  • Si las tareas son iguales y los datos regulares → paralelismo de datos.
  • Si hay iteración con dependencia del paso anterior → síncrono (o relajado si tolera asincronía).
  • Si hay dependencias arbitrarias → grafo; si son lineales → pipeline.
  • Si el problema es recursivo → divide y vencerás.
  • Si la solución se construye sobre tabla con dependencias entre celdas → PD con frente de onda.
  • Si las tareas se generan en ejecución y son irregulares → trabajadores replicados; si además hay poda, B&B.

Y la regla cruzada con la sesión 6: cualquiera de ellos solo escala si la comunicación crece más despacio que el cómputo. La elección de esquema condiciona ese ratio tanto como la elección de hardware.
