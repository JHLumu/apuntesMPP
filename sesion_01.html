Antes de programar en paralelo conviene mirar dónde se ejecutará el programa. Esta sesión inventaría los sistemas computacionales actuales y aclara vocabulario que se usará durante todo el curso.

Por qué necesitamos paralelismo

Los programas secuenciales tradicionales asumen que el procesador hace una instrucción cada vez. Hace dos décadas esa imagen ya no se corresponde con la realidad. Los sistemas computacionales actuales están constituidos por varios componentes, por lo que son sistemas paralelos por construcción.

El paralelismo se usa para tres tipos de problemas que la programación secuencial no resuelve bien:

  • Problemas grandes que no caben en la memoria de una única máquina.
  • Problemas costosos que tardarían días o meses en secuencial.
  • Problemas en tiempo real donde la respuesta tiene un plazo estricto (control de vuelo, vídeo, sensores).
Antes de seguir leyendo, intenta listar tres ejemplos de problemas computacionales que se beneficien de paralelismo. Generar tus propios ejemplos —en lugar de aceptar los del libro— activa la elaboración, una de las técnicas con mejor evidencia para retener conceptos abstractos.

El zoo de sistemas paralelos

No existe un único «ordenador paralelo». Los componentes pueden organizarse de formas muy distintas, y cada organización pide un paradigma de programación distinto.

Cada arquitectura impone un modelo de memoria —y por tanto un modelo de programación—.

Tres palabras que conviene no confundir

El vocabulario es la trampa más frecuente en este curso. Programación concurrente, paralela y en tiempo real suenan parecido y no son lo mismo.

Ejecución concurrente

Varias tareas aparentemente al mismo tiempo. Una sola CPU las alterna deprisa. El objetivo es gestionar simultaneidad lógica, no necesariamente ir más rápido. (Ej.: un servidor atendiendo cien clientes.)

Ejecución paralela

Varias tareas realmente al mismo tiempo, en hardware con más de una CPU. El objetivo es acelerar. Es la materialización del paralelismo cuando hay recursos.

REGLA MEMNÓNICA

«Concurrencia ≠ paralelismo»

Todo programa paralelo es concurrente, pero no al revés. Concurrencia es diseñar con varias unidades de proceso en mente; paralelismo es ejecutar esas unidades simultáneamente en hardware distinto.

Paralelismo dentro de una sola CPU

Antes de hablar de varios procesadores conviene saber que una sola CPU moderna ya hace muchas cosas en paralelo. Identificarlas evita malentendidos en clase:

1 · Múltiples unidades funcionales

El procesador tiene varios sumadores o multiplicadores. Permite hacer dos sumas en el mismo ciclo si no dependen entre sí.

2 · Segmentación (pipelining)

Una instrucción se descompone en etapas (fetch, decode, execute, writeback). Cuando una está en «execute», la siguiente ya está en «decode». Cada ciclo se completa una instrucción aunque cada una tarde varios.

3 · Unidades vectoriales (SIMD)

Una sola instrucción opera sobre 4, 8 o 16 datos a la vez. Librerías como SSE o AVX exponen esta capacidad. Es el modelo que llevarán las GPU al extremo.

4 · Multithreading hardware

Un núcleo físico expone dos hilos lógicos al sistema operativo (Hyper-Threading de Intel, SMT de AMD). Cuando un hilo está esperando memoria, el otro avanza.

Sesión 05 · Manycores

En GPU y Xeon Phi estas mismas ideas se llevan al extremo: cientos de unidades vectoriales coordinadas. La sesión 5 vuelve a este concepto desde el otro lado.

Práctica de recuperación

Pulsa cada tarjeta para revelar la respuesta. Intenta primero contestar mentalmente: el esfuerzo de recordar —incluso si fallas— es lo que consolida la memoria. Reléelo dentro de un día y vuelve a probar pasados tres.

¿Qué diferencia, en una sola frase, paralelismo de concurrencia?
Concurrencia es ejecución aparentemente simultánea (puede ser en una sola CPU); paralelismo es ejecución realmente simultánea en hardware distinto, con el objetivo explícito de acelerar.
Cita tres motivos por los que la programación paralela es necesaria hoy.
Resolver problemas mayores (caben en la memoria agregada), más rápidos (alto coste computacional) y de tiempo real (plazos estrictos).
¿Qué es Hyper-Threading / SMT y qué nivel de paralelismo representa?
Multithreading hardware: un núcleo físico expone dos hilos lógicos al sistema operativo. Es paralelismo dentro de un núcleo, no entre núcleos. Aprovecha latencias (espera de memoria) de uno de los hilos.
Nombra cuatro formas de paralelismo presentes dentro de una sola CPU «secuencial».
Múltiples unidades funcionales, segmentación (pipelining), unidades vectoriales (SIMD/SSE/AVX) y multithreading hardware.
¿Por qué un cluster pide MPI y un multicore pide OpenMP?
El cluster tiene memoria distribuida: los nodos no comparten direcciones, así que la única forma de comunicar es enviar mensajes (MPI). El multicore comparte la memoria, así que basta con coordinar hilos sobre variables comunes (OpenMP).
Qué es la computación heterogénea, en tus palabras.
Combinar en el mismo programa dispositivos de naturaleza distinta —típicamente CPU + GPU u otros aceleradores—, repartiendo el trabajo según las fortalezas de cada uno.
